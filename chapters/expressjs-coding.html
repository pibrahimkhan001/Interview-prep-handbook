Question 1

Write a simple ExpressJS server that responds with "Hello, World!" on the root route.

// server.js  — minimal Express "Hello, World"
import express from "express";

const app = express();
const PORT = process.env.PORT || 3000;

app.get("/", (req, res) => {
  res.send("Hello, World!");
});

// 404 for anything else
app.use((req, res) => res.status(404).send("Not Found"));

const server = app.listen(PORT, () => {
  console.log(`http://localhost:${PORT}`);
});

// graceful shutdown — flush in-flight requests on SIGTERM
process.on("SIGTERM", () => server.close(() => process.exit(0)));

Four steps: create the app, register a handler for GET /, add a catch-all 404, call listen. res.send("Hello, World!") automatically sets Content-Type: text/html; charset=utf-8 and ends the response. Use process.env.PORT so cloud platforms (Heroku, Render, Fly.io) can assign a port.

Gotchas: always separate the app object from server if you plan to test with supertest — export app from one module and call listen in another. Register SIGTERM handling before deploy so orchestrators can drain cleanly.

Question 2

Create a middleware in ExpressJS that logs the method and URL of every request.

import express from "express";
const app = express();

// logger middleware — runs on every request
app.use((req, res, next) => {
  const start = Date.now();
  res.on("finish", () => {
    const ms = Date.now() - start;
    console.log(`${req.method} ${req.originalUrl} ${res.statusCode} (${ms}ms)`);
  });
  next();                                  // pass control to next middleware
});

app.get("/",      (req, res) => res.send("home"));
app.get("/users", (req, res) => res.json([]));
app.listen(3000);

A middleware is just a function (req, res, next) — here it records the start time, attaches a finish listener to log method, URL, status, and duration, then immediately calls next(). Logging on finish captures the final status code (set by downstream handlers) and the total response time.

Gotchas: req.url shows the path after any mount prefix is stripped; use req.originalUrl for the full request path. For production, use pino-http or morgan — they add structured JSON output, redaction of sensitive headers, and log sampling. Always call next() exactly once or the request hangs.

Question 3

Implement a route in ExpressJS that handles a POST request and returns the JSON body of the request.

import express from "express";
const app = express();

app.use(express.json({ limit: "100kb" }));      // parse JSON body → req.body

app.post("/echo", (req, res) => {
  if (!req.body || Object.keys(req.body).length === 0) {
    return res.status(400).json({ error: "empty body" });
  }
  res.status(200).json(req.body);               // echo what we got
});

app.listen(3000);

Client:

curl -X POST http://localhost:3000/echo \
  -H "Content-Type: application/json" \
  -d '{"name":"Ada","age":36}'
# → {"name":"Ada","age":36}

The express.json() middleware parses bodies with Content-Type: application/json into req.body. Without it, req.body is undefined. The 400 guard handles clients that forget the Content-Type header or send an empty body.

Gotchas: set an explicit limit — the default (100kb in v4, 1mb in v5) is an easy DoS vector if left open-ended. For webhook endpoints that need to verify signatures, use verify: (req,res,buf) => { req.rawBody = buf; } to preserve the raw bytes before parsing.

Question 4

Write an ExpressJS route that handles a GET request with URL parameters and returns them as JSON.

import express from "express";
const app = express();

// route with URL parameters — /users/:userId/posts/:postId
app.get("/users/:userId/posts/:postId", (req, res) => {
  const { userId, postId } = req.params;

  // validate — URL params are always strings & come from the client
  if (!/^\d+$/.test(userId) || !/^\d+$/.test(postId)) {
    return res.status(400).json({ error: "IDs must be numeric" });
  }

  res.json({
    userId: Number(userId),
    postId: Number(postId),
    url:    req.originalUrl,
  });
});

app.listen(3000);
// GET /users/42/posts/7 → {"userId":42,"postId":7,"url":"/users/42/posts/7"}

Each :name token in the path becomes a key on req.params. Values are always strings; coerce to numbers/UUIDs explicitly. The regex guard returns 400 if the client sends /users/abc/posts/7 rather than letting a downstream DB layer fail mysteriously.

Gotchas: for type safety, use app.param("userId", (req,res,next,id) => ...) to run the validation once and attach a loaded resource, or use Zod: z.object({ userId: z.coerce.number() }).parse(req.params). Never splice req.params directly into SQL strings — parameterize queries.

Question 5

Implement an ExpressJS route that handles query parameters and returns them as JSON.

import express from "express";
import { z } from "zod";

const app = express();

// /search?q=express&page=2&limit=20&sort=newest
const SearchSchema = z.object({
  q:     z.string().min(1).max(100),
  page:  z.coerce.number().int().min(1).max(1000).default(1),
  limit: z.coerce.number().int().min(1).max(100).default(20),
  sort:  z.enum(["newest","oldest","top"]).default("newest"),
});

app.get("/search", (req, res) => {
  const parsed = SearchSchema.safeParse(req.query);
  if (!parsed.success) {
    return res.status(400).json({ error: "bad query", issues: parsed.error.issues });
  }
  res.json(parsed.data);
});

app.listen(3000);
// GET /search?q=api → {"q":"api","page":1,"limit":20,"sort":"newest"}

Query string values arrive in req.query as strings (or arrays if keys repeat: ?tag=a&tag=b). Zod's z.coerce.number() parses the string and enforces bounds in one step, and enums reject unexpected values. The defaults make ?q=foo work without supplying every parameter.

Gotchas: in Express 5 the default query parser switched to "simple" (no nested a[b]=1 syntax). Enable app.set("query parser", "extended") if you need nested objects. Bound numeric ranges — a client requesting ?limit=99999999 can otherwise crash your DB.

Question 6

Write a middleware function in ExpressJS that adds a timestamp to the request object.

import express from "express";
const app = express();

// attach a timestamp to each request
app.use((req, res, next) => {
  req.receivedAt   = new Date();              // Date object
  req.receivedAtMs = Date.now();              // epoch ms — handy for duration math
  next();
});

app.get("/", (req, res) => {
  res.json({
    now:     req.receivedAt.toISOString(),
    elapsed: Date.now() - req.receivedAtMs,   // how long we've been working
  });
});

app.listen(3000);

Attaching per-request state to req is the canonical Express pattern: a middleware decorates the request, and every downstream handler can read it. Timestamps are useful for duration logging, cache TTLs, audit records, and ordering events. Keep the middleware registered before your routes so the decoration always runs first.

Gotchas: use process.hrtime.bigint() if you need monotonic sub-millisecond precision — Date.now() can go backwards during NTP corrections. For TypeScript, declare the added property by augmenting the Express.Request interface so req.receivedAt is type-safe across the codebase.

Question 7

Create a route in ExpressJS that responds with a 404 error for any undefined routes.

import express from "express";
const app = express();

app.get("/",      (req, res) => res.send("home"));
app.get("/about", (req, res) => res.send("about"));

// 404 handler — MUST come after all real routes
app.use((req, res) => {
  // content-negotiate so both browsers and API clients get a useful response
  res.status(404).format({
    "application/json": () => res.json({ error: "Not Found", path: req.originalUrl }),
    "text/html":        () => res.send(`<h1>404 — ${req.originalUrl} not found</h1>`),
    default:            () => res.type("txt").send("Not Found"),
  });
});

app.listen(3000);

Express falls through every registered middleware/route in order; if nothing matches, the next registered handler runs. Placing a catch-all app.use((req,res)=>...) after all routes turns that fall-through into a proper 404. Using res.format() returns JSON to API clients (Accept: application/json) and HTML to browsers from the same handler.

Gotchas: don't confuse "no route matched" with "an error was thrown" — this 404 middleware handles the first; an error-handling middleware (4 args) handles the second. Always set res.status(404) explicitly; without it, clients and search engines see a 200 "Not Found" page.

Question 8

Write a route in ExpressJS that serves static files from a "public" directory.

import express from "express";
import path from "node:path";
import { fileURLToPath } from "node:url";

const __dirname = path.dirname(fileURLToPath(import.meta.url));
const app = express();

// serve everything in ./public at the root URL
app.use(express.static(path.join(__dirname, "public"), {
  maxAge:   "1y",                           // Cache-Control: max-age=31536000
  etag:     true,                           // enable conditional GETs
  index:    "index.html",                   // default file for / requests
  dotfiles: "ignore",                       // don't leak .env, .git, ...
}));

//  public/logo.png     → http://localhost:3000/logo.png
//  public/css/app.css  → http://localhost:3000/css/app.css
//  /                   → public/index.html

app.listen(3000);

The built-in express.static() streams files from a directory with correct MIME types and support for ETag/Last-Modified/range requests out of the box. maxAge: "1y" tells browsers to cache assets for a year — pair it with content-hashed filenames (app.a1b2c3.js) so cache-busting comes for free.

Gotchas: dotfiles: "ignore" is critical — without it, /.env is served if the file exists. Never point static() at your project root. In production, offload static serving to a CDN (Cloudflare, CloudFront) or reverse proxy (Nginx) — Node's event loop is better spent on dynamic work.

Question 9

Implement a middleware in ExpressJS to handle CORS.

// npm install cors
import express from "express";
import cors from "cors";

const app = express();

app.use(cors({
  origin: (origin, cb) => {
    // echo the origin if it's on the allowlist, else reject
    const allowed = ["https://app.example.com", "https://admin.example.com"];
    if (!origin || allowed.includes(origin)) cb(null, true);
    else                                     cb(new Error("CORS: origin not allowed"));
  },
  methods: ["GET","POST","PUT","DELETE"],
  allowedHeaders: ["Content-Type","Authorization","X-Request-Id"],
  credentials: true,                        // allow cookies / Authorization header
  maxAge: 86400,                            // cache preflight response 24h
}));

app.get("/api/data", (req, res) => res.json({ ok: true }));
app.listen(3000);

The cors middleware sets Access-Control-Allow-* headers and automatically responds to OPTIONS preflight requests. An explicit allowlist is safer than origin: "*" — especially when credentials: true, since the wildcard is illegal in that combination.

Gotchas: register cors() before any authentication middleware — preflights have no auth header and would otherwise fail with 401. Set maxAge up to 86400 (24h) so browsers cache the preflight and don't double your request count. For per-route CORS, pass cors(options) as a route-level middleware instead of mounting it app-wide.

Question 10

Create a route in ExpressJS that sends a JSON response with a custom status code.

import express from "express";
const app = express();

app.get("/users/:id", async (req, res) => {
  const user = await db.user.findUnique({ where: { id: req.params.id } });

  if (!user)           return res.status(404).json({ error: "User not found" });
  if (user.suspended)  return res.status(403).json({ error: "Account suspended" });

  res.status(200).json(user);
});

app.post("/users", async (req, res) => {
  const created = await db.user.create({ data: req.body });
  res
    .status(201)                                // 201 Created
    .location(`/users/${created.id}`)           // Location header
    .json(created);
});

app.delete("/users/:id", async (req, res) => {
  await db.user.delete({ where: { id: req.params.id } });
  res.status(204).end();                        // 204 No Content — no body
});

app.listen(3000);

res.status(code) is chainable — it only sets the status; you still need .json(), .send(), or .end() to finalize the response. Common pairings: 201 Created for POST success (with a Location header), 204 No Content for DELETE (no body), 404 for missing resources, 403 for "you exist but can't do this".

Gotchas: res.sendStatus(204) is shorthand for res.status(204).send("No Content") but includes the text body — use res.status(204).end() for a truly empty body. Always match the status to REST semantics; 200 OK on a failed create is a lie that confuses clients.

Question 11

Write a middleware in ExpressJS that limits the number of requests from a single IP address.

// npm install express-rate-limit
import express from "express";
import rateLimit from "express-rate-limit";

const app = express();
app.set("trust proxy", 1);                    // trust first proxy hop (X-Forwarded-For)

const limiter = rateLimit({
  windowMs: 15 * 60 * 1000,                   // 15-minute window
  max: 100,                                   // 100 req / IP / window
  standardHeaders: true,                      // RFC 9331 RateLimit-* headers
  legacyHeaders:   false,
  message: { error: "Too many requests, try again later." },
});

app.use("/api", limiter);

// tighter limit on login — 5 failed attempts / 15 min / IP
const loginLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 5,
  skipSuccessfulRequests: true,               // only count failures
});
app.post("/api/login", loginLimiter, loginHandler);

app.listen(3000);

express-rate-limit keeps a counter per key (default: client IP) and returns 429 Too Many Requests when it exceeds the limit. The tighter per-route limiter on login protects against brute-force attacks while keeping normal API traffic flowing.

Gotchas: app.set("trust proxy", 1) is essential behind a load balancer — without it, every client looks like the proxy's IP and limits trigger immediately. The default in-memory store is per-process: with multiple instances, install rate-limit-redis to share state. For truly hostile traffic, rate-limit at the edge (Cloudflare, AWS WAF) so requests never reach your Node process.

Question 12

Implement a route in ExpressJS that handles file uploads using multer.

// npm install multer
import express from "express";
import multer from "multer";
import path from "node:path";

const app = express();

// disk storage — files end up in ./uploads with unique names
const storage = multer.diskStorage({
  destination: "./uploads",
  filename: (req, file, cb) => {
    const ext = path.extname(file.originalname).toLowerCase();
    cb(null, `${Date.now()}-${Math.random().toString(36).slice(2)}${ext}`);
  },
});

const upload = multer({
  storage,
  limits: { fileSize: 5 * 1024 * 1024, files: 3 },  // 5 MB, max 3 files
  fileFilter: (req, file, cb) => {
    const ok = ["image/jpeg","image/png","image/webp"].includes(file.mimetype);
    cb(ok ? null : new Error("Invalid file type"), ok);
  },
});

// single file under form field "photo"
app.post("/upload", upload.single("photo"), (req, res) => {
  if (!req.file) return res.status(400).json({ error: "No file uploaded" });
  res.json({
    filename: req.file.filename,
    size:     req.file.size,
    mime:     req.file.mimetype,
    url:      `/uploads/${req.file.filename}`,
  });
});

app.listen(3000);

multer parses multipart/form-data. upload.single("photo") accepts one file under that field name and exposes it as req.file; other form fields land on req.body. upload.array("photos", 5) handles multiple files, upload.fields([...]) handles differently-named fields.

Gotchas: fileFilter only checks the client-reported MIME type — trust nothing; verify by inspecting magic bytes with file-type or by running an image decoder like sharp. Don't use express.json() in front of multer — it mangles the multipart body. For production, stream straight to S3 with multer-s3 rather than writing to local disk.

Question 13

Create an ExpressJS route that sends a file as a response.

import express from "express";
import path from "node:path";
import { fileURLToPath } from "node:url";

const __dirname = path.dirname(fileURLToPath(import.meta.url));
const app = express();

// 1. inline — browser displays if it can (PDF, images, text)
app.get("/view/:name", (req, res, next) => {
  const abs = path.resolve(__dirname, "files", req.params.name);
  // path-traversal guard — ensure resolved path stays in ./files
  if (!abs.startsWith(path.resolve(__dirname, "files") + path.sep)) {
    return res.sendStatus(400);
  }
  res.sendFile(abs, (err) => {
    if (err) next(err);                       // forward to error middleware, don't crash
  });
});

const inFlight = new Map();

async function cached(key, ttlSeconds, loader) {
  const hit = await redis.get(key);
  if (hit) return JSON.parse(hit);

  // coalesce concurrent misses on the same key

      const fresh = await loader();
      await redis.setex(key, ttlSeconds, JSON.stringify(fresh));
      return fresh;
    } finally {
      inFlight.delete(key);
    }
  })();
  inFlight.set(key, promise);
  return promise;
}

    userAgent: z.string().max(500).optional(),
    buildId:   z.string().max(50).optional(),
  }).strict().optional(),
}).strict();

// prevent feedback-spam
const feedbackLimiter = rateLimit({
  windowMs: 60 * 60 * 1000, max: 20,                         // 20 submissions / hour / IP
  keyGenerator: (req) => req.user?.id ?? req.ip,
});

app.post("/feedback", feedbackLimiter, async (req, res, next) => {
  try {
    const data = FeedbackSchema.parse(req.body);

    const feedback = await db.feedback.create({
      data: {
        ...data,
        userId: req.user?.sub ?? null,
        ip:     req.ip,
        userAgent: req.get("user-agent"),
      },
    });

    // fire-and-forget notifications — don't block the response
    queueNotifications(feedback).catch((err) => req.log?.error({ err }, "notify failed"));

    res.status(201).json({
      id: feedback.id,
      message: "Thanks! Your feedback has been received.",
    });
  } catch (err) {
    if (err.name === "ZodError")
      return res.status(400).json({ error: "Validation failed", issues: err.issues });
    next(err);
  }
});

async function queueNotifications(feedback) {
  // post to Slack, create Linear ticket, etc. — via a queue
  await queue.add("notify-feedback", { id: feedback.id });
}

app.listen(3000);

A feedback endpoint needs: schema validation (catch typos, reject junk), rate limiting (prevent spam), optional authentication (accept both logged-in and anonymous submissions), and side-effect queuing (Slack notifications, ticket creation) that doesn't block the response. Store context fields (URL, user agent, build ID) for reproducibility — users almost never remember which page they were on.

Gotchas: sanitize the message before displaying in an admin UI — HTML in feedback can XSS your own support team. Don't email the submitted address without verification; attackers will spoof others' emails to trigger abuse reports. Log IP and user agent but redact them in public-facing views; keep them only for abuse investigation. Queue heavy work — your /feedback endpoint shouldn't wait on Slack's API to respond.

Question 100

Create an ExpressJS route that handles real-time updates using WebSockets and Redis.

// npm install socket.io @socket.io/redis-adapter ioredis
import express from "express";
import http from "node:http";
import { Server } from "socket.io";
import { createAdapter } from "@socket.io/redis-adapter";
import Redis from "ioredis";

const app = express();
const server = http.createServer(app);

// two Redis connections — one to publish, one to subscribe
const pub = new Redis(process.env.REDIS_URL);
const sub = pub.duplicate();

const io = new Server(server, { cors: { origin: process.env.APP_URL } });
io.adapter(createAdapter(pub, sub));                   // broadcasts cross instances

// auth on the upgrade handshake
io.use(async (socket, next) => {
  try {
    socket.data.user = await verifyJwt(socket.handshake.auth?.token);
    next();
  } catch { next(new Error("Unauthorized")); }
});

io.on("connection", (socket) => {
  const userId = socket.data.user.id;
  socket.join(`user:${userId}`);                       // personal channel

  socket.on("room:join", (room) => {
    if (!canJoin(userId, room)) return;
    socket.join(`room:${room}`);
  });

  socket.on("message:send", async ({ room, text }) => {
    if (!text || text.length > 2000) return;
    const msg = { id: crypto.randomUUID(), from: userId, text, at: Date.now() };
    // persist asynchronously so the broadcast isn't gated on DB
    db.message.create({ data: { room, ...msg } }).catch(console.error);
    io.to(`room:${room}`).emit("message:new", msg);
  });
});

// server-side endpoint to push notifications to a specific user
app.post("/notify/:userId", requireInternalAuth, express.json(), async (req, res) => {
  io.to(`user:${req.params.userId}`).emit("notification", req.body);
  res.sendStatus(204);
});

server.listen(3000);

Socket.IO + Redis adapter = horizontally-scalable real-time. Each Node instance subscribes to Redis pub/sub; when one instance emits to a room, every other instance broadcasts to its local sockets in that room. Without the adapter, users connected to instance A never hear messages from instance B.

Gotchas: reserve two Redis connections (pub/sub) — Redis pub/sub blocks the subscribing connection, so it can't serve regular commands. Authenticate on the handshake (not on the first message) so unauthenticated clients can't flood your event handlers. For 10k+ concurrent connections per instance, raise ulimit -n, enable perMessageDeflate: false for CPU savings, and keep messages small (WebSocket frames aren't compressed by default). Ably and Pusher are hosted alternatives for teams that don't want to operate the realtime layer.
