Question 1

Explain how you would implement OAuth 2.0 in an API.

OAuth 2.0 is a delegation framework, not authentication — it lets an application act on a user's behalf without holding their credentials. In 2026, OAuth 2.1 is the consolidated spec: it mandates PKCE for all clients (including confidential ones), prohibits the implicit and password grants, and requires exact redirect-URI matching.

Grant typeWhen to use
Authorization Code + PKCEUser-facing apps (web, mobile, SPA). The only flow most apps need.
Client CredentialsService-to-service; no user involved.
Device CodeInput-constrained devices (TVs, CLIs).
Refresh TokenCompanion flow — rotate on every use.

Implementation essentials: use a vetted library (openid-client, better-auth) rather than rolling your own — the spec has subtle attack surfaces (CSRF via state, replay via nonce, token substitution, open redirects). Store (codeVerifier, state, nonce) server-side keyed to the session so the callback can verify them. Identify users by the stable sub claim, not email (which can change or collide across providers).

Production concerns: mint your own session tokens after the OAuth handshake — don't expose the IdP's access token to browsers. Use short access-token lifetimes (5-15 min) with rotating refresh tokens (reuse detection catches token theft). For public clients (SPAs, mobile) with no secret, PKCE + short-lived tokens + HttpOnly refresh cookies is the secure pattern. The dynamic OIDC flavor adds an ID token with user claims so you don't need a separate /userinfo call.

Question 2

How do you handle API versioning in a microservices architecture?

Microservices amplify the versioning problem: you're not evolving one contract, you're evolving many contracts across teams that release independently. The goal is allowing each service to move at its own pace without breaking consumers.

ApproachTrade-off
URL versioning (/v1/users)Visible, debuggable; URL churn between versions
Header versioning (Accept: application/vnd.api.v2+json)Clean URLs; less discoverable
Date-based (Stripe style: API-Version: 2026-03-15)Precise; clients pin to spec dates
Additive-only evolutionNo version needed; new fields added; old fields kept forever

Architectural patterns that help:

  • Gateway-level version routing. Public clients hit /v2/orders; the gateway translates to internal services. Internal services use a single internal contract; public contract stability is the gateway's concern.
  • Schema registry (Confluent, Buf) for event streams — enforces compatibility rules (backward, forward, full) at publish time.
  • Consumer-driven contracts (Pact) — consumers publish what they need; providers verify in CI before deploy.
  • Parallel change / expand-contract. Add the new field alongside the old, migrate consumers one by one, then remove the old field.

What to avoid: synchronized "big bang" version upgrades across services — they defeat the purpose of microservices. Deprecate gradually with Sunset and Deprecation headers (RFC 8594) plus email warnings; keep old versions running for 6-12 months. The bitter lesson: every breaking change across a service boundary is more expensive than you think; bias toward non-breaking evolution.

Question 3

What is HATEOAS, and how is it applied in RESTful APIs?

HATEOAS (Hypermedia as the Engine of Application State) is Roy Fielding's most-violated REST constraint: responses should embed links that guide the client's next actions, so the client doesn't hardcode URL templates.

// HATEOAS response — actions vary by resource state
{
  "id": 42, "status": "pending", "total": 99.99,
  "_links": {
    "self":   { "href": "/orders/42" },
    "cancel": { "href": "/orders/42/cancel", "method": "POST" },
    "pay":    { "href": "/orders/42/pay",    "method": "POST" }
  }
}

// after payment — different links, no "pay"
{
  "id": 42, "status": "paid", "total": 99.99,
  "_links": {
    "self":    { "href": "/orders/42" },
    "refund":  { "href": "/orders/42/refund", "method": "POST" },
    "invoice": { "href": "/orders/42/invoice" }
  }
}

The theoretical win: server controls the state machine — clients don't encode "after paid, show refund button"; they render whatever links the server sent. URL structure can evolve without client changes. It also makes APIs self-describing: a new client can discover capabilities by crawling from a root document.

Why it's rare in 2026: three forces push against it. Code generation from OpenAPI gives typed clients with compile-time safety — preferable to runtime link traversal. Bandwidth cost — HATEOAS doubles response size with metadata most clients ignore. Mobile/frontend reality — UI needs to know available actions ahead of time for layout; walking links at render time is clunky. Where HATEOAS shines: admin/CRUD UIs auto-generated from the API (PayPal, some enterprise systems), and hypermedia-driven stacks (HTMX, Turbo) where the server renders the UI anyway. For machine-to-machine JSON APIs, OpenAPI + typed clients wins.

Question 4

Describe how you would implement caching strategies for RESTful APIs.

Effective caching in REST is about layering — each layer absorbs reads that miss the faster layer above — and correctness, which comes down to invalidation. The infamous observation ("there are only two hard things in CS: cache invalidation and naming things") is a caching problem.

LayerTypical latencyShared across
Browser / client0One user
CDN / edge~10 msAll users near that POP
Reverse proxy (Varnish/Nginx)~1 msAll app servers
Redis / in-process LRU~1 ms / ~100 nsAll instances / one instance
Database buffer pool~10 msOne DB

HTTP caching hygiene: Cache-Control: public, max-age=60, stale-while-revalidate=300 lets CDNs serve stale for 5 minutes while revalidating in the background — near-zero latency with eventual freshness. Strong ETag + If-None-Match gives 304 responses (bandwidth savings without staleness). Always emit Vary for any header that changes the response (Accept, Accept-Language, Authorization).

Invalidation patterns:

  • TTL-based — simplest; accept brief staleness. Works for most content.
  • Write-through — writes update cache synchronously. Consistent but couples writes to cache availability.
  • Surrogate keys / cache tags (Fastly, Cloudflare, Varnish) — tag responses with entity ids; purge all responses tagged user:42 when user 42 changes. Gold standard for CDN invalidation.
  • Event-driven purge — subscribe to domain events; invalidate matching keys asynchronously.

Never cache responses with user-specific data on shared caches — use Cache-Control: private or explicit Vary: Authorization.

Question 5

How do you design an API to be idempotent, and why is idempotency important?

Idempotency means repeating a request has the same net effect as performing it once. It's the API property that lets clients retry safely — essential when mobile networks drop, users double-tap, or load balancers timeout and retry automatically.

MethodIdempotent by spec?Why
GET, HEAD, OPTIONSYesReads don't change state
PUTYes"Set resource to this value" — same twice = same state
DELETEYesDeleted stays deleted (return 204 on second call, not 404-as-error)
POSTNo"Create new thing" — default creates duplicates
PATCHUsuallyUnless operations aren't (e.g., { "counter": { "$inc": 1 } })

Making POST idempotent — the Idempotency-Key pattern (codified in RFC draft by Stripe, PayPal, Shopify):

POST /payments
Idempotency-Key: 2026-04-19-user-42-cart-abc
{ "amount": 1000, "currency": "USD" }

// server logic:
//   1. compute hash of (key, request body)
//   2. if (key exists with same hash) → return stored response
//   3. if (key exists with DIFFERENT hash) → 409 Conflict (reused key, different request)
//   4. else → process, store response under the key for 24 hours

Why it matters: without idempotency, a single network blip can double-charge a card or duplicate an order. Every serious write API (payments, messaging, compute provisioning) mandates idempotency keys. Store outcomes in a durable store (Postgres, Redis with AOF) — if the server crashes between "process request" and "store result," you must either (a) reprocess on retry and rely on DB constraints, or (b) use a transactional outbox. Generate keys client-side with a natural business id (order-id, payment-attempt-id) rather than a UUID per retry so retries actually collide.

Question 6

What are the benefits and drawbacks of using GraphQL over REST?

GraphQL solves specific problems REST has with complex, varied clients (over-fetching, under-fetching, schema evolution) and introduces specific problems of its own (caching, rate limiting, queries as attack surface).

GraphQL winsREST wins
Client picks exact fields — mobile efficiencyHTTP caching (GET + Cache-Control + ETag) just works
One request resolves nested graphsCDNs, reverse proxies, monitoring tools understand REST natively
Mandatory typed schema; codegen is excellentFile uploads, binary content, byte-range reads
Deprecation via @deprecated, no version bumpsLower backend complexity for simple CRUD
Built-in subscriptions for real-timeSimpler rate limiting (URL + method, not query depth)

Hidden costs of GraphQL at scale:

  • N+1 queries — naive resolvers fetch "posts, then for each post fetch author" serially. DataLoader batches per-request; it's not optional.
  • Query complexity attacks — adversarial clients send deeply nested queries to DoS the server. You need complexity analysis (graphql-query-complexity), depth limits, and per-query cost budgets.
  • Caching. POST-only by default means CDN caching doesn't work; persisted queries + GET requests + Apollo client cache partly solve it.
  • Error model. 200 OK with an errors array breaks HTTP-level monitoring — integrate with your logging/alerting specifically.

Practical guidance (2026): use GraphQL for complex internal BFFs and frontends with varied views (many different screens against the same data). Keep REST for public APIs (universal consumer skill, CDN caching) and simple CRUD. tRPC (TypeScript end-to-end) replaces GraphQL entirely for all-TS stacks — typed without the schema language.

Question 7

How do you secure an API using JSON Web Tokens (JWT)?

JWT-based API security has a deceptively simple premise (sign a claims object, verify on each request) with many attack vectors — most breaches come from implementation details, not the algorithm.

ConcernDefense
alg: none attackPin algorithms server-side: jwt.verify(token, key, { algorithms: ["HS256"] })
Key confusion (RS256 ↔ HS256)Never accept the algorithm from the token header unconditionally
Stolen tokens (XSS)Never store in localStorage; use HttpOnly, Secure, SameSite cookies
No revocationShort expiry (5-15 min) + jti blocklist in Redis for immediate revocation
Stolen refresh tokensRotation + reuse detection: if an already-used refresh token is presented, revoke the entire family
Audience confusionPin iss (issuer) and aud (audience) at verify time

Signing algorithm choice: HS256 for single-service APIs (symmetric, fast, easy). RS256/ES256 when multiple services verify tokens signed by a central auth server — receivers only need the public key; a leaked verifier can't mint tokens.

The refresh-rotation pattern: access tokens are short-lived JWTs (15 min). Refresh tokens are opaque strings in HttpOnly cookies, stored server-side with a family id. Every refresh returns a new access + new refresh and invalidates the old refresh. If an old refresh is reused, someone stole it: revoke all tokens in that family, force re-login. This is the current industry-standard trade-off between statelessness and revocation.

Common mistakes: putting PII/secrets in the token (base64-decoded trivially), trusting email without verification, using JWT for session state that grows over time (refresh tokens become huge), or forgetting to validate exp and nbf.

Question 8

Explain the concept of API gateways and their role in microservices.

An API gateway is the single ingress point between clients and a set of backend services. It's the seam where cross-cutting concerns (auth, rate limits, routing, observability) are enforced consistently, letting individual services focus on domain logic.

Responsibilities the gateway centralizes:

  • Routing — path/host-based dispatch to the right service.
  • Authentication — validate tokens once at the edge; pass a verified identity header (X-User-Id, signed) downstream.
  • Authorization — coarse-grained checks (plan tier, role) before requests touch services.
  • Rate limiting & throttling — one consistent policy per consumer across all services.
  • TLS termination — internal traffic can be plaintext or use mesh-managed mTLS.
  • Request/response transformation — protocol bridging (gRPC ↔ REST), field masking, format translation.
  • Aggregation / BFF — fan out to multiple services, stitch a composite response.
  • Observability — emit consistent access logs, metrics, and trace spans for every call.
Gateway categoryExamples
Cloud-managedAWS API Gateway, Google Cloud Endpoints, Azure APIM
Self-hostedKong, KrakenD, Tyk, Traefik, Envoy
Programmable (code-first)Node/Go/Rust reverse proxy with your own middleware

Gateway vs service mesh: a gateway handles north-south traffic (client → services); a service mesh (Istio, Linkerd) handles east-west (service → service). Gateways do auth and aggregation; meshes do mTLS, retries, traffic shifting, and telemetry between services. Most mature microservice platforms use both.

Anti-pattern: putting business logic in the gateway. Keep it stateless and policy-focused; services own their rules. Otherwise you've built a distributed monolith with bonus network hops.

Question 9

How do you implement rate limiting in an API, and why is it important?

Rate limiting protects an API from four distinct threats: abuse (scrapers, bots, credential stuffing), accidents (buggy client retry loops), noisy neighbors (one tenant starving others), and cost overruns (downstream paid APIs). Choose algorithm and placement based on which threat dominates.

AlgorithmBehaviorBest for
Fixed windowCounter per window; resets at boundarySimple; bursty at boundary
Sliding windowCounts requests in trailing N secondsAccurate; higher memory
Token bucketBucket refills at steady rate; allows bursts up to bucket sizeMost natural-feeling to users
Leaky bucketQueue drains at fixed rateSmooths spikes; adds latency
Concurrency limitN in-flight requests maxSlow endpoints (ML, reports)

Placement matters as much as algorithm:

  • Edge (Cloudflare, AWS WAF, API gateway) — catches volumetric attacks before they hit your servers. Cheapest.
  • Application — per-route limits with business context (per user, per endpoint). Redis-backed to share counts across instances.
  • Per-dependency — token bucket wrapping outbound calls to a third-party API so your system-wide consumption stays under their cap.

Production essentials: key by user id when authenticated, fall back to IP otherwise. Emit RFC 9331 headers (RateLimit-Limit, RateLimit-Remaining, RateLimit-Reset) so well-behaved clients self-pace. Always include Retry-After on 429. Set tighter limits on auth endpoints (5/15-min on login) to slow brute force. Consider global rate limits separately from per-user ones — they defend against distributed attacks where each IP looks innocent.

Question 10

Describe how you would handle long-running processes in an API.

Long-running operations break the HTTP request-response model: load balancers timeout at 30-60s, mobile clients lose connections, and blocking a thread for minutes wastes capacity. Every mature API exposes them asynchronously.

// 1. client submits work — server returns immediately with a job handle
POST /reports
{ "type": "annual", "year": 2026 }

HTTP/1.1 202 Accepted
Location: /jobs/job_abc123
{ "jobId": "job_abc123", "status": "queued" }

// 2. client polls OR receives webhook/SSE notification
GET /jobs/job_abc123
{ "jobId": "...", "status": "running", "progress": 0.42 }

// 3. done — result retrievable
{ "jobId": "...", "status": "done", "resultUrl": "/reports/rpt_xyz.pdf" }

Three delivery patterns:

PatternWhen
PollingSimple; no client infra. Backoff from 1s → 10s → 60s.
WebhookServer-to-server consumers with stable HTTPS endpoint.
Server-Sent Events / WebSocketLive progress in browsers/apps; supports granular updates.

Architecture: the API writes a job record (Postgres or Redis), emits a message (SQS, RabbitMQ, Redis stream, Kafka), returns 202. A worker pool consumes from the queue, updates progress, stores the result. Workers must be idempotent — messages can be delivered twice; use natural keys or transactional outbox. Include exponential retry with dead-letter queue so transient failures retry but poison messages don't block the queue forever.

Production concerns: bound job lifetime (reject if queued > 5 min, timeout running > 30 min), persist results with a TTL (24h typical), rate-limit per-user job submissions (one free-tier user shouldn't fill the queue), and emit metrics per job type (p95 latency, failure rate, queue depth) — these are usually the slowest parts of the system and deserve the most observability.

Question 11

What are the common pitfalls of designing RESTful APIs?

Real REST APIs fail the same ways repeatedly. These aren't theoretical purist complaints — each pitfall causes real bugs, breaches, or developer friction.

  • Verbs in URLs. POST /getUser — breaks HTTP semantics. Use nouns + methods. GET /users/42.
  • Wrong status codes. 200 OK with {"error": "..."} body breaks every HTTP tool. Match status to outcome: 400/401/403/404/409/422/429/5xx.
  • Inconsistent error shapes. Some endpoints return { error }, some { message }, some plain text. Standardize on RFC 7807 (application/problem+json) or a house format applied everywhere.
  • Breaking changes without versioning. Renaming a field is a breach of contract. Add new fields; never remove or rename existing ones without a version bump.
  • Over-fetching. Returning the entire user object (including passwordHash, internal flags) because a SELECT * leaked. Explicitly select fields.
  • IDOR (Insecure Direct Object Reference). Checking that the caller is authenticated but not that they own the record they're acting on. The #1 bug in OWASP API Top 10.
  • Ignoring idempotency. POSTs without idempotency keys double-charge on retry.
  • No pagination. GET /users returning 10,000 rows. Always paginate; cap limits.
  • Deep nesting. /users/42/posts/7/comments/3/replies/11 — hard to build URLs, hard to cache. Flatten: /comments/3/replies.
  • No request timeout. Slow upstream calls tie up connection pools; bound every network call.
  • Treating PATCH as PUT. Accepting partial bodies on PUT silently drops fields.
  • No rate limits. Launch-day scraper takes the site down.

Most of these compound: an API with missing ownership checks + no rate limiting + verbose error messages + plaintext IDs in URLs is trivially scrapable. The remedy isn't any single pattern — it's consistent hygiene enforced via API design reviews, an OpenAPI spec as the source of truth, and automated contract testing that catches regressions before deploy.

Question 12

How do you handle API errors and provide meaningful error messages?

API error handling has two audiences: the client developer diagnosing a bug, and tools (monitoring, retries, logs) that automate behavior. A good error response serves both.

HTTP/1.1 422 Unprocessable Entity
Content-Type: application/problem+json    // RFC 7807 standard

{
  "type":     "https://docs.example.com/errors/validation",
  "title":    "Validation failed",
  "status":   422,
  "detail":   "Request body has invalid fields",
  "instance": "/orders",
  "reqId":    "req_abc123",
  "errors": [
    { "field": "email",  "code": "invalid_format", "message": "Not a valid email" },
    { "field": "amount", "code": "out_of_range",   "message": "Must be between 1 and 1000000" }
  ]
}

Anatomy of a good error response:

  • Correct HTTP status — the class tells clients how to react (retry, re-auth, give up).
  • Machine-readable error code (INVALID_EMAIL) — clients switch on it; humans can't reliably parse prose.
  • Human message — for logs and developer tools. Never translate messages to end-users directly; that's the client's job.
  • Field-level details for validation — return all issues at once, not one-at-a-time.
  • Request id — users quote it in support tickets; you pivot to logs instantly.
  • Stable schema — same shape for all errors; type + code enumerate failures.

Critical don'ts: never leak stack traces, SQL errors, or internal hostnames in 500 responses — attackers map your stack through error messages. Never return 200 with an error body. Never mix error shapes across endpoints. Never vary error messages based on whether the resource exists vs whether the user owns it — that enumerates private data.

Operational side: log full error context server-side with the same reqId. Alert on 5xx rate and latency; don't alert on 4xx (those are the client's fault). RFC 7807 (Problem Details for HTTP APIs) is the standard — most frameworks have helpers.

Question 13

Explain the concept of API contracts and how they are maintained.

An API contract is the formal, machine-readable description of what an API accepts and returns: endpoints, request shapes, response schemas, error formats, auth requirements. It's the agreement between producer and consumer — and the source of truth for docs, codegen, mocks, and tests.

Artifact typeContract format
RESTOpenAPI 3.1 (JSON Schema internally)

How do you handle authentication and authorization in a RESTful API?

Authentication proves who you are; authorization decides what you can do. They're two separate pipeline stages, often conflated.

Authentication patterns:

  • Bearer JWT / OAuth token — most APIs. Stateless verification; pair with refresh tokens.
  • Session cookie — same-origin browser apps; HttpOnly, Secure, SameSite=Lax.
  • API key — server-to-server, SDK integrations. Hash in DB, never log raw keys.

Question 17

What is the role of middleware in an API, and how do you implement it?

Middleware is the composable pipeline that sits between the raw HTTP request and your handler. Each piece inspects, modifies, or short-circuits the request, then either calls the next or responds directly. It's the mechanism that keeps handlers focused on business logic.

// Express 5 — async middleware with native error propagation
function requireAuth() {
  return async (req, res, next) => {
    const [, token] = (req.headers.authorization || "").split(" ");
    if (!token) return res.sendStatus(401);
    try {
      req.user = await verifyJwt(token);
      next();                                    // → pass to next middleware
    } catch {
      res.sendStatus(401);                       // short-circuit
    }
  };
}

// composition — order matters
app.use(requestLogger);        // 1. log every request (captures final status on finish)
app.use(cors(corsOptions));    // 2. before auth (preflight carries no token)
app.use(express.json());       // 3. parse body
app.use(rateLimiter);          // 4. reject abuse before expensive work
app.use("/api", requireAuth()); // 5. auth on /api routes
app.use("/api", apiRouter);    // 6. handlers
app.use(errorHandler);         // 7. last — catches thrown errors

Why ordering matters:

  • CORS before auth — preflight OPTIONS has no token; if auth runs first it 401s the preflight and browsers abort.
  • Body parser before validation — validator needs parsed req.body.
  • Rate limiter before expensive work — reject before hitting DB or downstream services.
  • Error handler last — it's a 4-arg signature Express recognizes as the catcher.

Cross-cutting patterns middleware handles: logging, correlation id propagation (AsyncLocalStorage), compression, CORS, body parsing, validation, auth, rate limiting, CSRF tokens, security headers (Helmet), response time tracking, feature flags, circuit breakers. Keeping these out of handlers is the difference between maintainable code and 200-line functions that mix concerns.

Same concept in other frameworks: Koa's onion model (middleware wraps before + after), Fastify hooks (preHandler, onResponse), Django middleware, Rails Rack, FastAPI dependencies, Spring filters/interceptors. The word differs; the idea is universal.

Question 18

Explain the principles of RESTful architecture.

REST is an architectural style — a set of constraints defined by Roy Fielding (2000) that, applied together, yield systems that are scalable, cacheable, and evolvable. Fielding described six constraints.

  1. Client-Server. Separation of concerns: clients handle UI and state; servers handle data and logic. Either can evolve independently.
  2. Stateless. Each request carries all context it needs (auth, parameters). Servers don't remember previous calls — any server can handle any request, which is what enables trivial horizontal scaling.
  3. Cacheable. Responses explicitly declare what can be cached (Cache-Control, ETag). The entire HTTP caching stack (browsers, CDNs, reverse proxies) works for free.
  4. Uniform Interface. Four sub-principles: (a) resources identified by URIs, (b) manipulated via representations (JSON), (c) self-descriptive messages (status codes, content types), (d) HATEOAS.
  5. Layered System. Clients can't tell whether they're talking to the origin server, a cache, or a proxy. Enables CDNs, load balancers, security layers.
  6. Code on Demand (optional). Servers can send executable code to clients. Rarely used outside browsers.

Real-world REST in 2026: most "RESTful" APIs honor 1-5 but skip HATEOAS. Strict Fielding purists call this REST-ish; the industry calls it REST. The shared conventions (nouns as URLs, HTTP methods carry meaning, standard status codes, JSON) yield most of the benefit even without full hypermedia.

Why these constraints won: they describe how the web itself already worked in 1994. Adopting them gets you scalability, caching, and interoperability via infrastructure that's already deployed — browsers, load balancers, CDNs, proxies. Systems built against the grain of HTTP (SOAP-over-HTTP-as-transport, tunneled RPC) never got those benefits. REST's enduring success is the lesson: when designing distributed systems, leverage existing infrastructure conventions rather than hiding them.

Question 19

How do you design an API for scalability?

Scaling an API is about eliminating shared state, bounding expensive work, and moving reads out of the hot path. The techniques compound; you don't pick one.

DimensionTechnique

CachingHTTP caching (CDN, browser); application cache (Redis, in-process LRU); DB query cache
DatabaseRead replicas for reads; primary for writes; sharding by tenant/user for massive tables

Design patterns that enable scale:

  • Idempotency keys — safe retries under failure without duplicates.
  • Cursor pagination — O(limit) reads regardless of depth; offset pagination degrades at scale.
  • Batch endpoints — 100 requests in one call saves 99 round-trips.
  • CQRS — separate read models from write models; optimize each independently.
  • Event-driven updates — writes emit events; read projections are rebuilt asynchronously.

What actually breaks first under load: rarely CPU; usually DB connection pool exhaustion, unindexed queries, or N+1 queries. Profile before scaling out. The cultural piece: scaling APIs means saying no to synchronous operations that don't need to be sync. "Send an email in the request" is a latency bomb — queue it. "Compute analytics on read" should be pre-aggregated. The design discipline is more valuable than any single pattern.

Question 20

What is an API facade, and when would you use it?

An API facade is a thin API layer that wraps one or more backend APIs (internal services, third-party APIs, or legacy systems) to present a cleaner, more stable, client-friendly interface.

client  ──▶  Facade API  ──▶  user-service       (internal)
                          ──▶  legacy-mainframe   (SOAP + XML)
                          ──▶  stripe             (third-party)
                          ──▶  sendgrid           (third-party)

What a facade does for clients:

  • Aggregates multiple backend calls into one response (BFF pattern).
  • Translates formats — SOAP/XML upstream → REST/JSON to clients; gRPC → REST.
  • Normalizes auth — one auth scheme for clients even if backends use different ones.
  • Shields clients from backend churn — backend refactor without client changes.
  • Hides complexity — client doesn't need to know which backend owns which data.
  • Centralizes cross-cutting concerns — rate limits, caching, logging, circuit breakers.

When to reach for a facade:

  • Legacy integration — exposing a mainframe or old enterprise app as JSON/REST.
  • Multi-service aggregation — one page load needs 8 service calls; facade does the orchestration server-side.
  • Third-party abstraction — if Stripe, Braintree, and Adyen implement "payment," a facade lets you swap providers without client changes.
  • Mobile BFF — tight mobile bandwidth; facade trims and shapes response per app version.
  • Gradual migration — facade routes some endpoints to old stack, some to new; clients don't see the cutover.

Risks: a facade adds a hop — latency + another failure point. It can turn into a distributed monolith if business logic creeps in. Keep it thin: orchestration and translation, not logic. The BFF (Backend for Frontend) pattern is a specific flavor of facade — one per client type (web, mobile, partner) — each tailored to its consumer's needs without bloating a universal API.

Question 21

How do you handle pagination in an API?

Pagination is mandatory for any list endpoint that could return more than a screenful of results. The choice of technique matters: offset pagination is simple but degrades at scale; cursor pagination is the industry standard for large datasets.

StyleURLTrade-offs
Offset / page?page=2&limit=20Simple; "jump to page N" possible; O(offset) DB scan; unstable across inserts
Cursor / keyset?cursor=eyJpZCI6MTAwfQO(limit) regardless of depth; stable; can't "jump to page 50"
Time-based?before=2026-04-01&limit=50Natural for feeds; tie-breaker id needed for duplicates

Why cursor wins at scale:

// offset — DB scans 10,000 rows and discards them
SELECT * FROM posts ORDER BY created_at DESC LIMIT 20 OFFSET 10000;   -- slow

// cursor — DB uses index to jump straight to the cursor position
SELECT * FROM posts
 WHERE (created_at, id) < ('2026-03-15 10:00', 'abc123')
 ORDER BY created_at DESC, id DESC
 LIMIT 20;                                                              -- fast, O(limit)

Critical design details:

  • Tie-breaker on the sort column. Sorting by created_at alone produces duplicates when timestamps collide. Add id as secondary sort.
  • Opaque cursors. Base64-encode the cursor payload so clients can't inspect or craft them — you can change the format later without breaking clients.
  • Cap limit. Hardcoded max (typically 100); clients requesting 10,000 get 100.
  • Fetch limit+1 to detect "has more" without a separate COUNT(*) query.
  • Skip COUNT(*) on paginated endpoints — O(N) per page; return hasMore instead.
  • Return nextCursor: null when there are no more results — clients stop without guessing.

Industry references: Stripe, GitHub, Shopify all use cursor pagination for reason. Offset pagination is fine for admin UIs and small datasets; if you expect a list to grow past ~10,000 rows, pick cursor from day one.

Question 23

How do you handle versioning in GraphQL APIs?

GraphQL handles versioning fundamentally differently from REST: there are no versions. Instead, the schema evolves continuously using field-level @deprecated, additive changes, and strict breaking-change review.

type User {
  id: ID!
  email: String!
  displayName: String!
  # old field kept for existing clients; new clients use displayName
  username: String @deprecated(reason: "Use displayName")

  # adding new optional fields is safe — no version bump
  avatarUrl: String
  pronouns: String
}

What counts as a safe change:

  • Adding a new type, field, or argument with a default.
  • Adding a new enum value (if clients handle unknowns).
  • Marking a field @deprecated.
  • Changing a nullable field to non-nullable is safe for returns, breaking for inputs.

What breaks clients:

  • Removing a field or type (even deprecated).
  • Renaming a field.
  • Changing a field's type.
  • Making an optional argument required.
  • Removing an enum value (clients may match on it).

Tools that enforce this: GraphQL Inspector or schema-diff run in CI, flagging breaking changes on every PR. Apollo Studio and Grafbase go further — they track which clients use which fields in production; you can't remove a field while even one client still calls it.

Benefits of this model: no version sprawl; no "v2 rewrite"; clients that only query the fields they need never see removed fields break them (because you don't remove fields while anyone uses them). Costs: schemas accumulate deprecation noise over years; removing truly dead code requires operational discipline and usage tracking. For long-lived APIs, this model is strictly better than REST's major-version bumps — but it only works if you enforce the rules in CI.

What is the role of WebSockets in modern API design?

WebSockets are the long-lived, bidirectional complement to request-response HTTP. They trade HTTP's statelessness for real-time, low-latency pushes in both directions — essential where request-response is the wrong shape.

Use caseWhy WebSocket fits
Chat / messagingServer pushes on new messages; client sends typing indicators
Collaborative editingOperational transforms need sub-100ms bidirectional sync
Live dashboardsMetrics streamed as they're generated
Gaming & tradingLatency-sensitive full-duplex updates
IoT device controlServer initiates commands to devices

When not WebSocket:

  • One-way server → client streams — SSE (Server-Sent Events) is simpler (plain HTTP, auto-reconnect, works through more proxies) and sufficient for stock prices, notifications, progress updates.
  • Occasional updates — polling with ETag is cheaper than keeping sockets open.
  • Straightforward CRUD — WebSocket adds operational complexity without benefit.

Operational challenges at scale:

  • Connection state — each socket consumes memory; 100K concurrent connections on one Node process is possible but needs careful tuning (ulimit, keep-alive, libraries like ws or uWebSockets.js).
  • Fan-out across instances — user's socket lives on instance A; event originates on instance B. Solve with Redis pub/sub or a dedicated layer (Socket.IO Redis adapter, NATS, Ably, Pusher).
  • Auth on handshake — authenticate at connection time, not per message. Include a token in the query string (no custom header support in browsers) or subprotocol.
  • Heartbeat + reconnection — mobile networks drop silently; implement ping/pong every 30s; clients reconnect with exponential backoff and resume via Last-Event-Id or sequence number.
  • Backpressure — slow client + fast producer = runaway memory. Drop or coalesce messages if the send buffer exceeds a threshold.

In 2026: managed platforms (Ably, Pusher, AWS AppSync real-time) handle the infrastructure; WebTransport (HTTP/3) is the emerging successor with better congestion control. WebSockets remain the workhorse for interactive real-time features.

How do you implement secure communication between microservices?

Secure microservice communication has three threat models to defend against: eavesdropping (network sniffing), spoofing (unauthorized service calls), and replay (captured requests reused). The industry has converged on a small set of layered defenses.

mTLS is the gold standard. Each service has a certificate signed by an internal CA. On every connection, both sides verify each other's cert. Question 29

How do you implement a webhook, and what are its use cases?

A webhook is a reverse API call: the provider POSTs an HTTP request to a URL you registered whenever an event happens. It flips the polling model — instead of asking "did anything change?", you're told when it does.

// 1. you register a webhook endpoint
POST https://api.stripe.com/v1/webhook_endpoints
{ "url": "https://myapp.com/webhooks/stripe",
  "events": ["payment_intent.succeeded", "payment_intent.failed"] }

// 2. later — Stripe POSTs to your URL on events
POST /webhooks/stripe
Stripe-Signature: t=1745068800,v1=abc123...
{ "type": "payment_intent.succeeded", "data": { ... }, "id": "evt_xyz" }

// 3. you verify signature, process idempotently, return 200 quickly

Production requirements — the five things that matter:

  1. Verify signatures. Anyone can POST to your public endpoint. Providers sign payloads with a secret; compute HMAC locally and compare in constant time (crypto.timingSafeEqual). Reject unsigned or mismatched requests.
  2. Return 200 fast — queue work async. Providers retry if you respond slow or error. Don't do heavy work in the handler; write to a queue and ack immediately. Target < 1 second response.
  3. Dedupe by event id. Retries deliver the same event twice (or three, or ten times). Store processed event ids with a TTL; skip duplicates.
  4. Idempotent processing. Even without dedupe, processing the same event twice shouldn't cause double effects — use natural keys and check-before-insert.
  5. Timestamp check. Reject webhooks older than 5 minutes (replay attack defense); most providers include the timestamp in the signed header.

Common use cases: payment notifications (Stripe, Braintree), version-control events (GitHub, GitLab pushes), collaboration (Slack messages, Jira updates), e-commerce (Shopify orders), communication (Twilio SMS delivery). Anywhere a third-party system needs to tell you something happened.

Operational challenges: unreliable delivery (providers retry with backoff but can give up), your endpoint must have a stable URL (tunneling via ngrok/localtunnel for dev; a dedicated subdomain in prod), observability (log every webhook with provider, event id, processing outcome). Tools like Hookdeck and svix handle retries, signing, and delivery observability as a managed layer.

Question 30

What is the significance of the HTTP OPTIONS method in CORS?

In CORS, the OPTIONS method powers the preflight request — a check the browser sends before the real cross-origin request, asking "am I allowed to make this?" The server's response to OPTIONS tells the browser whether to proceed.

// 1. Browser-generated preflight — sent BEFORE your fetch
OPTIONS /api/orders HTTP/1.1
Origin: https://app.example.com
Access-Control-Request-Method:  POST
Access-Control-Request-Headers: Content-Type, Authorization

// 2. Server's preflight response — grants (or denies) permission
HTTP/1.1 204 No Content
Access-Control-Allow-Origin:  https://app.example.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Max-Age:       86400               // cache 24h

// 3. Browser now proceeds with the real POST

When preflight fires: automatically, for any cross-origin request that isn't simple — specifically, requests that use non-standard methods (PUT, DELETE, PATCH), custom headers (Authorization, X-Request-Id), or Content-Type: application/json. "Simple requests" (GET, HEAD, or POST with form-encoded body and no custom headers) skip preflight.

Why preflight exists: it protects legacy servers that never expected cross-origin traffic — an attacker's JavaScript could otherwise cause the victim's browser to send a DELETE to their bank's old API. Preflight gives modern servers an opt-in mechanism to say "yes, I'm aware of cross-origin requests and here's what's allowed."

Performance considerations:

  • Every non-simple cross-origin request becomes two network round-trips — this doubles latency if you don't cache the preflight.
  • Access-Control-Max-Age: 86400 tells browsers to cache the preflight answer for 24 hours. Without it, every unique request preflights.
  • Avoid per-request custom headers when possible — each new header bust the cache.

Common bugs: (a) auth middleware running before CORS middleware, so OPTIONS returns 401 and the browser aborts; (b) Access-Control-Allow-Origin: * combined with credentials: true (browsers reject); (c) custom response headers not listed in Access-Control-Expose-Headers, invisible to fetch.

Question 31

How do you design a resilient API that can handle failures gracefully?

Resilient APIs accept that dependencies will fail and degrade gracefully rather than cascading failures. The patterns are well-established; applying them consistently is the hard part.

PatternPrevents
Timeouts everywhereOne slow dependency hanging your whole system
Retries with exponential backoff + jitterTransient failures; thundering herds on recovery
Circuit breakersRepeated calls to a known-broken dependency
BulkheadsOne slow tenant consuming all connection slots
Fallbacks & graceful degradationHard failures when a partial response is useful
IdempotencyDuplicate side effects from retries
Health checks & readiness probesTraffic to not-yet-ready instances
Load sheddingCascading failure from overload

Concrete essentials:

  • Every network call has a timeout. Default connect + read timeout of 2-5 seconds; never "no timeout."
  • Retry only idempotent operations (GET, PUT, DELETE, or POST with idempotency keys). Retry on transient errors (5xx, 429, network), not 4xx.
  • Exponential backoff with full jitter — 1s, 2s, 4s, 8s ± jitter. Without jitter, all clients retry simultaneously when a service recovers, re-breaking it.
  • Circuit breakers (opossum, resilience4j) open after N failures, fast-fail for a cool-off period, then half-open to test recovery. Prevents wasting resources on broken dependencies.
  • Bulkhead isolation — separate thread/connection pools per downstream dependency. A slow payments service can't starve your user lookups.
  • Fallback content — if the recommendations service is down, show popular items. If comments fail, hide them. 200 OK with degraded content > 500.

Testing resilience: chaos engineering — randomly kill pods, inject latency, drop connections. Tools: Chaos Mesh, AWS Fault Injection Simulator, Gremlin, toxiproxy. If you haven't tested your failure paths, they don't work. Service mesh (Istio, Linkerd) can enforce timeouts, retries, and circuit breakers at the infrastructure layer — takes the burden off application code.

Question 32

Explain how to handle API deprecation and version transitions.

API deprecation is the planned lifecycle: the API reaches end-of-life, but thousands of consumers depend on it. Done well, no one is surprised; done poorly, you break customers and lose trust.

The phased deprecation workflow:

  1. Ship v2 alongside v1. Never remove v1 the same day v2 launches. Run both in parallel.
  2. Announce early. Email all affected customers. Post a migration guide. Aim for 6-12 months notice for public APIs (Stripe gives years). Internal APIs can be shorter but still need lead time.
  3. Signal via headers. RFC 8594 Sunset + RFC draft Deprecation headers on v1 responses: 
    HTTP/1.1 200 OK
Deprecation: Wed, 01 Jan 2027 00:00:00 GMT
Sunset: Wed, 01 Jul 2027 00:00:00 GMT
Link: <https://docs.example.com/migrate/v2>; rel="deprecation"
  4. Track usage per-client. Who's still calling v1? Which endpoints? Reach out directly to remaining heavy users.
  5. Escalate the signal. As sunset approaches, add warning responses: slow v1 gradually, add banner in docs, log warnings in dev tools.
  6. Staged shutdown. Enforce brownouts before shutdown (return 410 Gone for 1 hour a day, then 2, escalating). Clients who missed the memo discover it before the hard cutoff.
  7. Final removal. Return 410 Gone with a migration link.

Avoiding deprecation entirely:

  • Additive evolution. Most changes — new fields, new endpoints — can be done without breaking clients. Reserve deprecation for genuinely incompatible shifts.
  • Versioned fields. Within one endpoint, accept both old and new field shapes; normalize internally.
  • Facade pattern. Keep the v1 URL serving; internally translate to v2 logic. Two contracts, one implementation.

Critical cultural discipline: know exactly what counts as breaking (remove/rename field, tighten validation, change type) vs safe (add optional field, new endpoint). Enforce in CI with oasdiff or schema comparators. Teams that ship breaking changes without the process erode external trust fastest — a reputation that takes years to rebuild.

Question 33

What are the strategies for optimizing API performance?

API performance optimization proceeds from architecture (what work we're doing) to implementation (how fast we do it). Architecture changes typically 10-100x; implementation changes typically 10-50%.

LayerTechnique

Where to focus, in priority order:

  1. Measure first. Profile with APM (Datadog, New Relic, Honeycomb); find the slowest 10% of requests. Optimize them, not averages.
  2. Eliminate N+1 queries. A list endpoint fetching 50 related records in 50 separate queries is the #1 silent killer. Batch with IN clauses, join, or use DataLoader.
  3. Add the missing index. One unindexed WHERE clause turns a 1ms query into a 10-second table scan. EXPLAIN ANALYZE is your friend.
  4. Cache at the right layer. HTTP/CDN for public data; Redis for query results; in-process LRU for hot lookups. Each layer eliminates 10x the load on the next.
  5. Parallelize I/O. await sequentially when there's no data dependency is pure waste. Use Promise.all or allSettled.
  6. Shrink payloads. Return only needed fields; gzip is free.

Question 34

How do you implement a circuit breaker pattern in an API?

A circuit breaker wraps calls to a downstream dependency and fails fast when that dependency is broken. Without one, retries and slow calls cascade — a failing payments service takes down order creation, then checkout, then the whole product.

Three states, like a physical circuit breaker:

  • Closed — normal operation; calls flow through.
  • Open — dependency is failing; all calls fail immediately without even trying. No resources wasted.
  • Half-open — after a cool-off period, allow one probe call. Success → closed; failure → open again.
import CircuitBreaker from "opossum";

const breaker = new CircuitBreaker(
  (orderId) => fetch(`http://payments/charge/${orderId}`, { signal: AbortSignal.timeout(3000) }),
  {
    timeout:             3000,           // treat slow as failure
    errorThresholdPercentage: 50,        // open after 50% failure in window
    resetTimeout:        30_000,         // try half-open after 30s
    rollingCountTimeout: 10_000,         // measure over 10s windows
    volumeThreshold:     10,             // min 10 requests before opening
  }
);

// fallback on open — return a sensible default
breaker.fallback(() => ({ status: "queued", message: "Payments degraded" }));

breaker.on("open",      () => log.warn("payments circuit OPEN"));
breaker.on("halfOpen",  () => log.info("payments circuit HALF-OPEN"));
breaker.on("close",     () => log.info("payments circuit CLOSED"));

// usage
const result = await breaker.fire(orderId);

Tuning parameters:

  • Error threshold — 50% is a common default; too low causes flapping, too high delays protection.
  • Volume threshold — don't open based on "2 of 3 failed"; statistical noise. Require ~10 requests.
  • Reset timeout — how long before probing. 30s-60s for most cases; matches typical recovery times.
  • Timeout — shorter than the dependency's typical p99 + margin; slow counts as failed.

Fallback strategy is critical: a circuit breaker that only fails fast is half the pattern. Define what to do when open: serve cached content, skip the non-essential feature, queue the work, return a sensible default. Without a fallback, your API just returns 503 faster.

Service mesh alternative: Istio, Linkerd, Envoy implement circuit breakers at the sidecar, no application code needed. Good for systems that already run a mesh; the application-level library is right for finer control and bespoke fallbacks.

Question 35

What is API gateway pattern, and what problems does it solve?

The API gateway pattern consolidates cross-cutting concerns (auth, routing, rate limiting, observability) into a single edge layer that fronts a collection of backend services. It solves the problems that emerge when multiple services need to be exposed to external clients consistently.

Problems it solves:

  • Client complexity. Without a gateway, clients must know every service URL, call them in parallel, aggregate responses, and handle per-service auth. The gateway abstracts this — one endpoint, one auth flow.
  • Duplicated cross-cutting concerns. Every service implementing auth, rate limiting, logging separately is wasted effort and inconsistency risk. The gateway centralizes.
  • Tight coupling to internal topology. Clients shouldn't care whether users and profiles are one service or two. Gateway routes URLs abstractly.
  • Security surface. Services can run on private networks; only the gateway is public.
  • Protocol bridging. Public REST → internal gRPC, GraphQL, or Thrift.
  • Response aggregation (BFF). One client request → parallel calls to 5 services → composed response.
  • Inconsistent monitoring. Every service logs differently; the gateway emits uniform access logs for all external traffic.
VariantPurpose
Single gatewayOne for all clients (web, mobile, partners)
Backend for Frontend (BFF)One gateway per client type, tailored to its needs
Edge gatewayPlus caching and CDN-level logic near users

Anti-patterns that defeat the purpose:

  • Business logic in the gateway. Turns it into a distributed monolith. Keep orchestration thin.
  • Single point of failure. Run multiple gateway instances behind a load balancer; monitor its own health.
  • Performance bottleneck. Every request adds a hop; gateway must be high-performance and horizontally scalable.
  • Monolith gateway config. Config for all services managed in one place becomes impossible to own — split by team/domain.

Complement with a service mesh (Istio, Linkerd) for east-west traffic — the gateway handles client-to-service (north-south); the mesh handles service-to-service. In 2026, this layered approach is the default for microservice architectures.

Question 36

How do you handle API quotas and rate limits for different users?

Quotas and rate limits aren't just about preventing abuse — they're how you enforce fairness, enable tiered pricing, and protect downstream dependencies. Different users should get different treatment based on plan, role, or business rules.

DimensionExample policy
Plan tierFree: 100/hr; Pro: 10k/hr; Enterprise: unlimited
User roleAdmin: 10× normal user
Endpoint costSearch: 10 credits; GET: 1 credit; ML call: 100 credits
Temporal windowPer-second (throttle), per-minute, per-hour (quota), per-month (billing)
ConcurrencyMax 5 in-flight ML calls per user

Implementation pattern — token buckets per key:

// Redis-backed token bucket, keyed by (user + tier + endpoint)
async function consumeTokens(userId, endpoint, cost = 1) {
  const plan   = await getPlan(userId);                 // free/pro/enterprise
  const limit  = LIMITS[plan][endpoint] || LIMITS[plan].default;
  const key    = `rl:${plan}:${userId}:${endpoint}`;

  const script = `
    local bucket  = redis.call('HMGET', KEYS[1], 'tokens', 'lastRefill')
    local tokens  = tonumber(bucket[1]) or tonumber(ARGV[1])
    local last    = tonumber(bucket[2]) or tonumber(ARGV[2])
    local now     = tonumber(ARGV[2])
    local refill  = (now - last) * tonumber(ARGV[3])
    tokens = math.min(tonumber(ARGV[1]), tokens + refill)
    if tokens >= tonumber(ARGV[4]) then
      tokens = tokens - tonumber(ARGV[4])
      redis.call('HMSET', KEYS[1], 'tokens', tokens, 'lastRefill', now)
      redis.call('EXPIRE', KEYS[1], 3600)
      return {1, tokens}
    else
      return {0, tokens}
    end
  `;
  const [allowed, remaining] = await redis.eval(script, 1, key,
    limit.capacity, Date.now() / 1000, limit.refillPerSec, cost);
  return { allowed: !!allowed, remaining };
}

Essential production features:

  • RFC 9331 headers (RateLimit-Limit, RateLimit-Remaining, RateLimit-Reset) on every response — well-behaved clients self-pace.
  • Retry-After on 429 — clients know when to retry; libraries honor it.
  • Atomic Redis Lua scripts — naive GET/SET races let users burst over the limit.
  • Multiple limits per request — per-second throttle + per-hour quota + per-month billing; fail at the tightest.
  • Soft vs hard limits — soft triggers warning + email; hard rejects. Enterprise plans often want soft only.

Monetization angle: quota overages become upsell triggers. Track approach-to-limit in 80%/90%/100% buckets; notify via email; offer higher-tier upgrade. API management platforms (Kong, Apigee) do this out of the box.

Question 37

Describe the role of the HTTP Accept header in content negotiation.

The Accept header is the client's side of HTTP content negotiation — telling the server what media types (and optionally languages, encodings) it can handle. The server picks the best match from what it can produce.

// client expresses preferences with quality values
GET /reports/q4
Accept: application/json;q=1.0, text/csv;q=0.8, application/xml;q=0.3, */*;q=0.1

// server picks best match; advertises its choice
HTTP/1.1 200 OK
Content-Type: application/json
Vary: Accept                         // tells caches that responses differ by Accept

Quality values (q-weights): default to 1.0 (most preferred). The server picks the highest-q type it can produce. */* matches anything; type/* matches any subtype. If no match, server responds 406 Not Acceptable.

HeaderNegotiates
AcceptMedia type (application/json, text/html, image/webp)
Accept-LanguageHuman language (en-US, de, ja)
Accept-EncodingCompression (gzip, br, zstd)
Accept-CharsetCharacter encoding (nearly obsolete; UTF-8 is standard)

Use cases for content negotiation in REST:

  • Multi-format responses — same URL serves JSON to apps, CSV to spreadsheets, HTML to browsers.
  • API versioning via media types — Accept: application/vnd.api.v2+json. GitHub used this for its v3 API. Keeps URLs clean; less discoverable than URL versioning.
  • Hypermedia formats — application/hal+json, application/problem+json, application/activity+json (ActivityPub) all negotiated via Accept.
  • Image format optimization — CDN checks Accept: image/webp,image/avif and serves the best supported format.

The Vary header is critical: when responses depend on the Accept header (or any request header), emit Vary: Accept so shared caches don't serve the wrong variant. Missing Vary is a classic CDN bug — one user requests JSON, the next gets the cached JSON response even though they asked for CSV.

Practical 2026 reality: most APIs default to JSON regardless of Accept, reserving content negotiation for specific cases (image formats, versioning, multiple export formats). Simplicity wins; supporting 5 formats "just because" creates maintenance burden without clear benefit.

Question 39

What is the role of API keys, and how do you manage them securely?

API keys identify the calling application (not the user) and gate access to your API. They're simple enough to leak easily and powerful enough to cause real damage — hence the emphasis on secure management.

Key issuance & format:

  • Prefix the key for identification: sk_live_ (secret/live), pk_live_ (publishable), sk_test_ (test). Humans and tools can spot leaked keys instantly.
  • Use cryptographic randomness (256 bits / 32 bytes from crypto.randomBytes). Never sequential ids or simple UUIDs treated as secrets.
  • Show the raw key once, on creation. The dashboard can never re-show it; users who lose it must regenerate.

Storage — never plaintext:

// DB stores only the HASH — a leaked DB dump can't be replayed
const rawKey    = `sk_live_${crypto.randomBytes(32).toString("base64url")}`;
const keyHash   = crypto.createHash("sha256").update(rawKey).digest("hex");

await db.apiKey.create({
  data: { keyHash, userId, name: "Production", scopes: ["read"], lastUsedAt: null },
});

// on incoming request — hash and look up
const record = await db.apiKey.findUnique({
  where: { keyHash: sha256(req.headers["x-api-key"]) },
});

Management essentials:

  • Scopes. Each key has minimum permissions (read-only, write, admin). Damage from a leak is bounded.
  • Rotation. Allow multiple keys per account so rotation doesn't cause downtime — issue new, migrate, revoke old.
  • Revocation. A single column update invalidates a leaked key instantly.
  • Expiry. Optional TTL on keys; forces rotation hygiene.
  • Last-used tracking. Auto-alert on keys not used in 90 days (potential candidates to remove).
  • Anomaly detection. Sudden volume spike, new IP range, new user agent — flag for review.

Transport & placement: headers only (X-API-Key or Authorization: Bearer) — never query strings (end up in logs). HTTPS always. Never accept keys in redirects. Secret-scanning tools (GitHub secret scanning, TruffleHog, Gitleaks) auto-detect the prefixed format in committed code; enable them in CI.

Public vs secret keys: if you have a browser-facing SDK (Stripe publishable keys), distinguish them explicitly in the prefix — public keys are safe to ship to clients; secret keys are server-only. Enforce at the API layer: secret-key operations reject if called with a public key.

How do you implement a multi-tenant architecture in an API?

Multi-tenant APIs serve many customers (tenants) from a single deployment. The design decisions are fundamentally about data isolation and noisy-neighbor protection.

Tenant resolution: the API must identify the tenant on every request. Common methods:

  • Subdomain — acme.api.example.com. Classic SaaS pattern; visible to users.

  • Scope every query by tenant id. A single forgotten WHERE tenant_id = ? leaks everything. Use Postgres Row-Level Security (RLS) or a repository layer that enforces it automatically. Never trust handler-level filtering alone.
  • Rate limit per tenant. One customer's loop shouldn't starve others. Redis keys: rl:tenant_123:endpoint.

  • Connection pooling. Avoid per-tenant DB pools (N * pool_size connections); share the pool, set SET app.tenant_id per transaction for RLS.
  • Resource quotas. Storage limits, row counts, compute usage tracked per tenant.

Compliance angle: regulated customers (healthcare, finance, EU) often demand database-level isolation. Offering tiered isolation (shared for SMB, dedicated for enterprise) is a common go-to-market strategy.

Question 42

How do you handle large file uploads and downloads in an API?

Large file handling — both directions — is fundamentally about never buffering the whole file in memory or on a single hop. Every technique aims at that: streaming, direct S3 transfers, chunking, and offloading.

ScenarioTechnique
Uploads < 10 MBMultipart form + multer/busboy; stream to disk or memory
Uploads 10 MB - 5 GBS3 multipart upload with client-side chunked PUT; server signs each part
Uploads > 5 GBtus protocol (resumable) or S3 multipart with resumability

Downloads large / cloudPre-signed URL redirect; client downloads direct from S3/CloudFront

Pre-signed URLs — the preferred pattern for both directions:

// upload init — client gets a signed PUT URL; uploads direct to S3
const url = await getSignedUrl(s3,
  new PutObjectCommand({ Bucket, Key, ContentType }),
  { expiresIn: 3600 });
// bytes go from client → S3, never through your API

// download — client gets a signed GET URL
const url = await getSignedUrl(s3,
  new GetObjectCommand({ Bucket, Key,
    ResponseContentDisposition: `attachment; filename="${name}"` }),
  { expiresIn: 300 });

Why this pattern wins:

  • Your API stays small. Node memory is O(1) regardless of file size; no connection tied up for minutes.
  • Throughput. S3 / CloudFront serve at 10+ Gbps; your Node process does not.
  • Cost. Egress from S3 with a CDN is often cheaper than egress from your app servers.
  • Range requests, resume, caching. S3 handles these natively; implementing them yourself is painful.

For streaming downloads you generate on-the-fly (CSV export, PDF generation), use Node streams with backpressure (res.write() return value + drain event) or pipeline. Never build the full file in memory — 10 concurrent exports of 500 MB each will OOM your server.

Security: validate uploaded content with magic bytes (file-type), not just MIME headers; cap upload size aggressively; use random filenames (path traversal defense); scan uploads for malware (ClamAV) if they'll be redistributed.

  • Dashboards by audience — Retool or Metabase for product/biz; Grafana/Datadog for engineering.

Full per-request logs can be sampled (1%) or head-sampled at high volume, while keeping 100% of errors. Honeycomb / Datadog APM automate this.

Question 45

How do you design an API to be backward compatible?

Backward compatibility means existing clients keep working as your API evolves. It's a discipline of additive change, tolerant readers, and explicit versioning when breaks are unavoidable.

Safe changes (no version bump)Breaking changes (need version bump or long deprecation)
Add new optional field to responseRemove field from response
Add new endpointRemove endpoint
Add optional request parameter with defaultRename field
Loosen validation (accept more inputs)Change field type
New status code for existing case (200 → 201 for CREATED)Make optional field required
Deprecate field (keep returning it)Change URL structure

Design practices that preserve compatibility:

  • Plural everything. Return arrays even when there's one item; easier to add batch support later without breaking.
  • Wrap scalars. Return { "total": 42 } not bare 42 — add sibling fields later ({ "total": 42, "unit": "USD" }).
  • Consistent error shape — don't change error bodies between endpoints or versions.
  • Forward-compatible enums. Clients should tolerate unknown enum values (log + treat as default); servers don't add values clients can't handle.
  • Stable pagination. Once nextCursor is added to responses, never remove it or clients stop iterating correctly.
  • Nullable vs required. Be explicit and stable. Changing "optional returns null" to "omitted entirely" breaks tolerant readers that didn't handle absence.

Tolerant reader principle: clients should ignore unknown fields rather than error. Document this expectation. Libraries like zod.passthrough() and JSON parsers default to tolerant behavior; strict validators (Swagger clients with additionalProperties: false) bite users on server updates.

Automation that enforces it:

  • oasdiff in CI — blocks PRs that introduce breaking changes.
  • GraphQL Inspector / buf — same for GraphQL and protobuf.
  • Consumer-driven contracts (Pact) — fail deploys if consumers' expectations are violated.
  • Canary deploys — catch subtle issues before full rollout.

When you must break: parallel versions, announce deprecation with Sunset headers, communicate via email + changelog, offer migration guides, monitor v1 usage, sunset only after it's near-zero. Breaking consumers without notice is how APIs lose trust; regaining it takes years.

Question 47

Explain the concept of request validation in APIs and how to implement it.

Request validation is the first line of defense: reject malformed input before it touches business logic. Done well, it blocks injection attacks, data corruption, mass-assignment escalation, and invalid state transitions — in one consistent layer.

Validation happens at three levels:

LevelPurpose
SyntacticType, length, format (valid email, UUID, ISO date)
SemanticCross-field rules (from <= to, amount > 0)
BusinessStateful rules (user owns this resource, stock available, not duplicate)

Implementation with Zod (factory-style middleware):

const CreateOrderSchema = z.object({
  items: z.array(z.object({
    productId: z.string().uuid(),
    qty:       z.number().int().min(1).max(100),
  })).min(1).max(50),
  shippingAddressId: z.string().uuid(),
  couponCode: z.string().regex(/^[A-Z0-9]{4,12}$/).optional(),
}).strict();                              // reject unknown keys — mass-assignment defense

function validate(schemas) {
  return (req, res, next) => {
    for (const loc of ["body", "query", "params"]) {
      if (!schemas[loc]) continue;
      const parsed = schemas[loc].safeParse(req[loc]);
      if (!parsed.success) {
        return res.status(400).json({
          error: "Validation failed",
          issues: parsed.error.issues.map((i) => ({
            path:    i.path.join("."),
            message: i.message,
            code:    i.code,
          })),
        });
      }
      req[loc] = parsed.data;              // replace with cleaned, coerced data
    }
    next();
  };
}

app.post("/orders", requireAuth,
  validate({ body: CreateOrderSchema }),
  createOrderHandler);

Critical techniques:

  • .strict() rejects unknown fields — prevents mass-assignment (user sends role: "admin" trying to escalate).
  • Coercion — z.coerce.number(), z.coerce.date() handle query strings that arrive as strings.
  • Return all errors at once — better UX than one-at-a-time.
  • Bound string lengths + array sizes — prevent DoS from enormous payloads.
  • Cross-field rules via .refine() — from <= to, if foo then bar required.
  • Validate query AND params AND body — query strings are user input too.
  • Standardized error shape — RFC 7807 Problem Details or a house format applied uniformly.

Schema as source of truth: if you use OpenAPI, generate Zod schemas (or vice versa) with tools like openapi-zod-client, orval, or zod-to-openapi. Single source of truth keeps docs, types, mock data, and validation in lockstep. This is the 2026 gold standard — schema once, verify everywhere (runtime validation + TS types + OpenAPI docs + generated clients).

Question 48

How do you implement OAuth 2.0 for third-party API integrations?

OAuth 2.0 for third-party integration means your users grant your app access to their data at a third-party service (Google, GitHub, Slack). The canonical flow is Authorization Code + PKCE (mandatory in OAuth 2.1).

// 1. user clicks "Connect Google Calendar" → redirect to Google
GET https://accounts.google.com/o/oauth2/v2/auth?
  client_id={YOUR_CLIENT_ID}&
  redirect_uri=https://myapp.com/oauth/google/callback&
  scope=https://www.googleapis.com/auth/calendar.readonly&
  response_type=code&
  state={csrf_token}&
  code_challenge={s256_of_verifier}&
  code_challenge_method=S256&
  access_type=offline&                        // for refresh token
  prompt=consent                              // force consent to get refresh on re-auth

// 2. user consents → Google redirects back with an auth code
GET https://myapp.com/oauth/google/callback?code={AUTH_CODE}&state={csrf_token}

// 3. your server exchanges code for tokens (server-to-server, client_secret included)
POST https://oauth2.googleapis.com/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code&
code={AUTH_CODE}&
client_id={YOUR_CLIENT_ID}&
client_secret={YOUR_CLIENT_SECRET}&

{ "access_token": "ya29.a0...", "refresh_token": "1//04...",
  "expires_in": 3599, "scope": "...", "token_type": "Bearer" }

Critical safety mechanisms:

  • PKCE (mandatory in 2.1) prevents authorization-code interception.
  • state parameter prevents CSRF on the callback; store it session-scoped and verify.
  • Exact redirect_uri match — registered with the IdP; any mismatch fails. Wildcards are an open-redirect hazard.
  • scope minimization — ask for only what you need; users are more likely to approve narrow scopes.
  • Incremental authorization — ask for additional scopes only when actually needed, not upfront.

Token storage: store access_token and refresh_token encrypted per-user in your DB (KMS-encrypted columns). Never log them. On every third-party API call, check if access_token is near expiry; if so, refresh first. Handle 401 by refreshing once, then failing if still invalid (user revoked access).

Revocation & disconnect: implement a user-facing "disconnect Google" flow that calls the revocation endpoint (https://oauth2.googleapis.com/revoke) and deletes stored tokens. GDPR and user trust demand this.

In 2026: use openid-client (Node), authlib (Python), better-auth for provider-agnostic integration — hand-rolling is a footgun. For common providers (Google, GitHub, Slack), their official SDKs simplify the auth + API usage.

How do you handle data serialization and deserialization in APIs?

Serialization converts in-memory objects to bytes for transport; deserialization goes the other way. Choice of format and handling affects performance, compatibility, and security.

FormatShapeSweet spot
JSONText; language-agnosticPublic REST APIs; universal default
Protocol BuffersBinary; schema-firstgRPC; service-to-service; 5-10× smaller than JSON
MessagePackBinary JSON-equivalentSchema-less binary; modest size win over JSON
AvroBinary; schema embedded or registryEvent streams (Kafka); schema evolution
Cap'n Proto / FlatBuffersBinary; zero-copyVery high-performance games, trading
CBORBinary JSON-likeIoT, constrained devices, WebAuthn payloads

Serialization concerns in APIs:

  • Dates. JSON has no native date. Use ISO 8601 strings (2026-04-19T12:00:00Z). Agree on timezone convention; always UTC at the wire; convert for display client-side.
  • Numbers. JavaScript Number is 64-bit float — integers beyond 2^53 lose precision. Use strings for large ids ("1234567890123456789") or BigInt in systems that support it.
  • Binary data. Base64-encode in JSON (33% size overhead) or use multipart/form-data for file uploads.
  • Null vs absent. { "field": null } and {} differ semantically — the first explicitly clears; the second leaves alone (JSON Merge Patch semantics). Document which you mean.
  • Unknown fields. Be a tolerant reader — ignore fields you don't recognize. Enables additive evolution.

Deserialization security:

  • Never trust input. Validate types, bounds, required fields before use.
  • Prototype pollution — attackers include __proto__ or constructor keys; strip them before parsing into objects. Modern parsers + Zod reject by default.
  • Depth bombs — deeply nested JSON can exhaust stack. Cap depth and size at the edge.
  • Unsafe formats. Python pickle, Java serialization, PHP unserialize — never deserialize untrusted input. Known vectors for RCE.

Practical stack for 2026: JSON externally (universal), Protocol Buffers internally where performance matters (gRPC between services), Zod/AJV/Pydantic at the edge for validation + type safety. Question 52

How do you design an API for real-time data updates?

Real-time APIs push data to clients as it becomes available, instead of requiring clients to poll. The right mechanism depends on whether updates are one-way or bidirectional, client count, and infrastructure constraints.

Long pollingServer → clientFallback; works everywhere; high overhead

Pick SSE when: server-to-client is all you need. Plain HTTP, auto-reconnect built into browsers (EventSource), survives corporate proxies, easy to fan-out with Redis pub/sub. Resumable via Last-Event-Id.

Pick WebSocket when: true bidirectional is required (chat with typing indicators, collaborative editing). Be ready for the operational cost.

Architecture essentials:

  • Authentication at handshake — don't accept connections and then check; reject unauthorized at connect time.
  • Heartbeat + auto-disconnect — send pings every 20-30s; clients that don't respond are disconnected (mobile networks drop silently).
  • Horizontal fan-out. If user's socket is on instance A and the event originates on instance B, you need a shared pub/sub (Redis, NATS, Kafka) to route events.
  • Backpressure. A slow client with a fast producer will exhaust your socket buffer. Drop, batch, or coalesce messages when buffer grows.
  • Resumability. Reconnecting clients need to catch up; Last-Event-Id header in SSE; sequence number + replay buffer in WS.
  • Scale constraints. Each Node process can handle 50k-100k concurrent sockets; above that, multiple processes + sticky routing or a dedicated platform (Ably, Pusher, AWS AppSync).

Modern reality: most "real-time" user features can be SSE; use WebSocket only when bidirectional is essential. Managed platforms (Ably, Pusher, Supabase Realtime) handle all the operational complexity for a fee — worth it unless you're operating at massive scale or have tight integration needs.

Question 54

How do you implement secure API gateways?

Securing an API gateway means hardening the single point through which all traffic flows. Because it's the edge, vulnerabilities here expose every service behind it; getting it right is leverage.

The defensive layers, outside in:

  • WAF / DDoS protection — Cloudflare, AWS WAF, Fastly in front of the gateway. Blocks obvious attacks (SQLi probes, XSS patterns, known bot networks) and absorbs volumetric attacks.

  • Authorization — coarse-grained policy checks (plan tier, role) at the gateway; fine-grained ownership checks at the service level.
  • Rate limiting & throttling — per-IP, per-user, per-endpoint; stricter on auth endpoints; tighter for unauthenticated traffic.
  • Input validation — schema-validate requests against OpenAPI; reject malformed before reaching services.
  • Body size caps — reject massive payloads (DoS defense).
  • Header sanitization — strip internal headers from incoming requests; strip sensitive headers from outgoing responses.
  • CORS policy — strict allowlist; no * with credentials.
  • Security headers — X-Content-Type-Options: nosniff, X-Frame-Options, Referrer-Policy, Content-Security-Policy (when serving HTML).

Question 55

What are the benefits of using gRPC over traditional REST APIs?

gRPC's benefits over REST are concentrated in service-to-service communication inside a datacenter. It's engineered for performance, typed contracts, and streaming — areas where REST traditionally compromises.

Concrete benefits:

  • HTTP/2 multiplexing. Many RPCs over one connection; no head-of-line blocking at the HTTP layer (though TCP-level HOL remains; HTTP/3 fixes that too).
  • Strong typing end-to-end. Compiler catches parameter mismatches that are runtime 400s in JSON.
  • Streaming as first-class. Server streams (log tail), client streams (batch uploads), bidirectional streams (chat) — all one primitive.

  • Browsers. gRPC needs gRPC-Web + a proxy (Envoy); adds operational complexity.
  • Public APIs. "Just call our gRPC service" is a higher barrier than "curl this URL". REST is the universal consumer skill.
  • Debuggability. Human-readable JSON in tcpdump/browser DevTools vs binary protobuf.
  • HTTP caching. CDNs understand HTTP caching for REST; gRPC responses aren't cached.
  • Simple CRUD. The gain is marginal; the tooling cost is real.

Hybrid approach (common in 2026): gRPC internally (service-to-service), REST/GraphQL at the edge (via a gateway translating). You get gRPC performance inside and REST universality outside. Adaptive throttlingTighten limits when downstream latency risesBack-pressure propagation

Hard vs soft throttling:

  • Hard (429 immediately) — simple, cheap, puts back-off burden on the client.
  • Soft (leaky bucket with queueing) — client gets a response (maybe slow) and never sees an error. More server memory; can cascade if the queue grows unbounded.

Critical production details:

  • Always emit RFC 9331 headers (RateLimit-Limit, RateLimit-Remaining, RateLimit-Reset) so well-behaved clients self-pace before you have to throttle.
  • Cap queue depth. Unbounded queues become unbounded memory; reject when queue exceeds a threshold rather than OOMing.
  • Per-dependency throttling — wrap outbound third-party calls in a throttler so your system-wide usage stays under their cap.
  • Shed load under distress — return 503 or lower quality responses when downstream health drops (adaptive throttling via circuit-breaker-like logic).

Ecosystem: AWS API Gateway / Kong / Envoy all expose throttling primitives declaratively. Application-level libraries: bottleneck (Node, full-featured), p-throttle, limiter. For global coordination across instances: Redis with atomic Lua scripts, or a dedicated tool like Stripe's open-sourced fly.io/throttled. The default reflex — reject at the gateway — works for 90% of cases; switch to queueing only when clients can't easily retry.

Question 57

Describe the process of implementing a federated API.

A federated API unifies multiple independently-owned APIs behind a single graph/endpoint, so clients see one coherent API while teams ship at their own pace. It's the architectural answer to "how do 50 teams serve a product UI without a monolith?"

The canonical implementation — GraphQL Federation (Apollo Federation, graphql-mesh):

# Users subgraph — owned by the users team
type User @key(fields: "id") {
  id: ID!
  email: String!
  name: String!
}

# Orders subgraph — owned by the orders team; extends the User type
extend type User @key(fields: "id") {
  id: ID! @external
  orders: [Order!]!        # added by orders team, no users-team work required
}

type Order @key(fields: "id") {
  id: ID!
  userId: ID!
  total: Float!
}

A gateway (Apollo Router, GraphQL Mesh, Wundergraph) composes the subgraphs at runtime, validates the merged schema, and plans queries across them. Clients hit one endpoint; subgraphs stay isolated.

The process:

  1. Define ownership boundaries — each subgraph owns a domain (users, orders, catalog). One type may be extended across subgraphs using keys.
  2. Each team builds a subgraph — independently deployed, independently scaled.
  3. Publish schemas to a registry (Apollo Studio, Grafbase) — composition is validated on every change; breaking changes blocked.
  4. Gateway composes and routes — single endpoint executes queries across subgraphs in parallel.

REST-based federation uses a gateway that aggregates: /orders/42 fetches from the orders service, also calls users service for the author, stitches a response. Less elegant than GraphQL Federation but simpler operationally.

Benefits: teams ship independently, no cross-team coordination for additive changes; client sees one API; ownership clear; schema is a contract enforced in CI.

Challenges: query planning — federated queries can fan out to many subgraphs; watch for N+1 across subgraph boundaries (Apollo's @key batching helps). Distributed tracing is mandatory — which subgraph is slow? Policy consistency — auth, rate limiting, observability must be uniform; put them at the gateway or as a shared lib. Breaking-change detection — one team renames a field, another breaks. Registries with CI checks prevent this. For startups, federation is overkill; for 10+ teams on one product, it's often the only sustainable architecture.

Question 58

How do you design an API for high availability?

High availability (HA) means the API keeps serving traffic despite hardware failures, software bugs, dependency outages, and traffic spikes. The usual goal is stated as "nines" of uptime: 99.9% (~8 hrs/yr downtime), 99.99% (~52 min/yr), 99.999% (~5 min/yr — rare outside hyperscale).

The core patterns, layered:

  • Stateless compute + horizontal scaling. Any instance handles any request; losing one is imperceptible. Run N+1 or N+2 instances so capacity survives failures.
  • Multi-AZ deployment. Instances spread across availability zones; one AZ outage doesn't take the service down.
  • Multi-region for the highest availability; with health-check-based DNS failover or active-active load balancing.
  • Database HA. Managed Postgres (Aurora, Cloud SQL, Azure DB) with automatic failover; read replicas for scaling reads; async replication across regions.
  • Caching layer. Redis with replication (Sentinel or Cluster); short TTL reduces impact of primary outage.
  • Circuit breakers + fallbacks. Don't let downstream failures cascade; serve degraded responses.
  • Async processing. Queues decouple ingestion from processing; a slow downstream doesn't back up the API.
  • Graceful shutdown. SIGTERM → stop accepting new requests, drain in-flight, then exit. Rolling deploys without dropping requests.
  • Load balancer health checks. Unhealthy instances removed from rotation quickly; readiness probes distinct from liveness.

Question 59

Explain the concept of API rate limiting and its implementation.

Rate limiting caps how many requests a client can make in a time window. It's one of the most universally deployed API patterns because it defends against four orthogonal threats: abuse (scrapers, credential stuffing), accidents (buggy retry loops), noisy neighbors (one tenant monopolizing capacity), and cost overruns (downstream API charges).

The algorithms, from simplest to most accurate:

AlgorithmMechanismTrade-off
Fixed window counterCount per window; reset at boundarySimple; 2× burst at boundary
Sliding window logList of timestamps; count in last N secondsExact; memory-heavy
Sliding window counterWeighted average of two fixed windowsGood balance; <1% error
Token bucketBucket refills at rate R; each request takes 1 token; bucket size allows burstsNatural-feeling; user can burst when they've been quiet
Leaky bucketQueue drains at fixed rate; smooths burstsAdds latency; good for downstream protection

Implementation — atomic Redis Lua script for token bucket:

// key: rl:user:42:orders  value: tokens remaining; updated atomically
const script = `
  local tokens = tonumber(redis.call('HGET', KEYS[1], 'tokens')) or ARGV[1]
  local last   = tonumber(redis.call('HGET', KEYS[1], 'last'))   or ARGV[2]
  local now    = tonumber(ARGV[2])
  local refill = (now - last) * tonumber(ARGV[3])                -- tokens per second
  tokens = math.min(tonumber(ARGV[1]), tokens + refill)
  if tokens >= 1 then
    tokens = tokens - 1
    redis.call('HSET', KEYS[1], 'tokens', tokens, 'last', now)
    redis.call('EXPIRE', KEYS[1], 3600)
    return 1
  end
  return 0
`;
const allowed = await redis.eval(script, 1, key, capacity, Date.now()/1000, refillRate);

Critical design choices:

  • Key strategy. Authenticated → user id; unauthenticated → IP; per-endpoint limits if certain endpoints are expensive.
  • Distributed counters in Redis — in-memory counters diverge across instances, leaking the limit N×.
  • Atomic operations. Naive GET/SET races let users burst over. Lua or native Redis commands (INCR) are atomic.
  • Trust proxy config correctly — without it, req.ip is the load balancer's IP, and every user shares one bucket.
  • RFC 9331 headers — RateLimit-Limit, RateLimit-Remaining, RateLimit-Reset; Retry-After on 429.
  • Tighter limits on auth endpoints — 5/15-min on login with skipSuccessfulRequests to slow brute force.

Placement: edge (WAF, CDN) for volumetric protection, application layer for business-context-aware limits (per tenant, per plan). Both layers compose — edge catches obvious attacks, application enforces policy.

Question 60

How do you ensure data integrity in distributed APIs?

Data integrity in distributed APIs means data stays correct and consistent despite partial failures, concurrent writes, and retries across multiple services and databases. The CAP theorem forces trade-offs; the patterns are well-studied.

Threat to integrityDefense
Duplicate operations (retries)Idempotency keys on writes
Partial failure (one of N services succeeded)Sagas + compensations; transactional outbox
Lost writes (concurrent updates clobber each other)Optimistic concurrency with ETag / If-Match; DB row versioning
Event loss (message bus dropped)At-least-once delivery + idempotent consumers
Stale reads from replicasRead-your-writes consistency; read from primary for critical paths
Data corruptionSchema validation; checksums; audit trails

Key patterns:

  • Transactional outbox. Writing to the DB and publishing an event must be atomic, or you get "DB updated but event lost" or "event published but DB rolled back." Solution: write the event to an outbox table in the same DB transaction as the business write. A separate process publishes outbox rows to the message bus and deletes them on ack. This guarantees "DB changed ↔ event published."
  • Idempotent consumers. Messages delivered at-least-once; consumers store processed message ids (or use natural business keys) and no-op on replay.
  • Sagas. For workflows spanning services, model each step with a compensating action. On failure, run compensations in reverse. Orchestrate with Temporal, AWS Step Functions, or Camunda for durability across crashes.
  • Optimistic concurrency. Every row has a version; update WHERE id = ? AND version = ?; failed updates mean someone else got there first — return 409 Conflict. Clients retry with fresh data.
  • Event sourcing. Store every change as an immutable event; current state is derived. Audit is free; time-travel is possible.

Consistency models — pick one explicitly:

  • Strong consistency — reads see the latest writes; required for money, inventory, permissions. Usually a single DB with no eventual replication.
  • Eventual consistency — reads may see stale data briefly; acceptable for feeds, analytics, caches. Most microservice boundaries.
  • Causal consistency — writes seen in causal order; stronger than eventual.

Practical discipline: never assume retries are free (they're not — without idempotency keys, they double-apply); never assume messages are delivered exactly-once (they aren't — consumers must be idempotent); never trust "I got a 200 back" (the response may have been lost; the write may have succeeded anyway). Design for failure at every hop.

Question 62

How do you implement a microservices architecture with RESTful APIs?

Implementing a microservices architecture with REST means decomposing a system into small, independently-deployable services that communicate via HTTP. Done well, teams ship faster; done poorly, you've built a distributed monolith with extra network latency.

The foundational decisions:

  • Service boundaries follow business domains. One bounded context per service (user identity, orders, payments, inventory). Not "technical layers" (don't split into "data service" + "business logic service" + "presentation service" — that's a distributed monolith).
  • Each service owns its database. No shared database between services; otherwise schema changes couple teams. Services exchange data via APIs or events, never by reading each other's DBs.
  • API contracts are explicit. Each service publishes an OpenAPI spec; consumers pin to a version; breaking changes are detected in CI.
  • Events for decoupling. When service A needs to notify others, emit a domain event (OrderCreated) to a bus; interested services subscribe. Avoids direct coupling.

Anti-patterns that undo the benefits:

  • Chatty APIs — one UI page triggers 20 service calls in series. Latency compounds. Fix with a BFF or aggregation at the gateway.
  • Synchronous chains — A calls B calls C calls D. Any one down, all fail. Break with async events.
  • Shared database — two services reading/writing the same tables = instant coupling.
  • Distributed transactions over REST — 2PC across HTTP is painful; use sagas.
  • Too fine-grained — a "user-email" service separate from "user-profile" — extra latency for zero benefit.
  • No contract testing — service B deploys, service A breaks silently in prod.

When microservices make sense: 5+ teams, clear domain boundaries, deployment frequency where coupling is a bottleneck. For small teams, a well-organized monolith (modular, with clear package boundaries) is often strictly better — no network, no eventual consistency, one deploy, one DB transaction. Evolve to services when organizational scale demands it.

Question 63

What is the role of API orchestration, and how do you implement it?

API orchestration is one service coordinating calls to multiple others, driving a workflow through sequential (or parallel) steps and handling failure. It contrasts with choreography, where services react to events with no central coordinator.

PatternWho drivesTrade-offs
OrchestrationCentral coordinator (workflow engine or API)Explicit flow; easy to reason about; coordinator is a dependency
ChoreographyEach service reacts to eventsLoosely coupled; harder to see the full flow; no central retry logic

Where orchestration fits:

  • Multi-step business workflows (order fulfillment: reserve stock → charge card → ship → notify).
  • Backend-for-Frontend aggregation (fan out to 8 services, stitch a response).
  • Long-running jobs with branching logic and human-in-the-loop steps.
  • Processes with compensations (sagas) that need explicit rollback on failure.

Implementation options:

  • Plain code in an orchestration service. Simple but fragile — process crashes lose in-flight workflows.
  • Workflow engine (Temporal, AWS Step Functions, Camunda, Cadence). Durable: workflows survive crashes; activities retry automatically; state persisted. The mature choice for complex flows.
  • API gateway with orchestration (AWS API Gateway + Step Functions integration; Apigee shared flows). Good for edge-layer aggregation.
// Temporal-style orchestration — durable; survives process restarts
async function orderWorkflow(input) {
  const reservation = await activities.reserveStock(input.items,
    { retryPolicy: { maxAttempts: 3 } });

  try {
    const charge = await activities.chargeCard(input.paymentMethod, input.total,
      { timeout: "30s" });

    const shipment = await activities.scheduleShipment(input.address);
    await activities.notifyCustomer(input.userId, shipment);
    return { status: "complete", shipment };

  } catch (err) {
    // compensations in reverse order
    await activities.refundPayment(charge.id);
    await activities.releaseStock(reservation.id);
    throw err;
  }
}

Best practices:

  • Idempotent activities. Workflow engines retry on failure; activities must tolerate duplicate invocation.
  • Explicit timeouts at each step — without them, a stuck dependency freezes the workflow.
  • Compensation for each forward step — "reserve-stock" pairs with "release-stock"; "charge-card" pairs with "refund."
  • Observability at the workflow level — which step is slow? which compensates most often? Workflow engines expose this.
  • Version workflows carefully. Long-running workflows started with version N shouldn't break when you deploy version N+1.

When to choreograph instead: simple reactive flows ("on user signup, send welcome email, enable dashboard, record analytics") — emit the event, let each service consume what it cares about. No central code to maintain. Most real systems combine both — orchestration for critical paths, choreography for decoration.

Question 68

How do you handle nested resources in RESTful APIs?

Nested resources in REST model parent-child relationships as URL hierarchy. The classic example: /users/42/posts/7/comments/3. The pattern has real benefits and real pitfalls; know when each applies.

When nesting works well:

  • Ownership is absolute — a comment can only belong to one post; a line item can only belong to one order. Accessing /orders/42/items/5 implies (and verifies) membership.
  • Clients almost always have the parent id — navigating the UI naturally provides it; URLs are intuitive.
  • Authorization is anchored on the parent — one ownership check at the parent cascades to children.
// nested routes — Express
app.get("/posts/:postId/comments", ...);
app.post("/posts/:postId/comments", ...);
app.get("/posts/:postId/comments/:commentId", ...);
app.patch("/comments/:commentId", ...);           // short form when you have the id
app.delete("/comments/:commentId", ...);

// authorization pattern
async function loadOwnPost(req, res, next) {
  const post = await db.post.findUnique({ where: { id: req.params.postId } });
  if (!post || post.authorId !== req.user.sub) return res.sendStatus(404);
  req.post = post;
  next();
}

Two addressing styles — and the compromise:

StyleURLTrade-off
Nested only/posts/5/comments/3Pretty; inflexible if children move between parents; deep URLs ugly
Flat only/comments/3Concise; loses discoverability; nested list queries awkward
Hybrid (recommended)/posts/5/comments for listing; /comments/3 for individualBest of both; most APIs (GitHub, Stripe) use this

Design rules that avoid pain:

  • Limit nesting depth to 2-3 levels. /users/42/posts/7/comments/3/replies/11 becomes unmanageable. Flatten: /comments/3/replies.
  • Use flat URLs for individual resources. Changing a comment shouldn't require its post id — PATCH /comments/3, not PATCH /posts/5/comments/3.
  • Use nested URLs for list/create operations. GET /posts/5/comments is natural; POST /posts/5/comments implies the parent.
  • Authorization is anchored at the parent — to access a nested resource, you must have access to the parent. Use middleware that loads the parent, verifies ownership, attaches it to the request.
  • Don't nest unrelated types. /users/42/invoices works if users own invoices; /users/42/weather doesn't.

Alternative for complex graphs: once relationships get complex (a comment on a post also mentioning users, with attachments), flat URLs + ?expand=author,post or GraphQL serves better than deeply nested REST paths.

// Node proxy — adds auth, caches, rate-limits calls to an upstream
import { LRUCache } from "lru-cache";

const cache = new LRUCache({ max: 1000, ttl: 60_000 });

app.all("/proxy/weather/*", requireAuth, async (req, res, next) => {
  const upstreamPath = req.path.replace("/proxy/weather", "");
  const cacheKey = `${req.method}:${upstreamPath}:${JSON.stringify(req.query)}`;

  if (req.method === "GET") {
    const hit = cache.get(cacheKey);
    if (hit) return res.set("X-Cache", "HIT").json(hit);
  }

  try {
    const upstream = await fetch(
      `https://api.weather.com${upstreamPath}?${new URLSearchParams(req.query)}`,
      {
        method: req.method,
        headers: { "x-api-key": process.env.WEATHER_KEY },  // server-side secret
        body: req.method !== "GET" ? JSON.stringify(req.body) : undefined,
        signal: AbortSignal.timeout(5000),
      }
    );

    if (!upstream.ok) return res.status(upstream.status).send(await upstream.text());
    const data = await upstream.json();
    if (req.method === "GET") cache.set(cacheKey, data);
    res.set("X-Cache", "MISS").json(data);
  } catch (err) { next(err); }
});

Benefits:

  • Security: secrets stay server-side; you authorize before forwarding.
  • Cost & performance: cache aggressive; deduplicate requests (singleflight); batch.
  • Insulation: when the upstream changes, only the proxy changes — clients untouched.
  • Observability: one place to log all outbound calls; centralized metrics and alerts.
  • Policy: transform or block calls by policy without touching client code.

Risks: the proxy becomes a single point of failure and latency addition. Scale it horizontally; cache aggressively; monitor its health separately. Don't do heavy transformation in the hot path — keep the proxy thin.

Related pattern — Backend-for-Frontend (BFF): a proxy specifically for one client type that aggregates, transforms, and shapes data for that client. Netflix and Soundcloud famously use BFFs for web vs mobile vs TV.

Question 71

What are the challenges of API versioning, and how do you address them?

API versioning challenges compound over time: every version doubles the surface area, every break damages trust, every deprecation requires coordinating with consumers you don't fully control. The best strategy is not to need versions — and when you do, version carefully.

The core challenges:

  • Defining what "breaking" means. Adding a field is safe for tolerant readers but may break strict schema validators. Changing validation from lenient to strict rejects previously-accepted requests.
  • Communicating version transitions. Even with Sunset headers, customers miss them. Direct email and in-dashboard warnings matter more than headers.
  • Running multiple versions in parallel. Double the code paths, double the tests, double the security patches. Teams tire of maintaining v1 while building v3.
  • Client inertia. Customers ship your SDK once, then never touch it. Enterprise integrations may stay on v1 for years after you want to kill it.
  • Silent breakages. Field renames may not crash clients immediately but produce wrong data or UI bugs months later.
  • Internal vs external consumers. You can force internal updates; you can't force external ones. Version policies must reflect this.
  • Version sprawl. v1, v2, v3 all in production. Each has its own bugs, its own docs, its own SDK. Complexity is organizational as much as technical.

Header versioning (Accept: application/vnd.api.v2+json)Prefer clean URLsLess discoverable; easier to forget

Mitigation practices:

  • CI-enforced compatibility checks — oasdiff blocks breaking PRs; GraphQL Inspector for GraphQL.
  • Consumer-driven contracts (Pact) — providers verify against real consumer expectations.
  • Per-client usage tracking — know who's on what version; reach out before cutting off.
  • Long deprecation windows — 6-12 months for public APIs; Stripe gives years.
  • Brownouts before removal — return 410 Gone for 1 hour a day, escalating; forces migration before hard cutoff.

  • Versioned SDKs — clients upgrade SDK in their own schedule; old SDKs pin to old versions.

The most successful public APIs (Stripe, Twilio, GitHub) invest heavily in not breaking — most "version bumps" are silent evolution that tolerant readers never notice. Breaking changes are rare, telegraphed, and accompanied by tooling to migrate.

Question 72

How do you handle cross-origin requests in APIs?

Cross-origin requests occur when a web page at one origin (scheme+host+port) calls an API at a different origin. The browser's Same-Origin Policy (SOP) blocks these by default to prevent malicious sites from reading authenticated responses — CORS is the opt-in mechanism that selectively relaxes SOP.

Two request types the browser distinguishes:

  • Simple requests. GET, HEAD, or POST with form-encoded body + no custom headers. Browser sends directly; checks CORS headers on response.
  • Preflighted requests. Anything else (PUT, DELETE, PATCH, custom headers, Content-Type: application/json). Browser first sends an OPTIONS request to ask permission; only if the preflight succeeds does it send the real request.
// server — express cors middleware, correctly configured
import cors from "cors";

app.use(cors({
  origin: (origin, cb) => {
    if (!origin) return cb(null, true);                  // non-browser (curl, mobile) has no Origin
    const allowed = ["https://app.example.com", "https://admin.example.com"];
    cb(null, allowed.includes(origin));
  },
  credentials: true,                                      // allow cookies across origins
  methods: ["GET","POST","PUT","PATCH","DELETE"],
  allowedHeaders: ["Content-Type","Authorization","X-Request-Id","Idempotency-Key"],
  exposedHeaders: ["X-Request-Id","RateLimit-Remaining"],
  maxAge: 86400,                                          // cache preflight 24h
}));

app.options("*", cors());                                 // ensure OPTIONS handled

The response headers that matter:

HeaderMeaning
Access-Control-Allow-OriginWhich origin is allowed (or * with no credentials)
Access-Control-Allow-CredentialsWhether cookies/Authorization may be sent
Access-Control-Allow-MethodsHTTP methods allowed (preflight response)
Access-Control-Allow-HeadersCustom request headers allowed (preflight response)
Access-Control-Expose-HeadersResponse headers that JS may read
Access-Control-Max-AgeHow long to cache the preflight (seconds)

Common bugs:

  • Auth middleware before CORS middleware — preflight OPTIONS carries no token; auth 401s it; browser aborts. Always register cors() before auth.
  • Allow-Origin: * + Allow-Credentials: true — browsers reject this combination; use a function origin with allowlist.
  • Missing Vary: Origin on caches — CDN may serve the wrong origin's cached response.
  • Custom response headers invisible to JS — must be listed in Access-Control-Expose-Headers.
  • No Max-Age — every unique request doubles its network round-trips (preflight + real).

Production strategy: strict allowlist for production origins; looser (localhost + staging subdomain regex) in dev; never * for authenticated endpoints. Proxy public data through your own origin to avoid CORS entirely for sensitive integrations (and to keep third-party keys server-side).

Question 73

Explain the concept of a RESTful API resource.

A RESTful resource is the central abstraction of REST. Roy Fielding defined a resource as "any information that can be named" — a document, an image, a collection, a service, a temporal concept. In practice, it's whatever noun your API exposes: user, order, comment, session, subscription.

Properties of a well-designed resource:

  • Identified by a URI. /users/42 names a specific user; /users names the collection. URIs are stable — renaming breaks clients.
  • Has a representation. The JSON (or HTML, CSV, etc.) you return is a representation of the resource at a point in time — not the resource itself. Content negotiation can serve different formats.
  • Manipulated via HTTP verbs. GET reads; POST creates (typically in a collection); PUT replaces; PATCH partially updates; DELETE removes.
  • Self-contained. Each request carries enough context (auth, parameters) to be understood in isolation — the stateless constraint.
Resource typeURI patternExample
Collection/postsList all; create new
Item/posts/42Read, update, delete one
Sub-collection (nested)/posts/42/commentsComments on post 42
Controller (verb exception)/orders/42/cancelState transition that doesn't fit CRUD
Filter on collection/posts?author=42Query parameters, not URI path
Pagination/posts?cursor=abcSame resource; different slice

Design rules that keep resources usable:

  • Plural nouns for collections — /users, not /user. Consistency across the API.
  • No verbs in URIs — POST /orders, not POST /createOrder. The method carries the verb.
  • Stable URIs — "Cool URIs don't change" (Berners-Lee). Once published, they're forever. Versioning protects you from feeling forced to change them.
  • Limit nesting depth — 2-3 levels max; flatten deeper relationships into standalone resources with ids.
  • Controllers for non-CRUD state transitions — /orders/42/cancel, /users/42/deactivate. Accepted REST pragmatism.
  • Clear correspondence to domain concepts — a resource should map to something users/developers think about. If you find yourself inventing resources to fit REST (/userEmailPreferenceUpdateRequests), step back.

Beyond CRUD — when resources aren't enough: sometimes the operation is the thing (search, batch operations, computation). Treat these as actions on controller resources (POST /search) or step outside pure REST for these endpoints. Don't contort your API; resources are a tool, not a religion.

Question 74

How do you implement data encryption in APIs?

Data encryption in APIs operates on two axes: in transit (between endpoints) and at rest (in storage). Both are table stakes in 2026; the interesting decisions are about scope, key management, and end-to-end flows.

LayerConcernSolution
In transitNetwork eavesdroppingTLS 1.3; HSTS; certificate pinning for mobile
At restDB dump / backup theftFull-disk encryption (AWS RDS, Cloud SQL); column-level for sensitive data
Field-level in DBInsider threat; subset of sensitive fieldsApp-level encryption with KMS-wrapped keys
End-to-endServer should not read contentClient-side encryption; server stores ciphertext only (Signal pattern)
TokenizationRegulatory scope (PCI-DSS)Replace card numbers with tokens; detokenize only where needed

Pattern: envelope encryption for sensitive fields.

// each sensitive row has a DEK (data encryption key), wrapped by a KMS KEK
// application-side encrypt on write, decrypt on read
import { KMSClient, EncryptCommand, DecryptCommand } from "@aws-sdk/client-kms";

async function encryptField(plaintext) {
  const dek = crypto.randomBytes(32);                       // new key per value
  const cipher = crypto.createCipheriv("aes-256-gcm", dek, iv);
  const ciphertext = Buffer.concat([cipher.update(plaintext), cipher.final()]);
  const tag = cipher.getAuthTag();

  // wrap DEK with KMS master key
  const { CiphertextBlob } = await kms.send(new EncryptCommand({
    KeyId: KMS_KEY_ID, Plaintext: dek,
  }));

  return {
    ciphertext: ciphertext.toString("base64"),
    iv:         iv.toString("base64"),
    tag:        tag.toString("base64"),
    wrappedDek: Buffer.from(CiphertextBlob).toString("base64"),
  };
}

Key management principles:

  • Never hardcode keys. KMS (AWS, Google Cloud, Azure), HashiCorp Vault, or managed HSM.
  • Rotation. Regular rotation of KEKs (master keys); re-wrap DEKs on rotation, no re-encryption of data needed.
  • Separation of concerns. The service that encrypts shouldn't also manage key rotation; use a dedicated key broker.
  • Audit trail. Log every key access; alert on anomalies.
  • Least privilege. Per-environment keys; prod keys never accessible from dev.

Compliance overlays:

  • PCI-DSS (card data): tokenize at ingestion; encrypt tokens in transit and at rest; strict audit logs.
  • HIPAA (PHI): field-level encryption; signed BAAs with cloud providers; access logs per record.
  • GDPR (EU PII): pseudonymization where possible; encrypted backups; right-to-erasure compatible storage.

Performance cost: TLS is nearly free with modern hardware (AES-NI). Field-level encryption adds ~100μs per encrypt/decrypt — negligible except for hot columns queried millions of times. For those, tokenization or searchable encryption (format-preserving, blind indexes) keeps queries possible without decrypting in bulk.

Question 75

What are the best practices for designing API endpoints?

Designing API endpoints is where consistency, discoverability, and evolvability are won or lost. The accumulated wisdom of the REST community (Google AIPs, Microsoft REST Guidelines, Stripe, Zalando) converges on a set of practices.

URL design:

  • Plural nouns for collections: /users, not /user. Consistent across the API.
  • No verbs in URIs: method conveys action. POST /orders, not POST /createOrder.
  • lowercase, hyphen-separated: /shipping-addresses, not /ShippingAddresses or /shipping_addresses.
  • Stable IDs in URLs: UUIDs or opaque IDs; avoid sequential integers that leak counts.
  • Nesting for ownership, flat for direct access. GET /posts/42/comments for the list; PATCH /comments/5 for individual.
  • Filters as query parameters: /users?role=admin, not /users/admin.
  • Cap nesting at 2-3 levels.

HTTP methods used correctly:

MethodSemanticsIdempotent?
GETRetrieve; no side effects; cacheableYes
POSTCreate new; or catch-all for non-idempotent opsNo
PUTReplace resource at URI (full replacement)Yes
PATCHPartial updateUsually
DELETERemoveYes (deleted stays deleted)

Status codes that mean something: 200 OK, 201 Created with Location header, 204 No Content (delete success), 400 (client error, validation), 401 (unauthenticated), 403 (unauthorized), 404 (not found or hidden), 409 (conflict / optimistic concurrency), 422 (semantic validation), 429 (rate limited), 5xx (server error, retry-worthy).

Payload conventions:

  • Consistent envelope or no envelope — decide once. Bare { "id": 42, ... } for a single item; { "data": [...], "nextCursor": "..." } for collections.
  • camelCase or snake_case — consistent across every endpoint.
  • ISO 8601 dates (2026-04-19T12:00:00Z); always UTC.
  • Enums as strings, not magic numbers.
  • Strings for large integers (IDs > 2^53) — JavaScript precision.
  • Cursor pagination for any list that could grow.
  • Standardized error shape (RFC 7807 Problem Details) everywhere.

Cross-cutting:

  • Idempotency keys on all POSTs that create side effects.
  • ETag + If-Match for optimistic concurrency on updates.
  • Cache-Control explicit on every response.
  • Request id in responses for debugging.
  • Rate limit headers on every response.

Culture: document design decisions; enforce with a style guide + Spectral (OpenAPI linter) in CI. The best APIs feel uniform — a developer who's learned one endpoint predicts the next correctly. Consistency beats cleverness.

Question 77

Explain the role of API governance.

API governance is the discipline of keeping many APIs — across teams, services, and versions — consistent, discoverable, secure, and evolvable. Without governance, a 200-microservice company ends up with 200 inconsistent APIs: different auth schemes, different error formats, different naming, no central catalog, duplicate functionality. Governance prevents this drift.

Core governance functions:

  • Standards & style guides: URL conventions, error format (RFC 7807), pagination, versioning, auth. Codified in documents like Microsoft's REST API Guidelines or Google AIPs.
  • Spec-first design: OpenAPI / GraphQL SDL / Protobuf reviewed before code. Changes gated by peer review.
  • Automated linting: Spectral rulesets catch violations in CI. Example: "all 4xx responses must include a problem+json body," "all endpoints require auth unless tagged public."
  • Breaking change detection: oasdiff on every PR; required approvers for breaking changes.
  • API catalog: central portal (Backstage, Apigee developer portal, Stoplight) lists every API, its owner, SLOs, and how to get access.
  • Lifecycle management: clear stages (experimental / beta / stable / deprecated / retired) with rules for each.
  • Security & compliance: PII handling review, encryption requirements, audit logging, authz model.

Federated vs centralized: small orgs can run governance from a central API platform team. Large orgs federate — each team owns its APIs but must pass automated checks and follow shared standards. The platform team owns the rules, not the APIs. Spotify's golden path, Netflix's API platform, and Zalando's restful-api-guidelines are public exemplars.

Why it matters commercially: inconsistent APIs are slow to integrate. Developers copy-paste auth code because every service is slightly different. Security reviews are repeated per service. Governance turns that friction into a reusable set of patterns and automations — 10 minutes to integrate a new internal service, not 2 days.

Question 78

How do you implement secure API deployments?

Secure API deployment extends beyond code to the entire pipeline and runtime. A perfectly-coded API can still be breached by a leaked deployment key, an unpatched base image, or a misconfigured load balancer.

Build pipeline:

  • Signed commits & protected branches: require GPG-signed commits on main; prevent force-pushes. Defends against compromised developer accounts.
  • SAST & dependency scanning: Semgrep, CodeQL, Snyk, or GitHub Advanced Security run on every PR. Block merge on high-severity findings.
  • Container image scanning: Trivy or Grype on every build; fail on CVEs above threshold. Use distroless or minimal base images.
  • SBOM generation: produce a Software Bill of Materials (CycloneDX, SPDX) per build for supply-chain auditing.
  • Signed artifacts: Sigstore / cosign sign container images; runtime verifies signatures before pulling.

Runtime:

  • Secrets management: secrets from Vault / AWS Secrets Manager / GCP Secret Manager injected at runtime, never in env files or code. Rotate on schedule.
  • mTLS between services: via service mesh (Istio, Linkerd) or infrastructure. No plaintext traffic inside the cluster.
  • Least-privilege IAM: each service has a unique identity with only the permissions it needs. Audit regularly.
  • Network segmentation: NetworkPolicies in Kubernetes restrict which pods can talk to which; bastion/proxy required for DB access.
  • WAF + DDoS protection: Cloudflare, AWS WAF, or similar in front of public endpoints. Bot management for credential stuffing.
  • Rate limiting at edge: so a bad actor can't exhaust your origin resources.

Deployment strategies: blue/green or canary deploys detect broken auth or validation before full rollout. Observability (structured logs, traces, metrics) catches anomalies in production; alerting on 4xx spikes surfaces attempted exploits. Incident response runbook predefines how to revoke keys, rotate secrets, and roll back if a breach is suspected. Together these turn a one-time "secure code" exercise into a continuously secure deployed system.

Question 79

What are the benefits of using a schema registry in APIs?

A schema registry is a service that stores, versions, and distributes message schemas — typically for event streaming (Kafka, Pulsar) but increasingly for any async or contract-heavy communication. Producers register schemas before publishing; consumers fetch schemas to deserialize messages. The registry enforces compatibility rules on every new version.

Benefits:

  • Enforced compatibility: the registry rejects a schema that breaks consumers. Compatibility modes include BACKWARD (new consumers read old data), FORWARD (old consumers read new data), and FULL (both). This replaces fragile social contracts with a runtime guarantee.
  • Efficient wire format: Avro or Protobuf encode only a schema ID plus data bytes. Consumers fetch the schema once, cache it, then decode all messages cheaply. Typical payloads shrink 3-10x vs JSON.
  • Central discovery: every message type is documented in one place with its owners, producers, and consumers. Replaces Slack-thread "what's the shape of this event?"
  • Evolution audit trail: the registry tracks every schema version and who made the change. Rollback is a metadata operation, not a code deploy.
  • Type-safe codegen: Avro/Protobuf schemas generate strongly-typed classes in any language. Consumers can't accidentally miss required fields.

Evolution rules worth memorizing:

ChangeBackward-compatForward-compat
Add optional field with defaultYesYes
Add required fieldNoNo
Remove fieldNo (old consumers may rely on it)Yes
Rename fieldNoNo
Change type (int → long)Usually noUsually no

Common implementations (2026): Confluent Schema Registry (Avro/Protobuf/JSON Schema), AWS Glue Schema Registry, Apicurio (open-source), Buf Schema Registry for Protobuf. Where it applies beyond Kafka: async APIs via AsyncAPI, gRPC via .proto files stored in a Buf registry, GraphQL via schema change tracking in Apollo Studio or GraphQL Hive. The underlying pattern — central contract with compatibility enforcement — is broadly useful.

REST's statelessness requires each request to carry enough information to be processed independently — the server keeps no per-client session memory between requests. Yet real applications have inherently stateful flows: shopping carts, multi-step wizards, conversations. The tension is resolved by externalizing state from the server's memory to a location either the client or an external store owns.

Resource-centric is the REST-native answer. Question 81

Explain the concept of content negotiation in APIs.

Content negotiation is HTTP's mechanism for a client and server to agree on the representation of a resource. The same logical resource (e.g., a user) can be delivered as JSON, XML, CSV, or HTML depending on what the client asks for — via headers, not URLs.

Negotiation headers:

HeaderPurposeExample
AcceptMedia type preferenceapplication/json
Accept-LanguageLanguage preferenceen-US, fr;q=0.8
Accept-EncodingCompressiongzip, br
Accept-CharsetCharacter set (mostly obsolete)utf-8

The q-weight system lets clients express preferences: Accept: application/json;q=1.0, application/xml;q=0.5 means "prefer JSON, accept XML if JSON is unavailable." The server picks the best match it can produce.

// Express content negotiation
app.get("/users/:id", (req, res) => {
  const user = loadUser(req.params.id);
  res.format({
    "application/json": () => res.json(user),
    "application/xml":  () => res.type("xml").send(toXml(user)),
    "text/csv":         () => res.type("csv").send(toCsv([user])),
    "text/html":        () => res.render("user", { user }),
    "default":          () => res.status(406).send("Not Acceptable"),
  });
});

Vary header is critical. When a cache stores a JSON response and a later client asks for XML, the cache must not serve JSON. Vary: Accept tells caches to key by the Accept header. Missing Vary is a common bug that causes cross-client contamination through CDNs.

Use cases beyond format: media-type versioning (Accept: application/vnd.api.v2+json) embeds version in the header rather than URL; localization (Accept-Language) picks translations; compression (Accept-Encoding) enables gzip/brotli. Most modern APIs stick to JSON and skip format negotiation, but language and encoding remain universally useful. For public APIs with diverse clients (enterprise SaaS, integrations), offering CSV and XML alongside JSON is a small cost that adds significant integration value.

Question 82

How do you implement API access control?

API access control is the decision layer that answers "is this caller allowed to perform this action on this resource?" after authentication has answered "who is this caller?" Getting the distinction wrong produces the most common vulnerability class: Insecure Direct Object Reference (IDOR).

Access control models:

ModelPolicy expressionBest for
RBACRole has permissions; user assigned rolesInternal tools, org hierarchies
ABACAttributes of user + resource + contextCompliance-heavy (healthcare, finance)
ReBACRelationships (owner, member, viewer)Collaboration apps (Google Docs, GitHub)
PBACPolicy-as-code (OPA, Cedar)Multi-app consistent policy
Scope-basedToken carries scopes (OAuth)Public APIs, third-party integrations

Defense-in-depth layers: an API gateway checks authentication and coarse scopes (e.g., reports:read); the service checks business-rule authorization (e.g., "user belongs to this tenant"); the data layer enforces row-level security (e.g., PostgreSQL RLS) so a bug in service code can't leak cross-tenant data. Each layer fails closed independently.

// explicit ownership check — the line that prevents IDOR
const report = await db.report.findUnique({ where: { id: params.id } });
if (!report) return res.sendStatus(404);
if (report.tenantId !== req.user.tenantId) return res.sendStatus(404);  // not 403 — hide existence
if (req.user.role !== "viewer" && req.user.role !== "admin") return res.sendStatus(403);

Policy engines centralize authorization logic. OPA (Open Policy Agent) with Rego evaluates allow(user, action, resource, context); used widely with Kubernetes admission and microservice sidecars. OpenFGA / Zanzibar model permissions as relationships in a graph, used by Auth0 FGA, SpiceDB, and emulated from Google's internal Zanzibar paper. Cedar (AWS) is a purpose-built authz policy language. Centralization means one audit surface and consistent policy across many services — but adds operational complexity; small apps are fine with well-structured RBAC in the DB.

Question 83

What are the common techniques for API payload compression?

Payload compression trades CPU time for network time, which is almost always a good trade: networks are slower than memory by 10-100x. A 1 MB JSON response compressed to 80 KB with gzip saves ~100ms on a typical mobile connection — far more than the 2ms CPU cost to compress it.

Common algorithms:

AlgorithmRatioCPUWhen
gzip~4xLowUniversal fallback; widely supported
brotli (br)~5xMedium (static), low (decode)Preferred for modern browsers (all current)
zstd~4-5xVery lowHigh-throughput server-to-server
deflate~4xLowLegacy; rarely needed
LZ4 / Snappy~2-3xExtremely lowReal-time streaming, log pipelines

HTTP compression negotiation: client sends Accept-Encoding: br, gzip; server picks the best supported algorithm and responds with Content-Encoding: br. Most frameworks handle this automatically — Express compression() middleware, Nginx gzip on, CDN compression at the edge. Edge compression (Cloudflare, Fastly) is preferable for public APIs because the CDN compresses once and serves to many.

When NOT to compress: already-compressed formats (JPEG, PNG, MP4, ZIP) — you waste CPU and sometimes increase size. Very small payloads (<1 KB) — the compression header overhead exceeds the savings; set a threshold in the compression middleware. Encrypted payloads over compression create the CRIME / BREACH attack surface — mitigate by disabling compression on sensitive endpoints or isolating secret values in unpredictable response locations.

Beyond HTTP compression: binary protocols like Protobuf and MessagePack are already 2-5x smaller than JSON before compression — they're conceptually "always-on compression." gRPC uses Protobuf over HTTP/2 with optional gzip/zstd for another 2-3x gain. For services where bytes are expensive (mobile, IoT, cross-region) the compound effect of binary encoding + compression can reduce bandwidth 15-20x vs plain JSON.

Question 84

How do you design an API for multi-region deployment?

Multi-region deployment places API instances close to users across the globe. Latency from London to us-east-1 is ~80ms; from London to eu-west-1 is ~5ms. For latency-sensitive APIs, multi-region is the difference between "snappy" and "slow." It also provides disaster recovery if an entire region fails.

Routing strategies:

StrategyHow it worksCaveats
Geo-DNS (Route 53, Cloudflare)DNS returns closest region per clientDNS caching creates stickiness; TTL tuning matters
AnycastSame IP advertised from all regions; BGP picks closestRequires CDN/edge infrastructure
Client-side selectionClient measures regions, picks fastestExtra round-trips at startup

The hard part is data, not code. Stateless APIs copy trivially across regions; the database is where multi-region gets interesting:

  • Single-writer + read replicas: writes go to one region; reads served locally. Simple, but write latency for distant users is painful. Best for read-heavy workloads.
  • Regional sharding: each region owns its own users; data never crosses. Low latency but cross-region queries are hard. Works for B2C where users don't interact across regions.
  • Multi-master replication: all regions accept writes; conflicts resolved by vector clocks, CRDTs, or last-write-wins. Database choice: DynamoDB Global Tables, CockroachDB, Spanner, YugabyteDB, Cassandra.
  • Global distributed DB: Spanner / CockroachDB / Yugabyte present a single consistent view across regions using Paxos/Raft. Writes cost ~100ms (consensus across regions) but correctness is strong.

Consistency trade-off: CAP theorem forces a choice when the network partitions. Most multi-region systems pick AP (availability + partition tolerance) with eventual consistency — good enough for feeds and profiles. Financial and inventory systems pick CP, accepting that a partitioned region returns 5xx rather than stale balances.

Operational concerns: deploy rollouts per region (not globally) so a bad deploy affects one region. Health checks drive DNS failover within seconds. Observability aggregates metrics with region dimension. Test chaos scenarios quarterly — "kill region X" should produce an incident-free failover.

Question 85

Explain the importance of API consistency and how to achieve it.

API consistency is the property of the API "feeling the same" across endpoints, versions, and services. Inconsistent APIs force developers to re-learn each endpoint, increase integration time, and make client libraries hard to generate. Consistency is invisible when it works and infuriating when it doesn't.

Dimensions of consistency:

  • Naming: userId everywhere, never sometimes user_id and sometimes uid. camelCase or snake_case — pick one and enforce it.
  • URL structure: plural collections (/users); nesting max 2-3 levels; lowercase with hyphens.
  • HTTP methods: semantic use — POST creates, PATCH partially updates, DELETE removes.
  • Status codes: same meaning everywhere — 201 with Location on create; 204 on delete; 404 for missing or hidden; 409 for conflicts.
  • Error format: one shape (RFC 7807 application/problem+json) across every endpoint and service.
  • Pagination: same parameters (?limit=&cursor=); same response envelope ({ data, pagination }).
  • Date/time: always RFC 3339 UTC strings; never Unix epochs mixed with ISO dates.
  • Versioning: one strategy (URL path, header) across services.

How to achieve it:

TechniqueMechanism
Style guideWritten standard (Zalando, Stripe public guides are models)
Shared librariesOne error-formatting lib, one pagination helper, one auth middleware
Spec lintingSpectral / Redocly rules in CI enforce style on every OpenAPI spec
Code generatorsScaffold new services from a template that bakes in consistency
API gateway normalizationRewrite headers / error shapes at the edge to mask legacy inconsistencies
Architecture reviewPeer review of OpenAPI spec before implementation

Legacy inconsistency: retrofitting consistency onto existing APIs is painful but possible. Strategy: version bump (v2 is consistent; v1 lives on for compatibility) or gateway transformation (gateway rewrites legacy responses to the new shape, bodies in parallel). Document both shapes during the transition; track consumer migration; sunset v1 eventually. Rushing causes more breakage than the inconsistency itself.

Question 86

How do you handle API migrations and data transformations?

API migrations happen whenever the data model evolves: a field split, a type change, a resource renamed, a new required attribute. Done well, consumers don't notice. Done badly, every consumer integration breaks at once. The key is running old and new in parallel long enough for consumers to migrate.

Migration patterns by severity:

PatternWhenConsumer impact
Additive-onlyNew optional fields, new endpointsZero — ignore if unused
Expand-and-contractRename, type changeZero during parallel phase
New version alongsideBreaking shape changesConsumers migrate when ready
Transformation layerOld-shape consumers behind a gatewayZero, gateway translates

Expand-and-contract is the workhorse for renames and type changes. Phases:

  1. Expand: add the new field; populate both old and new simultaneously. Both fields return on reads.
  2. Migrate consumers: announce deprecation; monitor usage of old field.
  3. Contract: once old-field usage is zero, remove it. Continue writing to new field only.

Data transformations for large backfills:

  • Dual-write: application writes both shapes during migration window.
  • Async backfill: worker job iterates existing records, transforms, writes new shape. Idempotent so it can restart.
  • Outbox pattern: domain changes emit events; downstream projections (search index, cache) rebuild asynchronously.
  • Shadow reads: new code path runs in parallel, compares results with old, logs discrepancies — validation before switching.
// shadow read — validate new query matches old before cutover
const oldResult = await legacyQuery(id);
if (process.env.SHADOW_READ === "true") {
  queueMicrotask(async () => {
    const newResult = await newQuery(id);
    if (!deepEqual(oldResult, newResult)) log.warn("shadow mismatch", { id });
  });
}
return oldResult;

Common mistakes: doing the migration in a single big-bang release (no rollback path); forgetting to verify consumer adoption before removing old fields (breaking integrations); missing the async consumers (webhooks, ETL jobs) that aren't in the request log and are invisible to metrics. Always grep your own logs for old-field references before contracting — clients aren't always listed.

Question 87

What are the challenges of API scalability, and how do you address them?

API scalability challenges surface as load grows and touch every layer: compute, data, network, and human processes. Each layer has characteristic limits and standard remedies.

Compute scaling: horizontal scaling (add instances) is straightforward for stateless APIs behind a load balancer. Challenges: session affinity if you haven't externalized state; startup time for autoscaling responsiveness; cold starts for serverless. Kubernetes HPA or cloud autoscalers drive instance count from CPU or custom metrics (request rate, queue depth).

Data scaling is harder. The database is usually the bottleneck that breaks first:

ProblemRemedy
Read load on primaryRead replicas; route reads to replicas
Write loadSharding (by tenant, user, hash); CQRS
Hot keys / hot partitionsHash to spread load; cache at edge
Expensive analytics on OLTPReplicate to data warehouse (Snowflake, BigQuery); OLAP queries there
Slow queriesIndexes, materialized views, denormalization
Connection limitsPgBouncer, RDS Proxy for transaction pooling

Network scaling: CDN caching absorbs most GET traffic before it hits the origin; HTTP/2 or HTTP/3 reduces per-connection overhead; keep-alive and connection pooling to upstream services prevents TCP handshake thrashing.

Caching at every layer multiplies effective capacity. Browser cache → CDN → API gateway cache → application in-memory LRU → Redis → database query cache. Each layer with 90% hit rate divides origin load by 10. Cache invalidation is the hard part — tag-based invalidation (surrogate keys), TTLs, or event-driven invalidation via outbox.

Async work offloading: anything slow or fan-out should leave the request path via queues (SQS, RabbitMQ, Kafka). Email send, image resizing, webhook delivery, cross-service updates all belong in background workers. The API responds 202 with a resource URL the client can poll.

Human scalability matters as traffic grows: on-call rotations, runbooks, SLOs, error budgets. A system that scales technically but has no one to tune it regresses. Organizational scaling — dedicated platform team, per-service ownership, standardized tooling — is the unglamorous part that determines whether technical scalability holds under stress.

Question 88

How do you implement a custom authentication scheme in APIs?

Custom authentication schemes appear when standard mechanisms (OAuth, JWT Bearer, mTLS) don't fit the constraints — often for hardware-constrained IoT, legacy integrations, or ultra-low-latency trading systems. They're rare in 2026 because standards cover most needs; when you do build one, the cryptographic choices matter far more than the API surface.

When a custom scheme is justified:

  • Shared-secret HMAC signing: each request carries an HMAC-SHA256 of the canonicalized request, signed with a shared secret. AWS Signature V4 is the canonical example — excellent for machine-to-machine where neither side wants TLS overhead for every request.
  • Asymmetric signed requests: client signs with private key; server verifies with public key. Preferred for public APIs: no secret on the client, revocable per-key.
  • Challenge-response: server sends a nonce; client returns a signature over it. Defeats replay attacks without timestamps.
  • Device-attested tokens: mobile SDKs (Apple DeviceCheck, Android Play Integrity) sign tokens the server verifies against the platform's public key.
// HMAC-signed request verification
function verifyHmac(req, secret) {
  const ts = req.headers["x-timestamp"];
  const sig = req.headers["x-signature"];
  if (!ts || !sig) return false;

  // reject requests > 5 min old — replay protection
  if (Math.abs(Date.now() - Number(ts)) > 5 * 60_000) return false;

  // canonicalize — method, path, timestamp, body hash
  const bodyHash = crypto.createHash("sha256").update(req.rawBody || "").digest("hex");
  const canonical = `${req.method}\n${req.path}\n${ts}\n${bodyHash}`;
  const expected = crypto.createHmac("sha256", secret).update(canonical).digest("hex");

  return crypto.timingSafeEqual(Buffer.from(sig, "hex"), Buffer.from(expected, "hex"));
}

Essentials for every custom scheme:

  • Timestamp + window: prevents indefinite replay of captured requests.
  • Nonce or request-id: prevents replay within the window.
  • Canonicalization: method + path + body hash + critical headers included in the signature so tampering is detected.
  • Constant-time comparison (crypto.timingSafeEqual) — prevents timing attacks.
  • Key rotation — protocol allows multiple valid keys during rotation windows.
  • Rate-limit unauthenticated attempts to block brute force.

The warning: custom crypto is the #1 source of catastrophic API bugs. Before building, exhaust standards: OAuth 2.1 client credentials for M2M, mTLS for mutual auth, JWT with asymmetric keys for token-based, PAT with prefix+hash for simple API keys. Custom schemes justify themselves only when operational constraints make standards impossible — and even then, copy a well-analyzed design like AWS SigV4 rather than inventing.

Question 90

How do you handle asynchronous processing in APIs?

Asynchronous processing lets an API accept work quickly and return immediately, performing slow or heavy operations in the background. Without async, a 30-second email-blast endpoint ties up a worker thread, blocks other requests, and hits the proxy's 60-second timeout. With async, the endpoint returns 202 Accepted in <50ms and a worker handles the real work.

Async patterns for HTTP APIs:

PollingClient polls GET /jobs/:id until status = doneBatch jobs, reports; simple, firewall-friendly
Webhook callbackClient provides URL; server POSTs when completeLong-running workflows; no client polling
Server-Sent EventsClient opens stream; server pushes progressProgress bars, single-direction feeds
WebSocketBidirectional long-lived connectionChat, collaborative editing
Message queue to consumer serviceCaller fire-and-forgets; result seen elsewhereInternal workflows, event-driven

Standard HTTP pattern — 202 Accepted + polling:

// Accept and queue; return a Location to poll

→ 202 Accepted

  Content-Type: application/json
  { "jobId": "abc123", "status": "queued" }

→ 200 OK

Queue architecture: the endpoint pushes a message to a queue (SQS, RabbitMQ, Kafka, Redis Streams, BullMQ); background workers consume and process. Keep the message small — an ID, not the payload; workers load full data from the DB. Idempotency matters — queues retry on failure, so workers must handle the same message twice without duplicating effects. Store processedAt per job and skip already-processed messages.

Error handling: retries with exponential backoff for transient failures; dead-letter queue for messages that fail repeatedly — alerts on DLQ growth; timeout + heartbeat so a worker crash releases the message for another consumer. Observability: trace the request ID through the queue into the worker so a slow report's entire lifecycle is visible in one distributed trace.

Modern frameworks (Temporal, Trigger.dev, Inngest, Azure Durable Functions) encode workflows as code with durable execution — long-running orchestrations survive server restarts and can wait hours or days between steps. Simpler BullMQ/SQS + workers is fine for most cases; reach for workflow engines when orchestration logic is itself complex.

Question 91

What are the common patterns for API error handling?

API error handling is where users either feel guided or lost. Well-designed errors tell the caller what went wrong, why, and what to do next. Poorly designed errors return 500 with a stack trace or, worse, 200 OK with {"error": "failed"}.

Error categories map to HTTP classes:

ClassExampleAction
4xx client errors400, 401, 403, 404, 409, 422, 429Client must fix request
5xx server errors500, 502, 503, 504Client may retry; server logs
Domain errorsInsufficient funds, stock out4xx with domain error code

RFC 7807 Problem Details is the standard shape for error responses:

HTTP/1.1 422 Unprocessable Entity
Content-Type: application/problem+json

{
  "type":     "https://api.example.com/problems/validation-error",
  "title":    "Request validation failed",
  "status":   422,
  "detail":   "The email field is required",
  "instance": "/users",
  "errors": [
    { "field": "email",    "code": "required" },
    { "field": "age",      "code": "out_of_range", "message": "must be 13-120" }
  ],
  "traceId":  "4e9f2b..."
}

Pattern principles:

  • Machine-readable code AND human-readable message. Clients switch on stable codes (insufficient_funds); humans read messages.
  • Field-level errors for validation; let forms highlight specific inputs.
  • Include traceId / requestId so users can report "this failed — trace 4e9f2b" and you can find it in logs.
  • Don't leak internals: never return stack traces, SQL errors, or filesystem paths in 5xx responses.
  • Consistent shape across all endpoints, owned by a central error-formatting middleware.

Retry signals: Retry-After header on 429 and 503 tells well-behaved clients when to come back. RateLimit-Remaining lets clients pace themselves. Idempotency-key response echoes confirm which request was deduplicated.

Anti-patterns to avoid: 200 OK with errors in the body (breaks HTTP caching, monitoring, and every client library); generic {"error": true} (useless for debugging); raw exception messages (security risk); inconsistent shapes per endpoint (integration nightmare). A central error middleware that wraps all thrown exceptions into RFC 7807 is the smallest change that makes the biggest quality difference. Pair with unhandled-rejection and uncaught-exception handlers that log + exit rather than return cryptic 502s.

Question 92

How do you implement client-side rate limiting?

Client-side rate limiting is proactive self-pacing by the API consumer — a client that sends at most N requests per second regardless of what the server allows. It protects the client from burning through quotas, avoids triggering server-side limits, and smooths burst traffic. Contrast with server-side limiting, which rejects excess; client-side limiting prevents excess from happening at all.

Common algorithms (same as server-side, implemented in the client SDK):

  • Token bucket: refill tokens at rate R; each request consumes one; burst allowed up to bucket capacity. Most flexible — allows short bursts within the average limit.
  • Leaky bucket (queue): requests queue; dispatcher releases at rate R. Smooth output; bounded queue drops new requests when full.
  • Semaphore: cap on in-flight concurrent requests (not per-second rate). Useful when the constraint is upstream concurrency, not rate.
// token-bucket client limiter
class RateLimiter {
  constructor({ capacity, refillPerSec }) {
    this.capacity   = capacity;
    this.tokens     = capacity;
    this.refillRate = refillPerSec;
    this.lastRefill = Date.now();
    this.queue      = [];
  }

  async acquire() {
    this._refill();
    if (this.tokens >= 1) {
      this.tokens -= 1;
      return;
    }
    // wait until next token is available
    const waitMs = (1 / this.refillRate) * 1000;
    await new Promise((r) => setTimeout(r, waitMs));
    return this.acquire();
  }

  _refill() {
    const now   = Date.now();
    const add   = ((now - this.lastRefill) / 1000) * this.refillRate;
    this.tokens = Math.min(this.capacity, this.tokens + add);
    this.lastRefill = now;
  }
}

const limiter = new RateLimiter({ capacity: 10, refillPerSec: 5 });
async function call(url) {
  await limiter.acquire();
  return fetch(url);
}

Respect server hints. Well-behaved clients listen to RateLimit-Remaining, RateLimit-Reset, and Retry-After headers to adjust their own pacing. When the server says "you have 3 remaining in this window," slow the outgoing rate to match. On 429, respect Retry-After — don't hammer the limiter trying to beat it.

Adaptive limiting: advanced clients track server latency and error rate, slowing down when latency climbs or 5xx appears — the AIMD (Additive Increase, Multiplicative Decrease) pattern from TCP congestion control. Libraries like Netflix's Concurrency Limits and Resilience4j implement this. The result: client automatically pulls back during server stress, ramps up as the server recovers — no manual tuning needed for changing capacity.

Question 93

Explain the concept of an API sandbox and its use cases.

An API sandbox is a non-production environment that replicates the API's shape and behavior for development, testing, and onboarding. Calls don't affect real data, real money, or real external systems. Stripe, Twilio, Plaid, and every major SaaS provides one — and it's often the first developer experience, so quality matters.

Sandbox use cases:

Use caseWhat's simulated
New developer onboardingExplore API without risking production
Client SDK testingVerify integration end-to-end without real calls
CI/CD pipelinesIntegration tests against a stable fake API

Implementation approaches:

  • Shared multi-tenant sandbox: one environment serving all developers; pre-seeded test data (credit cards, accounts, webhooks). Cheapest to run; data may leak between developers.
  • Per-developer isolated sandbox: each API key gets its own database partition — changes visible only to that caller. Stripe's test mode is this shape. More complex to build; better security.
  • Ephemeral sandbox per CI run: spin up a fresh instance per build; tear down after. Testcontainers + Docker — zero state bleeding between runs.

  • Deterministic test identifiers: Stripe's tok_visa_chargeDeclined, 4000 0000 0000 0002 (declined) — specific inputs trigger specific behaviors. This is how sandbox becomes a testing superpower.
  • Webhook simulation: sandbox can fire webhooks to the developer's URL on a schedule or via a "send test event" button.
  • Rate limits and timing: mirror production limits so developers can test backoff logic; inject latency optionally.
  • Separate credentials clearly: sk_test_* vs sk_live_* prefixes make it obvious which environment you're in; runtime checks prevent test keys calling live endpoints.
  • No real side effects: emails, SMS, and webhooks all targeted to sandbox-only destinations or dev-provided URLs.
  • Reset capability: developers can wipe their sandbox state and start over.

The sandbox is part of DX. A slow, flaky, or inaccurate sandbox teaches developers to distrust the API before they've integrated. Treat it like a first-class product: monitoring, SLAs, changelog, examples. Public API providers who invest in sandbox quality (Stripe, Shopify, Twilio) see faster integration and lower support burden.

Question 94

How do you design APIs for mobile applications?

APIs for mobile applications optimize for constraints mobile platforms impose: intermittent connectivity, tight battery budgets, expensive bandwidth on cellular, high latency on 4G/5G, and heterogeneous devices from $50 Android phones to iPhone Pros. These constraints reshape API design.

Key design choices:

ConstraintImplication for API

Limited bandwidthCompression; small payloads; deltas

Backend for Frontend (BFF) pattern pays off here. Instead of the mobile app calling 5 microservices and assembling the result, a single GET /mobile/home endpoint aggregates everything the home screen needs into one response. Mobile saves 4 round-trips and many milliseconds. The BFF is mobile-specific — web gets its own.

GraphQL's mobile advantage is field selection: the app asks for exactly the fields it displays, no more. A feed on a slow connection returns 8 KB instead of 80 KB. Persisted queries (pre-registered query IDs) reduce the request body further and add security.

Offline-first sync:

  • Client cache + queue: mutations while offline queue locally; replay on reconnect with server-side deduplication via idempotency keys.
  • Delta sync: GET /feed?since=cursor returns only changes since the cursor. Full replace is expensive.
  • Conflict resolution: last-write-wins, vector clocks, CRDTs, or server-authoritative.
  • WatermelonDB, Replicache, PowerSync, Couchbase Lite are 2026 libraries that handle much of this.

Push vs poll: APNs (iOS) and FCM (Android) let the server wake the app when data changes — polling drains battery. For in-foreground real-time, use WebSocket or SSE. Background long-polling is the worst of both worlds.

Authentication: mobile typically uses OAuth 2.1 with PKCE (since mobile is a "public client" — no secret can be safely embedded); refresh tokens in secure keychain/keystore; short-lived access tokens.

Versioning matters more on mobile because of update lag — users don't update apps for months. Never break old clients abruptly. Version the API, maintain old versions, instrument to see the usage distribution, and sunset only when usage is near zero. Force-update dialogs are a last resort — they alienate users.

API secrets — database passwords, third-party API keys, JWT signing keys, encryption keys — are the keys to the kingdom. A leaked secret is often a company-ending event: credential-stuffing at scale, customer data exfiltration, infrastructure takeover. Managing them well is foundational security hygiene.

  • Per-service identity: each service has its own IAM role / workload identity. No "master key" that can access everything.
  • Audit logging: every secret access logged with who, when, which version. Anomalies alerted.
  • Envelope encryption: data-encryption keys (DEK) encrypt records; DEKs encrypted by a key-encryption key (KEK) in a KMS. KMS never exposes the raw KEK.

Rotation: every secret has an expiry. Long-lived secrets should rotate quarterly or monthly; dynamic secrets rotate automatically. Build rotation into applications from day one — retrofitting later is painful. Common pattern: old secret remains valid for a grace period after new one is issued so zero-downtime rotation works.

Detection:

  • Pre-commit hooks: gitleaks, detect-secrets prevent accidental commits.
  • Repository scanning: GitHub Advanced Security, TruffleHog scan history and new pushes.
  • Secret-scanning partnership: AWS, Stripe, GitHub, and others scan public repos for their key formats and auto-revoke leaked keys.
  • Canary secrets: fake keys that trigger alerts if ever used — someone accessed your repo.

Incident response: predefine the rotation runbook. When a leak is discovered, rotate within minutes, not hours; audit access logs for unauthorized use; notify customers if customer data was at risk. Speed matters: the window between leak and rotation is the blast radius.

Question 96

How do you handle real-time data synchronization in APIs?

Real-time data synchronization keeps multiple clients (browsers, mobile apps, services) in agreement about shared state as it changes. The classic example: two users editing a document — both see each other's changes within milliseconds. Under the hood, this is a distributed-systems problem involving transport, consistency, and conflict resolution.

Transport layer:

TransportDirectionFit
Server-Sent EventsServer → clientFeeds, notifications, one-way
WebSocketBidirectionalChat, live cursors, collaborative editing
WebTransport (HTTP/3)Bidirectional + unreliable optionLow-latency (games, AR); 2026 gaining adoption
Push (APNs / FCM)Server → mobileBackground, cross-platform
Long pollingPullLegacy compatibility

Sync models:

  • Server-authoritative: server holds truth; clients receive events. Simplest; conflict resolution = "server says so." Used in chat, feeds.
  • Operational Transform (OT): each client transforms its operations against others before applying. Google Docs' approach. Complex but battle-tested.
  • CRDTs (Conflict-free Replicated Data Types): mathematical data structures that converge regardless of merge order. Yjs and Automerge are the 2026 leaders. Peer-to-peer capable; offline-friendly.
  • Event sourcing + CQRS: all changes are events in an append-only log; clients replay to reconstruct state.

Architectural components of a real-time sync system:

  • Pub/sub bus: Redis Streams, Kafka, NATS fan-out events across instances.
  • Connection manager: tracks which clients are subscribed to which channels; routes events.
  • Presence: who's online, cursor positions, typing indicators — ephemeral per-connection state.
  • Persistence: latest state stored authoritatively so clients can resume after disconnect.
  • Delta or full sync on reconnect: client provides last-seen version; server sends diffs or full snapshot.

Hard problems: exactly-once delivery (generally impossible — use idempotency), ordering (use Lamport clocks or vector clocks for multi-writer), backpressure (slow clients can't keep up — drop updates or snapshot them), scale (one WebSocket instance holds tens of thousands of connections; horizontal scaling via sticky routing or Redis pub/sub).

Managed platforms (2026): Ably, Pusher, Liveblocks, Supabase Realtime, PartyKit, Convex abstract away the transport and fan-out layers. For greenfield projects these save months of infrastructure work; for deep integration with existing systems, rolling your own on WebSocket + Redis is still common.

Question 97

Explain the importance of API observability and how to achieve it.

API observability is the capacity to understand the system's behavior from the outside by examining its outputs. It's the difference between "users report the API is slow" and "I can see p99 on POST /orders spiked at 14:03 because the pricing service latency tripled, which was caused by DB connection exhaustion on pricing-db-2." Observability turns vague incidents into precise ones.

The three pillars:

PillarAnswersTools
Metrics"How much? How often? How fast?"Prometheus, Datadog, CloudWatch
Logs"What exactly happened with this request?"Loki, Elasticsearch, Datadog, Splunk
Traces"Where did time go in this request?"Jaeger, Tempo, Datadog APM, Honeycomb

Essential API metrics (RED method):

  • Rate: requests per second per endpoint.
  • Errors: error rate per endpoint, broken down by status code.
  • Duration: p50 / p95 / p99 latency per endpoint.

Combine with USE method (Utilization, Saturation, Errors) for infrastructure: CPU, memory, connection pool usage. SLOs are built on these — "99% of GET /orders respond in <200ms" — and alerts fire when error budget burns fast.

Structured logs (JSON with consistent fields) make logs queryable. Every log entry should carry reqId, userId, traceId, endpoint, and duration. AsyncLocalStorage threads these through async chains so every log from one request is correlated.

Distributed tracing shows a request's journey across services. A single request traversing API gateway → auth → order-service → inventory → payment each produces a span; the parent-child relationship is preserved; the full waterfall visualizes where time went. OpenTelemetry is the 2026 vendor-neutral standard; auto-instrumentation exists for Node, Python, Go, Java, .NET.

The fourth pillar: continuous profiling. Tools like Datadog Continuous Profiler, Pyroscope, and Grafana Profiles capture CPU and memory profiles from production at low overhead. When a code path regresses, you see exactly which function got slower — not just that the endpoint got slower.

Beyond the pillars: real user monitoring (browser-side metrics via Web Vitals), synthetic monitoring (scripted requests that catch regressions before users do), error tracking (Sentry aggregates exceptions with stack traces and breadcrumbs), event analytics (which features are used, by whom, how often). Observability isn't a tool you buy — it's a discipline of asking "if this breaks, how would I know and debug it?" at design time.

Question 98

How do you implement a custom rate limiting strategy in APIs?

Custom rate limiting addresses requirements that off-the-shelf algorithms can't express: hierarchical limits (per-user within per-organization within global), resource-weighted limits (a complex query consumes 10 tokens, a simple one 1), time-of-day adjustments, or tiered-plan differentiation. The core primitives are the same (counters, buckets); the composition is where custom value lives.

Dimensions to compose:

DimensionExample
IdentityPer-user, per-org, per-API-key, per-IP
EndpointPer-route, per-method, per-tag
CostQuery complexity, payload size, compute units
WindowPer-second, per-minute, per-day, per-month
ScopeFree tier, pro tier, enterprise burst allowances

Pattern: layered checks. Every request passes through multiple limiters in sequence; any exhausted limit rejects. Example: global DDoS protection → per-IP → per-user → per-endpoint → complexity-weighted. Each runs in O(1) via Redis.

// atomic multi-layer check via Redis Lua
const LUA = `
  local allowed = 1
  for i=1,#KEYS do
    local current = redis.call('INCR', KEYS[i])
    if current == 1 then redis.call('EXPIRE', KEYS[i], ARGV[i]) end
    if tonumber(current) > tonumber(ARGV[i + #KEYS]) then
      allowed = 0
      break
    end
  end
  return allowed
`;

// check: per-user 100/min, per-org 1000/min, per-endpoint 500/min
const allowed = await redis.eval(LUA, 3,
  `rl:u:${userId}`, `rl:o:${orgId}`, `rl:e:${endpoint}`,
  60, 60, 60,              // TTLs
  100, 1000, 500,          // limits
);
if (!allowed) return res.status(429).json({ error: "Rate limited" });

Weighted limits consume tokens proportional to cost. GraphQL query complexity analysis calculates a score per query; the limiter deducts that score from a bucket. GitHub's API uses this for its v4 GraphQL — simple queries cheap, complex ones expensive.

Tiered plans embed limits in the user record: user.plan.rateLimit = { perMin: 1000, perDay: 1_000_000 }. Each incoming request loads the user's tier once (cached) and applies their limits. Upgrades take effect immediately; paying customers see higher limits without operational work.

Burst tolerance: token-bucket naturally permits bursts. Customers appreciate "you can send 100/sec for 5 seconds, then 10/sec sustained" — burst capacity handles their morning sync without penalizing them for smooth operation. Leaky-bucket rejects bursts; pick the right shape for the use case.

Observable and transparent: return RateLimit-Remaining and RateLimit-Reset per layer (or the most-restrictive). Emit metrics per layer so you see which limit is firing. Expose limits in the dashboard and docs — customers shouldn't have to guess.

Question 99

What are the benefits and challenges of using serverless APIs?

Serverless APIs deploy individual functions rather than long-running servers. AWS Lambda, Cloudflare Workers, Vercel Functions, and Google Cloud Run Functions handle incoming HTTP requests by executing a function, scaling instances on demand, and idling to zero when no traffic. The model's appeal is operational — no servers to patch, no autoscaler to tune — but the trade-offs are real.

Benefits:

  • Zero-idle cost: pay per execution and per GB-second; APIs with sparse traffic cost cents.
  • Automatic scaling: from zero to thousands of concurrent executions in seconds. No capacity planning.
  • No server maintenance: platform handles OS patches, runtime updates, load balancing.
  • Fast rollback: deployments are immutable; reverting is a metadata change.
  • Event-driven integration: S3 uploads, queue messages, cron — all trigger functions naturally.

Challenges:

ChallengeMitigation
Cold starts (100ms-5s)Provisioned concurrency; use lightweight runtimes (Rust, Go, Node); edge platforms (Cloudflare Workers, Deno Deploy <5ms)
Execution time limits (15 min on Lambda)Break into steps; use workflow orchestrators (Step Functions, Temporal)
Connection pooling to databasesRDS Proxy, PgBouncer, or HTTP-based DBs (Neon serverless, PlanetScale)
Local developmentSAM local, Serverless Framework offline, wrangler dev
Observability fragmentationCloudWatch + structured logs + OpenTelemetry
Vendor lock-inKeep business logic framework-agnostic; handler is thin
Unpredictable bill at scaleSet concurrency and budget alarms; traffic spikes = cost spikes

Where serverless shines: webhook receivers, scheduled tasks, low-traffic internal APIs, event-driven pipelines, image processing, auth/login endpoints with bursty traffic, edge APIs (geo-distributed, low latency). Cloudflare Workers and Deno Deploy run at 300+ edge locations with <10ms cold starts; for content-light APIs this is hard to beat.

Where serverless struggles: sustained-high-traffic APIs (continuously-running containers are cheaper past ~100 req/s), long-running jobs (video encoding, ML inference with large models), APIs with heavy WebSocket usage (serverless WebSocket exists but is more expensive per connection), and stateful services. Lambda container images (10 GB limit) help with heavy dependencies but don't solve execution limits.

2026 trends: edge runtimes (Cloudflare Workers, Deno Deploy, Vercel Edge) are eroding cold-start concerns; HTTP-native databases (Neon, Turso, D1, Supabase) solve the connection-pool problem elegantly; ephemeral compute patterns (per-request Postgres branches, ephemeral Redis) are maturing. Serverless is less a single architecture and more a spectrum from traditional long-running to instant-scale edge functions; pick the slice that matches workload shape and team skills.

Question 100

How do you design an API for low latency and high throughput?

Low latency and high throughput are complementary but distinct. Latency is the time one request takes; throughput is requests per second served. A system can have one without the other — a slow endpoint that never fails at high load has throughput but poor latency; a fast endpoint that falls over at 100 req/s has latency but no throughput. Good design pursues both.

Latency reduction techniques (ordered by impact):

  1. Remove work: don't do it if you don't need to. Cache hits make the fastest implementation irrelevant. Edge caching → origin caching → in-process caching.
  2. Reduce network hops: co-locate services; deploy close to users (multi-region, CDN); avoid serial round-trips inside a request (parallel Promise.all).
  3. Optimize data access: indexes matching query patterns; denormalization for read paths; materialized views for expensive aggregations.
  4. Fast protocols: HTTP/2 or HTTP/3 for multiplexing; gRPC for inter-service; persistent connections instead of new TCP handshakes.
  5. Efficient encoding: Protobuf / MessagePack where latency matters; skip JSON parsing cost on hot paths.
  6. Low-level profiling: flame graphs find CPU hotspots; reduce allocations in hot loops.

Throughput techniques:

  • Horizontal scaling behind a load balancer — add instances; prerequisites are statelessness and externalized session data.
  • Async request handling: Node's event loop, Go's goroutines, async Python with FastAPI — one OS thread handles thousands of connections waiting on I/O.
  • Batching: DataLoader-style accumulators collect many requests into one DB call; inter-service batch APIs reduce per-request overhead.
  • Connection pooling: reuse DB connections, HTTP keep-alive to upstream services; eliminates handshake cost.
  • Queue-based load leveling: accept bursts into a queue; workers process at sustainable rate. Trades latency for throughput under spike.

Architecture patterns:

PatternImproves
Read replicasRead throughput
CQRSBoth (separate optimization)
CDNBoth (origin protection)
ShardingWrite throughput
Edge compute (Workers)Latency (global)
HTTP/2 multiplexingThroughput per connection
Streaming SSRTime-to-first-byte

Tail latency (p99) matters more than average. A p50 of 50ms with p99 of 5000ms is a user-facing disaster — 1 in 100 requests is painful, and page loads compound: 10 dependent requests, each with 1% chance of 5s, produces ~10% of page loads with a 5s request. Design for p99: set timeouts aggressively, hedge requests (send to two replicas, take the faster), remove long-tail outliers via load shedding.

Observability closes the loop. SLOs on p99 latency, RED metrics per endpoint, and continuous profiling detect regressions before they hit users. The best-performing APIs (Google Search ~100ms globally, Cloudflare Workers median 2ms) are operational achievements, not algorithmic tricks — the playbook above, executed rigorously.
