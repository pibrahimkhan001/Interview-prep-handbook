Question 1

Write an HTML document that displays "Hello, World!" in an H1 tag.

A minimal, valid HTML5 document with an <h1> heading:

<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <title>Hello Page</title>
</head>
<body>
  <h1>Hello, World!</h1>
</body>
</html>

Each piece explained:

  • <!DOCTYPE html> — tells browsers this is HTML5. Must be the first line.
  • <html lang="en"> — the root element; the lang attribute helps screen readers pronounce text correctly.
  • <meta charset="UTF-8"> — handles special characters and emoji. Always include it.
  • <title> — what appears in the browser tab.
  • <body> — visible content.

Save it as hello.html and open in any browser. That’s the foundation every HTML page builds on.

Question 2

Create a simple HTML form with text input and a submit button.

A simple form with a text input and a submit button:

<form action="/submit" method="post">
  <label for="name">Name:</label>
  <input type="text" id="name" name="name" required>

  <button type="submit">Submit</button>
</form>

Key pieces:

  • action — URL where the form data is sent.
  • method="post" — sends data in the request body; use get for search forms.
  • <label for="..."> linked to an input’s id makes clicking the label focus the input and helps screen readers.
  • name — the key under which the value is submitted (name=Alice).
  • required — browser blocks submission if the field is empty.

Always pair every input with a <label>. It’s the single easiest accessibility win in form design.

Question 3

Write the HTML to create a hyperlink that opens in a new tab.

Add target="_blank" to open a link in a new tab. Include rel="noopener noreferrer" for security.

<a href="https://example.com"
   target="_blank"
   rel="noopener noreferrer">
  Visit Example
</a>

Why rel matters:

  • noopener — prevents the new page from accessing your page via window.opener, blocking a common phishing trick.
  • noreferrer — hides your URL from the destination’s referrer log (optional but privacy-friendly).

Accessibility note: some users find new tabs confusing. Make it clear the link opens a new tab — with visible text (“opens in new tab”), an icon, or aria-label:

<a href="https://example.com"
   target="_blank"
   rel="noopener noreferrer"
   aria-label="Visit Example (opens in new tab)">
  Visit Example ↗
</a>

Reserve new tabs for external links or reference pages — never for internal navigation.

Question 4

Create an ordered list with three list items.

An ordered list uses the <ol> tag with <li> items inside:

<ol>
  <li>Preheat the oven to 180°C.</li>
  <li>Mix flour, sugar, and butter.</li>
  <li>Bake for 25 minutes.</li>
</ol>

Browsers automatically number each item: 1, 2, 3.

Useful <ol> attributes:

  • start="5" — begin numbering at 5 instead of 1.
  • reversed — count down (3, 2, 1).
  • type="A" — use letters (A, B, C) instead of numbers. Also a, I, i, 1.
<ol type="A" start="3">
  <li>Third step</li>    <!-- C -->
  <li>Fourth step</li>   <!-- D -->
</ol>

Use <ol> whenever order matters (recipes, instructions, rankings). For lists where order doesn’t matter, use <ul>.

Question 5

Create an unordered list with three list items.

An unordered list uses <ul> with <li> items. Browsers mark each item with a bullet by default.

<ul>
  <li>Apples</li>
  <li>Oranges</li>
  <li>Bananas</li>
</ul>

Use <ul> for collections where the order of items doesn’t matter — feature lists, ingredient lists, navigation menus, tags.

Styling the bullets:

<ul style="list-style-type: square;">
  <li>Red</li>
  <li>Green</li>
  <li>Blue</li>
</ul>

Common bullet styles:

  • disc — solid circle (default).
  • circle — hollow circle.
  • square — solid square.
  • none — no bullet; common for nav menus.

For navigation menus, put links inside <li> and remove bullets with CSS (list-style: none). The semantic structure is still correct, but it looks like a regular horizontal menu.

Question 6

Write the HTML to display an image with a specific width and height.

Set dimensions with the width and height attributes on <img>:

<img src="cat.jpg"
     alt="A cat napping in a sunbeam"
     width="400"
     height="300">

Values are in pixels and should not include units — width="400", never width="400px" in the attribute.

Why always include both attributes:

  • Prevents layout shift — the browser reserves space before the image loads, so the page doesn’t jump as images appear.
  • Improves Core Web Vitals — specifically the CLS (Cumulative Layout Shift) score that Google uses in ranking.
  • Fallback display — if the image fails to load, the alt-text box will match the intended size.

Combine with CSS for responsive behavior:

<img src="cat.jpg" alt="A cat" width="400" height="300"
     style="max-width: 100%; height: auto;">

The attributes set the natural aspect ratio; CSS scales the image down on narrow screens while keeping proportions. This combo gives the best of both worlds.

Question 7

Create a table with two rows and three columns.

A 2-row, 3-column table:

<table>
  <tr>
    <td>Apple</td>
    <td>Red</td>
    <td>Sweet</td>
  </tr>
  <tr>
    <td>Lime</td>
    <td>Green</td>
    <td>Sour</td>
  </tr>
</table>

Each <tr> is a row; each <td> is a cell inside that row.

With a header row and borders:

<table class="data">
  <thead>
    <tr>
      <th>Fruit</th>
      <th>Color</th>
      <th>Taste</th>
    </tr>
  </thead>
  <tbody>
    <tr><td>Apple</td><td>Red</td><td>Sweet</td></tr>
    <tr><td>Lime</td><td>Green</td><td>Sour</td></tr>
  </tbody>
</table>

<style>
  .data { border-collapse: collapse; }
  .data th, .data td { border: 1px solid #ccc; padding: 8px 12px; }
</style>

Use tables for tabular data only — never for page layout. For layout, CSS Grid or Flexbox is the correct tool.

Question 8

Write the HTML to create a form with a text input, radio buttons, and a submit button.

A form with text input, radio buttons, and a submit button:

<form action="/signup" method="post">
  <label for="name">Name:</label>
  <input type="text" id="name" name="name" required>

  <fieldset>
    <legend>Preferred contact method:</legend>

    <input type="radio" id="email" name="contact" value="email" checked>
    <label for="email">Email</label>

    <input type="radio" id="phone" name="contact" value="phone">
    <label for="phone">Phone</label>

    <input type="radio" id="mail" name="contact" value="mail">
    <label for="mail">Mail</label>
  </fieldset>

  <button type="submit">Sign Up</button>
</form>

Radio button rules:

  • Radios sharing the same name are mutually exclusive — only one can be selected.
  • Each needs its own id and a linked <label>.
  • Use checked to set the default selection.
  • Wrap related radios in <fieldset> + <legend> — screen readers announce the legend as context for every radio.

Submitted data looks like name=Alice&contact=email.

Question 9

Create a navigation bar with links to "Home," "About," and "Contact" pages.

Use semantic <nav> around a list of links:

<nav aria-label="Main navigation">
  <ul>
    <li><a href="/">Home</a></li>
    <li><a href="/about">About</a></li>
    <li><a href="/contact">Contact</a></li>
  </ul>
</nav>

<style>
  nav ul {
    display: flex;
    gap: 1.5em;
    list-style: none;
    padding: 0;
  }
  nav a {
    text-decoration: none;
    color: #0066cc;
  }
  nav a:hover { text-decoration: underline; }
</style>

Why this structure:

  • <nav> tells screen readers and search engines “this is a navigation region.”
  • aria-label="Main navigation" distinguishes this nav from others (e.g., a footer nav).
  • A <ul> of links is the standard menu pattern — preserves structure when CSS is disabled.
  • display: flex turns the vertical bullet list into a horizontal bar.

To mark the current page, add aria-current="page" to the active link — screen readers announce it, and you can style it with a[aria-current="page"] { font-weight: bold; }.

Question 10

Write the HTML to embed a video from YouTube.

Embed a YouTube video using an <iframe> with YouTube’s embed URL:

<iframe
  width="560"
  height="315"
  src="https://www.youtube.com/embed/dQw4w9WgXcQ"
  title="YouTube video player"
  frameborder="0"
  allow="accelerometer; autoplay; clipboard-write;
         encrypted-media; gyroscope; picture-in-picture"
  allowfullscreen>
</iframe>

The URL format is https://www.youtube.com/embed/VIDEO_ID — replace VIDEO_ID with the id from the video’s regular URL (after watch?v=).

Responsive embed (keeps 16:9 ratio on any screen):

<div class="video-wrapper">
  <iframe src="https://www.youtube.com/embed/dQw4w9WgXcQ"
          title="YouTube video player"
          allowfullscreen></iframe>
</div>

<style>
  .video-wrapper {
    position: relative;
    padding-bottom: 56.25%;    /* 16:9 ratio */
    height: 0;
  }
  .video-wrapper iframe {
    position: absolute;
    inset: 0;
    width: 100%;
    height: 100%;
  }
</style>

Always provide a meaningful title on the iframe — it’s what screen readers announce.

Question 11

Create a form with a dropdown list containing three options.

A dropdown list uses <select> with <option> children:

<form>
  <label for="country">Country:</label>
  <select id="country" name="country">
    <option value="">-- Choose one --</option>
    <option value="us">United States</option>
    <option value="uk">United Kingdom</option>
    <option value="ca">Canada</option>
  </select>

  <button type="submit">Save</button>
</form>

Key attributes:

  • value on <option> — what gets submitted; the visible text can differ.
  • selected — preselects an option.
  • disabled — grays out an option so it can’t be chosen.
  • multiple on <select> — allows multiple selections; the UI becomes a list box.

Group options with <optgroup>:

<select name="food">
  <optgroup label="Fruits">
    <option value="apple">Apple</option>
    <option value="banana">Banana</option>
  </optgroup>
  <optgroup label="Vegetables">
    <option value="carrot">Carrot</option>
  </optgroup>
</select>

An empty default option is helpful for required fields — it forces the user to make an actual choice.

Question 12

Write the HTML to create a table with a header row and a footer row.

A full table with both header (<thead>) and footer (<tfoot>):

<table>
  <thead>
    <tr>
      <th>Item</th>
      <th>Quantity</th>
      <th>Price</th>
    </tr>
  </thead>

  <tbody>
    <tr><td>Notebook</td><td>2</td><td>$8.00</td></tr>
    <tr><td>Pen</td><td>5</td><td>$5.00</td></tr>
    <tr><td>Eraser</td><td>3</td><td>$1.50</td></tr>
  </tbody>

  <tfoot>
    <tr>
      <th colspan="2">Total</th>
      <td>$14.50</td>
    </tr>
  </tfoot>
</table>

Why all three sections matter:

  • Semantic meaning — screen readers announce the header, body, and footer as separate regions.
  • Print layout — browsers repeat <thead> and <tfoot> at the top and bottom of every printed page.
  • Styling — target each region distinctly: thead th { background: #eee; }, tfoot td { font-weight: bold; }.

The colspan="2" on the footer lets “Total” span two columns so the total price lines up under the Price column.

Question 13

Create a form with a textarea and a submit button.

A <textarea> is the multi-line text input:

<form action="/feedback" method="post">
  <label for="message">Your message:</label>
  <textarea id="message"
            name="message"
            rows="5"
            cols="40"
            placeholder="Tell us what you think..."
            required></textarea>

  <button type="submit">Send</button>
</form>

Key attributes:

  • rows — initial visible height in text lines.
  • cols — initial visible width in characters.
  • placeholder — hint text shown when empty.
  • maxlength — character limit.
  • required — must be filled before submit.

Unlike <input>, <textarea> has a closing tag. Anything between the tags becomes initial content — but be careful:

<textarea>Already typed here.</textarea>

Don’t put whitespace between the tags unless you want it to appear in the field. For modern designs, CSS resize: vertical limits how users can resize, and width: 100% makes it responsive.

Question 14

Write the HTML to display an image with alt text.

Always include the alt attribute on <img> — it’s required by HTML and critical for accessibility:

<img src="beach.jpg"
     alt="A deserted tropical beach with palm trees and turquoise water">

When is the alt text used?

  • Screen readers announce it to blind users.
  • Broken images display the alt text in place of the missing image.
  • Text-only browsers show alt text instead.
  • Search engines read it to understand image content.

Writing good alt text:

  • Describe what’s important about the image in context — not every detail.
  • Don’t start with “image of...” — screen readers already say “image.”
  • For decorative images (pure visual flourishes), use empty alt: alt="". Screen readers skip them.
<!-- Meaningful image -->
<img src="ceo.jpg" alt="Jane Doe, CEO of Acme Corporation">

<!-- Decorative divider -->
<img src="divider.png" alt="">

Never use alt="image" or alt="photo" — it’s worse than nothing.

Question 15

Create a basic HTML document structure with a head, body, and title.

The standard skeleton of an HTML5 document:

<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>My Page</title>
  <link rel="stylesheet" href="styles.css">
</head>
<body>
  <h1>Welcome</h1>
  <p>This is my page.</p>

  <script src="main.js"></script>
</body>
</html>

Each section:

ElementPurpose
<!DOCTYPE html>Declares HTML5; must be first
<html lang="en">Root element; lang helps accessibility and search
<head>Metadata: title, charset, viewport, links to CSS
<title>Tab title and search result heading
<body>Visible content

The viewport meta tag is essential for mobile — without it, phones render pages as zoomed-out desktop sites. Put scripts at the end of <body> (or use defer in the head) so they don’t block page rendering.

Question 16

Write the HTML to create a hyperlink that links to an email address.

Use the mailto: URL scheme on an anchor. Clicking it opens the user’s email client with a new message:

<a href="mailto:hello@example.com">Email us</a>

Pre-fill subject and body:

<a href="mailto:support@example.com
         ?subject=Help%20request
         &body=Hi%20team%2C%0A%0A">
  Contact Support
</a>

URL-encoded characters to know:

  • %20 — space
  • %0A — newline
  • %2C — comma
  • %3F — question mark
  • & — separator between parameters

CC, BCC, and multiple recipients:

<a href="mailto:a@x.com,b@x.com?cc=c@x.com&bcc=d@x.com">
  Email team
</a>

Caveat: mailto: requires the user to have a configured email client. On kiosks or shared devices, clicking does nothing. For forms where users type their message in the browser, a regular HTML form is more reliable.

Question 17

Create a form with checkboxes and a submit button.

Checkboxes let users pick multiple independent options. Each has the same name but a different value:

<form action="/preferences" method="post">
  <fieldset>
    <legend>Subscribe to:</legend>

    <input type="checkbox" id="news" name="subs" value="news">
    <label for="news">Newsletter</label>

    <input type="checkbox" id="deals" name="subs" value="deals" checked>
    <label for="deals">Product deals</label>

    <input type="checkbox" id="tips" name="subs" value="tips">
    <label for="tips">Weekly tips</label>
  </fieldset>

  <input type="checkbox" id="agree" name="agree" value="yes" required>
  <label for="agree">I agree to the terms</label>

  <button type="submit">Save preferences</button>
</form>

Checkbox essentials:

  • Each checkbox needs its own id linked to a <label>.
  • Shared name + distinct value = group; the form submits one value per checked box.
  • checked sets the default state.
  • Unlike radios, checkboxes are independent — multiple can be selected.

Submitted data for the above: subs=deals&subs=tips&agree=yes (unchecked boxes send nothing).

Question 18

Write the HTML to create a table with merged cells using rowspan and colspan.

Merge cells with rowspan (vertical merge) or colspan (horizontal merge):

<table>
  <tr>
    <th rowspan="2">Name</th>
    <th colspan="2">Score</th>
  </tr>
  <tr>
    <th>Math</th>
    <th>English</th>
  </tr>

  <tr>
    <td>Alice</td>
    <td>95</td>
    <td>88</td>
  </tr>
  <tr>
    <td>Bob</td>
    <td colspan="2" style="text-align: center;">Absent</td>
  </tr>
</table>

Breaking down the merges:

  • <th rowspan="2">Name</th> — the Name header spans both rows of the header area.
  • <th colspan="2">Score</th> — Score spans two columns (Math + English).
  • In the row-2 header, only Math and English appear — Name was already covered by rowspan.
  • Bob’s “Absent” cell spans both score columns.

Key rule: when a cell spans multiple rows or columns, skip the covered cells in subsequent rows — don’t write a <td> where the spanning cell already occupies space, or the table will be misaligned.

Question 27

Create a responsive image using the srcset attribute.

The srcset attribute lets the browser pick the best image size based on the device:

<img
  src="photo-800.jpg"
  srcset="photo-400.jpg  400w,
          photo-800.jpg  800w,
          photo-1200.jpg 1200w,
          photo-2400.jpg 2400w"

         (max-width: 1200px) 50vw,

<img src="logo.png"
     srcset="logo.png 1x, logo@2x.png 2x, logo@3x.png 3x"
     alt="Logo">

Responsive images cut bandwidth dramatically on mobile — a phone gets a 40 KB image where a desktop gets 400 KB.

        err.textContent = `Must be at least ${min}`;
        valid = false;
      } else if (max && (input.type === "number" ? Number(value) > max : value.length > max)) {
        err.textContent = `Must be at most ${max}`;
        valid = false;
      } else if (rx && !new RegExp(rx).test(value)) {
        err.textContent = "Invalid characters";
        valid = false;
      }
    }

    if (!valid) event.preventDefault();
    return valid;
  }
</script>

Why use data-* attributes:

  • Validation rules live in HTML where you can read them at a glance.
  • Accessible in JS via element.dataset.* (camelCase conversion: data-min-length → dataset.minLength).
  • Clean separation — HTML carries config; JS carries logic.

Data attributes are also handy for tracking click targets, component state, and feature flags without polluting real HTML attributes.

Question 93

Create a form with a tabindex attribute for input fields.

The tabindex attribute controls the order of keyboard focus when users press Tab:

<form>
  <!-- Natural tab order follows source order -->
  <label for="name">Name:</label>
  <input type="text" id="name" name="name">

  <label for="email">Email:</label>
  <input type="email" id="email" name="email">

  <!-- tabindex="-1" removes from tab order but still focusable via JS -->
  <button type="button" tabindex="-1">Skip focus</button>

  <!-- tabindex="0" places in natural order (for non-interactive elements) -->
  <div tabindex="0" role="button" onclick="save()">Save</div>

  <button type="submit">Submit</button>
</form>

tabindex values:

ValueMeaning
tabindex="0"Include in natural tab order (useful for custom interactive elements)
tabindex="-1"Skip in tab order but allow .focus() via JS
tabindex="1+"Explicit order (avoid — causes accessibility issues)

Positive tabindex is almost always wrong. It overrides natural document order, making the tab sequence unpredictable. Sighted users and screen reader users get lost. Stick to 0 and -1.

Common patterns:

<!-- Custom button made accessible -->
<div role="button"
     tabindex="0"
     onclick="activate()"
     onkeydown="if (event.key === 'Enter' || event.key === ' ') activate()">
  Click me
</div>

<!-- Modal initial focus -->
<div id="modal" tabindex="-1">
  <h2>Confirm action</h2>
  ...
</div>
<script>modal.focus();</script>

Native elements (<button>, <a href>, form inputs) are keyboard-focusable without any tabindex — prefer them over ARIA-tagged divs whenever possible.

Create a form with a disabled attribute for input fields.

The disabled attribute prevents interaction with a form field — users can’t type, click, or submit it:

<form action="/account" method="post">
  <label for="email">Email (cannot be changed):</label>
  <input type="email"
         id="email"
         name="email"
         value="user@example.com"
         disabled>

         id="name"
         name="name"

    <input type="text" id="account-id" value="42-ABCD-1234">

    <label for="plan">Plan:</label>
    <select id="plan">
      <option>Pro</option>
    </select>
  </fieldset>

  <button type="submit">Save</button>
  <button type="button" disabled>Advanced (coming soon)</button>

Behavior of disabled:

  • Field is grayed out.
  • Keyboard and mouse interaction blocked.
  • Not submitted with the form — the value is omitted entirely.
  • Not focusable; Tab skips over it.
  • Applied to <fieldset>, it disables all contained fields.

disabled vs readonly:

disabledreadonly
Submits with formNoYes
User can focusNoYes
Visual styleGrayed outLooks normal
Screen readersAnnounced as disabledAnnounced as read-only

Use disabled when the field genuinely shouldn’t be part of the submission. Use readonly when the field should submit its current value but not be editable (e.g., a confirmation screen where values are visible but locked).

Question 97

Create a form with a readonly attribute for input fields.

The readonly attribute makes a field non-editable but still focusable and still submitted:

<form action="/confirm" method="post">
  <h2>Review your order</h2>

  <label for="order-id">Order ID:</label>
  <input type="text"
         id="order-id"
         name="orderId"
         value="ORD-2026-00042"
         readonly>

  <label for="total">Total amount:</label>
  <input type="text"
         id="total"
         name="total"
         value="$124.99"
         readonly>

  <label for="date">Order date:</label>
  <input type="date"
         id="date"
         name="date"
         value="2026-04-20"
         readonly>

  <label for="notes">Add notes:</label>
  <textarea id="notes" name="notes" rows="3"></textarea>

  <button type="submit">Confirm order</button>
</form>

Effects of readonly:

  • User can focus and select the value (copy it out).
  • Can’t type, modify, or erase the value.
  • The value is submitted with the form.
  • Validation (e.g., required, pattern) still applies.
  • Visually looks the same as a normal input by default — style it differently if appropriate.

Style readonly fields so users can tell:

input[readonly], textarea[readonly] {
  background-color: #f5f5f5;
  color: #555;
  cursor: default;
}

Use cases:

  • Confirmation screens — users review but don’t edit.
  • System-generated values — order numbers, timestamps, SKUs.
  • Pre-filled user data you want to submit unchanged (email on account update).

Always re-check server-side — client-side readonly can be bypassed via devtools. Use readonly for UX convenience, not as a security mechanism.

Question 99

Create a form with a formnovalidate attribute for specific input fields.

The formnovalidate attribute on a specific submit button disables HTML5 validation for just that submission — other submits still validate:

<form action="/posts" method="post">
  <label for="title">Title:</label>
  <input type="text"
         id="title"
         name="title"
         required
         minlength="5">

  <label for="content">Content:</label>
  <textarea id="content"
            name="content"
            required
            minlength="50"
            rows="10"></textarea>

  <label for="tags">Tags:</label>
  <input type="text" id="tags" name="tags" pattern="[a-z,\s]+">

  <div class="form-buttons">
    <!-- Full validation required -->
    <button type="submit"
            formaction="/posts/publish">
      Publish
    </button>

    <!-- Skip validation — save even if incomplete -->
    <button type="submit"
            formaction="/posts/draft"
            formnovalidate>
      Save as draft
    </button>

    <!-- Validation required — preview uses full validation -->
    <button type="submit"
            formaction="/posts/preview"
            formtarget="_blank">
      Preview
    </button>
  </div>
</form>

Why formnovalidate exists:

  • Draft saves — users save incomplete posts to finish later.
  • Mid-wizard save — save state before all steps are complete.
  • Cancel / back navigation — post partial state to track abandonment.

Related form-override attributes on submit buttons:

AttributeOverrides
formactionThe form’s action
formmethodThe form’s method
formtargetThe form’s target
formenctypeThe form’s enctype
formnovalidateThe form’s novalidate

These let multiple buttons in one form submit to different endpoints with different behavior — no extra JavaScript needed. The server can decide whether to strictly validate based on which endpoint received the data.
