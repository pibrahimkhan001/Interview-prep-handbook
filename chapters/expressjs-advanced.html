Question 1

How do you implement a custom middleware to log request details and response time in ExpressJS?

A production-grade request logger does four things: generate or propagate a correlation id, measure time with a monotonic clock, log as structured JSON so aggregators can parse it, and redact secrets.

import express from "express";
import crypto from "node:crypto";
import { AsyncLocalStorage } from "node:async_hooks";

const als = new AsyncLocalStorage();
export const currentReqId = () => als.getStore()?.reqId;

function logger() {
  return (req, res, next) => {
    const reqId = req.headers["x-request-id"] || crypto.randomUUID();
    res.setHeader("X-Request-Id", reqId);
    req.id = reqId;

    const start = process.hrtime.bigint();

    res.on("finish", () => {
      const durMs = Number(process.hrtime.bigint() - start) / 1e6;
      const level = res.statusCode >= 500 ? "error" : res.statusCode >= 400 ? "warn" : "info";

      process.stdout.write(JSON.stringify({
        time:   new Date().toISOString(),
        level, reqId,
        method: req.method,
        path:   req.originalUrl,
        status: res.statusCode,
        durMs:  Number(durMs.toFixed(1)),
        bytes:  Number(res.getHeader("content-length")) || 0,
        ip:     req.ip,
        ua:     req.headers["user-agent"],
        userId: req.user?.id,
      }) + "\n");
    });

    als.run({ reqId }, next);                    // propagate id through async calls
  };
}

Key design choices: use process.hrtime.bigint() (monotonic, nanosecond-precise) instead of Date.now() which can jump backward on NTP sync. Log on res.on("finish") so the final status code is captured after error middleware runs. AsyncLocalStorage threads the id through nested async calls without polluting function signatures — essential for correlating DB queries, outbound HTTP calls, and background work back to the originating request.

Production refinements: use pino-http (5-10x faster than console.log, async stdout with worker threads, declarative redaction). Sample verbose logs in high-traffic endpoints. Emit metrics (request count, p50/p95/p99 latency by route) to your observability platform alongside logs — logs for investigation, metrics for alerting.

Question 2

How can you optimize an ExpressJS application for performance and scalability?

Express performance tuning operates at four layers: framework (what Express does), process (how Node uses CPU/memory), I/O (DB, cache, HTTP), and edge (what never reaches Node).

LayerTechniques
FrameworkDisable x-powered-by; enable view cache in production; use a trie-based router like Express 5 or Fastify; avoid synchronous middleware in hot paths
ProcessRun one Node process per CPU core with PM2 cluster mode or Kubernetes replicas; offload CPU-bound work to worker threads; set --max-old-space-size above default 1.5 GB for heap-heavy workloads
I/OPooled DB connections (pg/mysql2); Redis cache with cache-aside and stampede prevention; keep-alive HTTP agents; batched DB queries with DataLoader pattern
EdgeCDN for static assets, HTML caching on short TTLs, rate limiting and bot mitigation at the load balancer; gzip/brotli at the edge, not in Node

Profiling workflow: measure before optimizing. Use clinic.js (doctor for event-loop lag, flamegraphs for CPU, bubbleprof for async). Set up continuous profiling in production (Datadog, Pyroscope) — the hot path in dev rarely matches prod. Track p95/p99 latency, not averages; tail latency reveals garbage-collection pauses and slow downstream calls.

Common wins: replace JSON.stringify with fast-json-stringify (schema-compiled, 2-3x faster), swap bcrypt cost for argon2id, cache rendered templates, batch notification sends. Switching from Express to Fastify typically yields 15-40% throughput with minimal API change.

Question 3

How do you handle multiple asynchronous operations in middleware using async/await in ExpressJS?

Middleware with multiple async steps needs disciplined error propagation and careful parallelism. The default pattern is sequential await; parallelize only when operations are independent.

// sequential — each step depends on the previous
app.get("/dashboard", async (req, res, next) => {
  try {
    const user = await db.user.findUnique({ where: { id: req.user.sub } });
    if (!user) return res.sendStatus(404);
    const orders = await db.order.findMany({ where: { userId: user.id } });
    res.json({ user, orders });
  } catch (err) { next(err); }
});

// parallel — Promise.all when operations are independent
app.get("/profile", async (req, res, next) => {
  try {
    const [user, stats, notifications] = await Promise.all([
      db.user.findUnique({ where: { id: req.user.sub } }),
      db.stats.forUser(req.user.sub),
      db.notification.findMany({ where: { userId: req.user.sub, unread: true } }),
    ]);
    res.json({ user, stats, notifications });
  } catch (err) { next(err); }
});

// partial-failure tolerance — Promise.allSettled when some operations can fail
const [userR, adsR] = await Promise.allSettled([loadUser(), loadAds()]);
const user = userR.status === "fulfilled" ? userR.value : null;
const ads  = adsR.status  === "fulfilled" ? adsR.value  : [];

Rules that matter: always wrap async handlers in try/catch in Express 4 (rejections don't auto-forward). Express 5 auto-forwards thrown errors but explicit handling still improves readability. Use Promise.all for independent operations — serial awaits waste latency when nothing shares state. Use Promise.allSettled when partial success is acceptable (e.g., a dashboard where a failing widget shouldn't break the whole page).

Cancellation: pass an AbortSignal through to every downstream call and abort it on req.on("close"). Without this, a client who drops the connection still incurs full DB/HTTP cost — a subtle amplifier during outages. For Prisma 5+ and undici, abort signals are first-class; for older ORMs you may need a wrapper.

Question 4

What are the best practices for error handling in an ExpressJS application?

Robust error handling in Express rests on four principles: centralize the response logic, normalize errors from different sources, distinguish operational from programmer errors, and observe failures with enough context to debug.

// 1. typed error class — carries status + public code
class HttpError extends Error {
  constructor(message, status = 500, code) {
    super(message);
    this.status = status; this.code = code;
  }
}

// 2. one async wrapper, no try/catch clutter in handlers
const ah = (fn) => (req, res, next) =>
  Promise.resolve(fn(req, res, next)).catch(next);

// 3. single error middleware normalizes then formats
app.use((err, req, res, _next) => {
  if (err.name === "ZodError")        err = new HttpError("Invalid input", 400, "VALIDATION");
  if (err.code === "P2002")           err = new HttpError("Already exists", 409, "CONFLICT");
  if (err.code === "P2025")           err = new HttpError("Not found", 404, "NOT_FOUND");
  if (err.name === "TokenExpiredError") err = new HttpError("Session expired", 401, "AUTH");

  const status = err.status || 500;
  const isServerError = status >= 500;

  // log full error server-side with context
  if (isServerError) req.log?.error({ err, reqId: req.id });
  else               req.log?.warn({ status, code: err.code, reqId: req.id }, err.message);

  // never leak stack traces or internal details to the client
  res.status(status).json({
    error: isServerError ? "Internal Server Error" : err.message,
    code:  err.code,
    reqId: req.id,
  });
});

// 4. last-resort crash handler for unhandled rejections
process.on("unhandledRejection", (err) => {
  console.error("unhandledRejection:", err);
  // log, then exit — let your supervisor restart
});

Operational vs programmer errors: an operational error is expected (DB unavailable, invalid input) — handle and respond. A programmer error is a bug (undefined is not a function) — log, alert, and let the process die so a supervisor restarts a clean one. Don't try to "recover" from programmer errors; state is probably corrupt.

Production refinements: include a request id in every error response body so users can reference it in support tickets. Use Sentry for error tracking with release tags and source maps. Never log passwords, tokens, or full request bodies at error time — use a declarative redaction list (pino's redact option). Rate-limit error alerts so a single bad deploy doesn't page everyone 10,000 times.

Question 5

How do you secure an ExpressJS application against common vulnerabilities like XSS, CSRF, and SQL injection?

Security is layered defense — no single middleware stops everything. Address the OWASP Top 10 systematically, and treat every client input as hostile.

ThreatMitigation
XSSAuto-escaping template engines; strict CSP via helmet; avoid dangerouslySetInnerHTML; sanitize rich-text with DOMPurify; use HttpOnly cookies so JS can't read them
CSRFToken-based auth (Authorization: Bearer) is immune; cookie-based sessions need CSRF tokens via csrf-csrf plus SameSite=lax
SQL injectionParameterized queries always ($1, $2) — ORMs (Prisma, Drizzle) do this by default; never string-concat SQL
NoSQL injectionCoerce types before passing to MongoDB (reject {"$ne": null} passed as a filter); validate with Zod/Joi first
Broken authbcrypt/argon2id for passwords (cost 12+); short JWT lifetimes + refresh rotation; aggressive rate limits on login
SSRFBlock private-IP ranges and localhost in outbound fetches; allowlist destinations when possible
Open redirectValidate returnTo / continueUrl against an allowlist; reject external URLs
Dependency risknpm audit, Snyk or Dependabot, SBOM generation, lockfiles committed, signed package verification

Always-on middleware baseline: helmet() for security headers (CSP, HSTS, X-Content-Type-Options), cors() with an origin allowlist, express-rate-limit (Redis-backed), express.json({ limit: "100kb" }) to cap body size, and input validation at every route with Zod. Disable x-powered-by (app.disable("x-powered-by")) so attackers can't fingerprint Express.

Operational: rotate secrets regularly (JWT keys, DB passwords, API keys); store them in a secret manager (AWS SSM, HashiCorp Vault) not env files on disk. Log security-relevant events (failed logins, authorization denials, rate-limit triggers) with enough detail to detect probing. Penetration-test before launch, and again every time scope changes materially.

Question 6

Explain the purpose and usage of the Helmet middleware in securing an ExpressJS application.

helmet is a collection of 15+ small middlewares that each set one security-related HTTP header with sensible defaults. Installing it eliminates a family of easy mistakes — forgetting X-Content-Type-Options, shipping a missing CSP, or letting browsers frame your pages.

// npm install helmet
import helmet from "helmet";

app.use(helmet({
  contentSecurityPolicy: {
    useDefaults: true,
    directives: {
      "default-src": ["'self'"],
      "script-src":  ["'self'", "'unsafe-inline'", "cdn.example.com"],
      "img-src":     ["'self'", "data:", "user-uploads.example.com"],
      "connect-src": ["'self'", "api.example.com"],
      "frame-ancestors": ["'none'"],
    },
  },
  crossOriginResourcePolicy: { policy: "same-origin" },
  referrerPolicy: { policy: "strict-origin-when-cross-origin" },
  hsts: { maxAge: 31536000, includeSubDomains: true, preload: true },
  // disable the ones you don't need — defaults are conservative
}));

What each default buys you:

  • CSP — arguably the single most important header; blocks injected scripts. Start in Content-Security-Policy-Report-Only mode, iterate until clean, then enforce.
  • HSTS — forces HTTPS for a year; include preload only after submitting to the HSTS preload list.
  • X-Content-Type-Options: nosniff — stops browsers from second-guessing MIME types.
  • X-Frame-Options: DENY (now subsumed by CSP frame-ancestors) — prevents clickjacking.
  • Referrer-Policy — stops leaking full URLs to third parties.

Production realities: CSP is where apps struggle — inline scripts, third-party SDKs, and analytics require explicit allowances. Use nonces (generated per request) over 'unsafe-inline' for inline scripts you control. Set up CSP reporting (report-uri) to catch violations. Disable helmet modules that conflict with your CDN or analytics setup (e.g., crossOriginEmbedderPolicy breaks many third-party resources). Helmet is a floor, not a ceiling — you still need app-level security.

Question 7

How can you implement request validation and sanitization using express-validator in an ExpressJS application?

express-validator is a middleware built on validator.js that chains per-field validators and sanitizers. It's a solid choice for JavaScript projects; TypeScript projects typically prefer Zod for type inference.

import { body, query, param, validationResult } from "express-validator";

function handleValidation(req, res, next) {
  const errors = validationResult(req);
  if (!errors.isEmpty()) {
    return res.status(400).json({
      error: "Validation failed",
      issues: errors.array().map((e) => ({ field: e.path, message: e.msg })),
    });
  }
  next();
}

app.post("/signup",
  body("email").isEmail().normalizeEmail(),
  body("password").isLength({ min: 8, max: 128 })
    .matches(/[A-Z]/).withMessage("needs an uppercase letter")
    .matches(/[0-9]/).withMessage("needs a digit"),
  body("name").trim().isLength({ min: 1, max: 100 }).escape(),
  body("age").optional().isInt({ min: 13, max: 120 }).toInt(),
  handleValidation,
  async (req, res) => { /* req.body is cleaned + coerced */ }
);

Validation vs sanitization: validators check (isEmail, isLength); sanitizers transform (trim, normalizeEmail, toInt, escape). By the time your handler runs, req.body contains cleaned values — use them, don't re-fetch the raw input.

LibraryBest for
express-validatorJavaScript projects, chainable DSL, built-in sanitizers
ZodTypeScript projects — free type inference, composable schemas
JoiLarge declarative schemas, nested objects, long-running projects
AJVPeak performance, JSON Schema contracts (OpenAPI-first)
YupAsync validators (uniqueness checks), React Hook Form integration

Production gotchas: always call validationResult(req) — forgetting it silently skips validation. Collect all errors (don't abortEarly) so users see every problem at once. For complex rules that cross fields (password confirmation, conditional required fields), use a declarative schema validator (Zod/Joi) over express-validator chains, which get unwieldy. Validate every input surface: body, query, params, headers — not just the obvious one.

Question 8

Describe how you would handle role-based access control (RBAC) in an ExpressJS application.

Role-based access control separates authentication (who are you?) from authorization (what can you do?). A mature implementation has three layers: role (coarse), permission (fine-grained action), and resource ownership (per-record check).

// role hierarchy with permissions
const ROLE_PERMS = {
  user:   ["post:read", "post:create"],
  editor: ["post:read", "post:create", "post:update", "comment:moderate"],
  admin:  ["*"],
};

function requireRole(minRole) {
  const order = { user: 0, editor: 1, admin: 2 };
  return (req, res, next) => {
    if (!req.user) return res.sendStatus(401);
    if (order[req.user.role] < order[minRole]) return res.sendStatus(403);
    next();
  };
}

function requirePerm(...needed) {
  return (req, res, next) => {
    if (!req.user) return res.sendStatus(401);
    const have = new Set(ROLE_PERMS[req.user.role] ?? []);
    const ok = have.has("*") || needed.every(p => have.has(p));
    if (!ok) return res.sendStatus(403);
    next();
  };
}

// resource-scoped — "edit your own post, unless you're an editor"
async function canEditPost(req, res, next) {
  const post = await db.post.findUnique({ where: { id: req.params.id } });
  if (!post) return res.sendStatus(404);
  const isOwner  = post.authorId === req.user.sub;
  const isEditor = ROLE_PERMS[req.user.role]?.includes("post:update");
  if (!isOwner && !isEditor) return res.sendStatus(403);
  req.post = post;
  next();
}

RBAC vs ABAC vs ReBAC:

ModelDecides onWhen to use
RBACUser's roleMost apps; simple, auditable
ABACAttributes (role, department, time, IP)Enterprise, compliance-heavy
ReBACRelationships (owner, editor, viewer)Document sharing (Google Drive, Notion)

Scaling beyond roles: for "users can edit only their own posts, except moderators can edit any", pure RBAC breaks down. Use CASL (ability.can("update", post)) or Casbin for policy-based auth. For Google-Drive-style sharing, OpenFGA or SpiceDB model relationships explicitly. Always authorize server-side; never trust client-sent role fields. Log authorization denials with enough context to distinguish probing from configuration drift.

Question 9

How do you implement JWT-based authentication in an ExpressJS application?

JWT authentication trades server-side session state for self-contained, cryptographically-signed tokens. The payload holds claims (user id, role, expiry); the signature ensures integrity. Stateless verification means any server can validate without a shared session store — key to horizontal scaling.

import jwt from "jsonwebtoken";

// issue on login
const token = jwt.sign(
  { sub: user.id, role: user.role },
  process.env.JWT_SECRET,
  { expiresIn: "15m", issuer: "api.example.com", audience: "web" }
);

// verify middleware
function requireAuth(req, res, next) {
  const [scheme, token] = (req.headers.authorization || "").split(" ");
  if (scheme !== "Bearer" || !token) return res.sendStatus(401);
  try {
    req.user = jwt.verify(token, process.env.JWT_SECRET, {
      issuer:   "api.example.com",
      audience: "web",
      algorithms: ["HS256"],                   // pin algorithm — prevents "none" attack
    });
    next();
  } catch (err) {
    res.status(401).json({ error: err.name === "TokenExpiredError" ? "expired" : "invalid" });
  }
}

Question 10

Explain how to manage and rotate JWT tokens securely in ExpressJS.

Secure JWT rotation combines short-lived access tokens with long-lived refresh tokens, plus server-side tracking to enable revocation. The access token is stateless and validated by signature alone; the refresh token is tracked in a database so it can be revoked individually.

// login issues both
const jti = crypto.randomUUID();
const access  = jwt.sign({ sub: user.id }, ACCESS_SECRET,  { expiresIn: "15m" });
const refresh = jwt.sign({ sub: user.id, jti }, REFRESH_SECRET, { expiresIn: "30d" });

await db.refreshToken.create({
  data: { jti, userId: user.id, expiresAt: new Date(Date.now() + 30 * 86400_000) },
});

// HttpOnly cookie holds the refresh token (JS-inaccessible)
res.cookie("refreshToken", refresh, {
  httpOnly: true, secure: true, sameSite: "strict", path: "/auth",
});

// refresh endpoint — rotate both tokens on every use
app.post("/auth/refresh", async (req, res) => {
  const payload = jwt.verify(req.cookies.refreshToken, REFRESH_SECRET);
  const record = await db.refreshToken.findUnique({ where: { jti: payload.jti } });

  // replay detection — if revoked, assume theft, revoke ALL tokens for this user
  if (!record || record.revokedAt) {
    await db.refreshToken.updateMany({ where: { userId: payload.sub }, data: { revokedAt: new Date() } });
    return res.sendStatus(401);
  }

  // rotation — old refresh becomes invalid, issue new pair
  await db.refreshToken.update({ where: { jti: payload.jti }, data: { revokedAt: new Date() } });
  const newJti = crypto.randomUUID();
  /* ... issue new access + refresh tokens, set cookie ... */
});

The four rotation strategies:

  • Simple expiry: short access tokens, no refresh — users re-login often. Annoying but simplest.
  • Sliding refresh: refresh extends the session. Simple but a stolen refresh token grants indefinite access.
  • Rotation (shown above): every refresh invalidates the old one. A leaked refresh token is detectable on replay.
  • Rotation + family tracking: each refresh links to a family; detecting reuse revokes the whole family — strongest pattern.

Key revocation: for emergency revocation (compromised signing key, vulnerability), issue a new key and add the old key to a verification-only list for a grace period. Use JWKS (JSON Web Key Sets) if multiple services verify; the issuer publishes current keys at a known URL, clients cache them. Never invent your own signing scheme; use well-maintained libraries (jose, jsonwebtoken) and Auth.js / better-auth for full flows.

What are the strategies for handling CORS in an ExpressJS application?

CORS (Cross-Origin Resource Sharing) exists because browsers enforce the same-origin policy. CORS relaxes it selectively — the server tells the browser "this cross-origin request is allowed" via response headers. Strategy varies by API type.

import cors from "cors";

  origin: (origin, cb) => {
    const allowed = ["https://app.example.com", "https://admin.example.com"];
    if (!origin || allowed.includes(origin)) cb(null, true);
    else cb(new Error("CORS: origin not allowed"));
  },
  credentials: true,                           // allow cookies / Authorization
  maxAge: 86400,                               // cache preflight 24h
}));

// 2. Wide-open for genuinely public APIs (no auth, read-only)
app.use("/public-api", cors({ origin: "*" }));

// 3. Per-route — different rules for webhook endpoints
app.post("/webhook", cors({ origin: false }), handler);   // same-origin only

Common pitfalls:

  • origin: "*" + credentials: true is illegal per spec — browsers reject the combo. Always echo a specific origin when credentials flow.

Security insight: CORS protects users from malicious sites, not servers from malicious clients — never rely on CORS as authorization. An attacker running a curl script ignores CORS entirely. Authenticate and authorize regardless of origin.

How do you set up a reverse proxy with Nginx for an ExpressJS application?

Terminating traffic at Nginx in front of Node is the classic production topology: Nginx handles TLS, static files, compression, and connection management; Node focuses on business logic. Done right, it yields order-of-magnitude better concurrency and operational flexibility.

    server 127.0.0.1:3000;
    server 127.0.0.1:3001;                # multiple Node instances

}

    listen 443 ssl http2;
    server_name app.example.com;

    ssl_protocols TLSv1.2 TLSv1.3;

    location /static/ {
        root /var/www/app;

Question 15

Explain how to create a middleware for API request throttling in ExpressJS.

Throttling and rate limiting are cousins but solve different problems: rate limiting enforces a quota over a window (100 req / 15 min); throttling caps instantaneous rate (30 req / sec, bursts of 50). Most serious APIs need both.

// custom throttle — token-bucket style per key (user or IP)
class TokenBucket {
  constructor(ratePerSec, burst) {
    this.rate = ratePerSec; this.capacity = burst;
    this.tokens = burst; this.lastRefill = Date.now();
  }
  take() {
    const now = Date.now();
    this.tokens = Math.min(this.capacity, this.tokens + (now - this.lastRefill) / 1000 * this.rate);
    this.lastRefill = now;
    if (this.tokens < 1) return false;
    this.tokens -= 1;
    return true;
  }
}

const buckets = new Map();

function throttle({ rate, burst, keyFn }) {
  return (req, res, next) => {
    const key = keyFn(req);
    let bucket = buckets.get(key);
    if (!bucket) { bucket = new TokenBucket(rate, burst); buckets.set(key, bucket); }

    if (!bucket.take()) {
      res.set("Retry-After", "1");
      return res.status(429).json({ error: "Throttled" });
    }
    next();
  };
}

app.use(throttle({
  rate:  30,                                 // tokens per second
  burst: 50,                                 // initial + max capacity
  keyFn: (req) => req.user?.id ?? req.ip,
}));

Describe how to use streams for handling large data in an ExpressJS application.

Node.js streams are the correct abstraction whenever data volume exceeds memory or time-to-first-byte matters. They process data in chunks with automatic backpressure: when the consumer can't keep up, the producer pauses. Express supports streams at both request and response sides.

// Response streaming — CSV export of millions of rows
import { pipeline } from "node:stream/promises";
import { format } from "@fast-csv/format";

app.get("/export.csv", async (req, res, next) => {
  try {
    res.setHeader("Content-Type", "text/csv");
    res.setHeader("Content-Disposition", `attachment; filename="export.csv"`);

    const csv = format({ headers: true });
    csv.pipe(res);                           // backpressure flows end-to-end

    // cursor-paginate DB and write rows as they arrive
    let lastId = null;
    while (true) {
      const batch = await db.order.findMany({
        where: lastId ? { id: { gt: lastId } } : undefined,
        orderBy: { id: "asc" }, take: 500,
      });
      if (batch.length === 0) break;
      for (const row of batch) csv.write(row);   // writes respect drain events
      lastId = batch[batch.length - 1].id;
    }
    csv.end();
  } catch (err) { next(err); }
});

// Request streaming — parse large uploads without buffering
import busboy from "busboy";
app.post("/upload-stream", (req, res) => {
  const bb = busboy({ headers: req.headers, limits: { fileSize: 500 * 1024 * 1024 } });
  bb.on("file", (name, file, info) => {
    pipeline(file, s3UploadStream(info.filename))
      .catch((err) => req.log?.error({ err }));
  });
  bb.on("close", () => res.json({ ok: true }));
  req.pipe(bb);
});

// Transform stream — on-the-fly processing
import { Transform } from "node:stream";
const jsonLines = new Transform({
  writableObjectMode: true,
  transform(obj, enc, cb) { cb(null, JSON.stringify(obj) + "\n"); },
});

Why streams matter:

  • Constant memory. A 10 GB export uses < 100 KB of RAM; fs.readFile + res.send OOMs.
  • Progressive delivery. Clients start receiving bytes before the whole response is generated — faster TTFB, better UX.
  • Backpressure. When the socket buffer fills, the DB read pauses; when the client drains, it resumes. Without this, a slow client crashes a fast server.

Common mistakes: forgetting error handlers on intermediate streams (unhandled errors crash the process); mixing sync write() returns with manual flow control (use pipeline() from node:stream/promises — it handles errors and cleanup); compressing streams with compression middleware breaks SSE/chunked endpoints (disable per-route). For HTTP/2 server push or HTTP/3, use Node 22+ plus a runtime like Bun or Deno for native support; Express on Node uses HTTP/1.1 by default.

Question 23

How do you create a custom error handling middleware in ExpressJS?

A production-grade error middleware does five things: accept any thrown error, normalize it into a shape your clients understand, log with enough context to debug, map status codes sensibly, and respond with a format matching the client's Accept header.

class HttpError extends Error {
  constructor(message, status = 500, code, details) {
    super(message);
    this.name = "HttpError";
    this.status = status;
    this.code = code;
    this.details = details;
  }
}

// async wrapper — lifts rejections into next()
const ah = (fn) => (req, res, next) =>
  Promise.resolve(fn(req, res, next)).catch(next);

// normalize library-specific errors into HttpError
function normalize(err) {
  if (err.name === "ZodError")
    return new HttpError("Invalid input", 400, "VALIDATION", err.issues);
  if (err.name === "TokenExpiredError")
    return new HttpError("Session expired", 401, "AUTH_EXPIRED");
  if (err.code === "P2002")
    return new HttpError("Already exists", 409, "CONFLICT", err.meta);
  if (err.code === "P2025")
    return new HttpError("Not found", 404, "NOT_FOUND");
  if (err.code === "ECONNREFUSED")
    return new HttpError("Upstream unavailable", 503, "UPSTREAM_DOWN");
  return err;
}

// the one error middleware — must come last, must have 4 args
app.use((err, req, res, _next) => {
  const norm = err.status ? err : normalize(err);
  const status = norm.status || 500;
  const isServerError = status >= 500;

  // log with severity and full error for server errors; less for client errors
  const logFn = isServerError ? req.log?.error : req.log?.warn;
  logFn?.({ err: norm, reqId: req.id, userId: req.user?.id }, norm.message);

  // if response already started streaming, we can't change status — just close
  if (res.headersSent) return res.destroy();

  // content-negotiate the response
  const body = {
    error: isServerError ? "Internal Server Error" : norm.message,
    code:  norm.code,
    reqId: req.id,
    ...(norm.details && !isServerError && { details: norm.details }),
    ...(process.env.NODE_ENV !== "production" && { stack: norm.stack }),
  };

  res.status(status).format({
    "application/json": () => res.json(body),
    "text/html":        () => res.render("error", { body, status }),
    default:            () => res.type("text/plain").send(body.error),
  });
});

Design principles:

  • One error middleware. Central normalization beats scattered try/catch with bespoke responses.
  • Include a request id so users can paste it into support tickets and you can instantly correlate logs.
  • Never leak internals to clients: on 5xx, return a generic message. Stack traces in production responses are an information-disclosure bug.
  • Handle headersSent: if the response already started streaming (SSE, long JSON), you can't change the status — destroy the socket and log.
  • Error classes carry HTTP semantics: HttpError(message, status, code) lets handlers throw freely without knowing about Express internals.
  • Alerting — stream error-level logs to Sentry / Datadog / Rollbar. Set alert thresholds by error code so a spike in VALIDATION doesn't page when the DB goes down.

Question 25

How can you implement a request/response transformation middleware in ExpressJS?

Request/response transformation middleware lets you intercept and modify data in both directions. Common uses: request — normalize headers, unwrap payloads, strip PII; response — reshape JSON envelopes, add pagination metadata, wrap errors.

// Request-side — normalize incoming payload
function unwrapPayload(req, res, next) {
  // accept both {data: {...}} and {...} at the top level
  if (req.body && typeof req.body === "object" && "data" in req.body) {
    req.body = req.body.data;
  }
  next();
}

// Response-side — wrap all JSON responses in an envelope
function envelopeResponses(req, res, next) {
  const origJson = res.json.bind(res);
  res.json = (body) => {
    const status = res.statusCode;
    const wrapped = status >= 400
      ? { ok: false, error: body, meta: { reqId: req.id } }
      : { ok: true,  data: body,  meta: { reqId: req.id, timestamp: Date.now() } };
    return origJson(wrapped);
  };
  next();
}

app.use(express.json(), unwrapPayload, envelopeResponses);

// Streaming transformation — gzip is the obvious one; on-the-fly filtering is another
import { Transform } from "node:stream";
function filterFields(...fieldsToRemove) {
  return new Transform({
    writableObjectMode: true,
    transform(row, enc, cb) {
      const clean = { ...row };
      for (const f of fieldsToRemove) delete clean[f];
      cb(null, JSON.stringify(clean) + "\n");
    },
  });
}

The monkey-patch tradeoff: overriding res.json (shown above) works but couples transformation to Express internals. Safer alternatives:

  • Helper function — write respond(res, status, body) and use it everywhere; no monkey-patching.
  • Response interceptor libraries (express-response-helper, or build your own) — provide a cleaner API.
  • Buffer-then-transform for non-streaming responses — capture res.write/res.end chunks and modify before sending.

When not to transform: response transformation breaks the direct relationship between handler code and wire output — a future teammate wonders why res.json({users: [...]}) returns {ok: true, data: {users: [...]}}. Document the envelope at the API boundary. For REST APIs, don't reshape every response — pagination metadata belongs in headers (Link, X-Total-Count) or response body by convention, not in a wrapping envelope that forces clients to unwrap everything.

Observability angle: request/response middleware is the natural place to instrument metrics and traces — measure duration, count by status code, sample bodies for debugging. OpenTelemetry's Express instrumentation already does this; if you have it installed, don't duplicate.

Question 26

Describe how to use the compression middleware to improve performance in ExpressJS.

The compression middleware inspects the client's Accept-Encoding header and negotiates gzip, deflate, or (with extra packages) brotli on the response body. For typical JSON APIs, payloads shrink 70-90%.

import compression from "compression";

app.use(compression({
  level: 6,                                     // 1 (fastest) — 9 (smallest); 6 is balanced
  threshold: 1024,                              // skip bodies < 1 KB
  filter: (req, res) => {
    if (req.headers["x-no-compression"]) return false;
    return compression.filter(req, res);        // default: compress by content-type
  },
}));

Trade-offs:

  • CPU vs bandwidth. Compression is CPU-bound. On a CPU-constrained Node instance serving thousands of req/sec, the cost may exceed the savings. Profile with and without.
  • Already-compressed formats (JPEG, MP4, PNG, .zip) get slightly larger with gzip — the default filter correctly skips them by MIME type.
  • Streaming + compression buffer output; SSE and long-poll endpoints should disable compression or call res.flush() explicitly so partial frames ship.
  • BREACH attack — gzip + user-controlled content + secrets in the same response leaks data via size analysis. Don't reflect secrets into responses, and use CSRF tokens per-request (not per-session).

Production guidance: offload compression to your reverse proxy (Nginx, Caddy, Cloudflare) — they do it in C, faster than Node's JS, and free Node's event loop for real work. Brotli (Cloudflare/Nginx support it natively) beats gzip 15-25% on text; combine with long Cache-Control on static assets for compounding wins. Measure first — if most traffic is mobile on slow links, compression is a clear win; if it's datacenter-to-datacenter, skip it.

Question 27

How do you implement a file upload system with Multer in ExpressJS?

Multer parses multipart/form-data, the only encoding that carries files over HTTP. Production uploads demand configuration beyond the defaults: strict limits, MIME filtering, magic-byte verification, and direct-to-storage streaming.

import multer from "multer";
import { fileTypeFromBuffer } from "file-type";

const upload = multer({
  storage: multer.memoryStorage(),            // or diskStorage / multer-s3
  limits: {
    fileSize: 10 * 1024 * 1024,               // per file
    files: 5,                                 // max files in one request
    fields: 20,                               // max non-file fields
  },
  fileFilter: (req, file, cb) => {
    const allowed = new Set(["image/jpeg","image/png","image/webp","application/pdf"]);
    cb(null, allowed.has(file.mimetype));
  },
});

app.post("/upload", upload.single("file"), async (req, res, next) => {
  if (!req.file) return res.status(400).json({ error: "No file" });
  // magic-byte verification — the client-reported MIME is unreliable
  const detected = await fileTypeFromBuffer(req.file.buffer);
  if (!detected || detected.mime !== req.file.mimetype) {
    return res.status(400).json({ error: "Type mismatch" });
  }
  // then: re-encode via sharp, virus scan, move to storage...
  res.status(201).json({ ok: true });
});

Three storage strategies:

  • Memory — fast access for processing (image resize, validation). RAM-bounded; unsuitable for large files.
  • Disk — files stream to local disk; convenient for development. Doesn't survive container restarts; needs separate garbage collection.
  • Direct-to-S3 (multer-s3) or pre-signed URL — production default; zero Node bandwidth for the upload itself.

Security essentials: never trust the client MIME type; always verify with file-type or re-encode with a real parser (sharp for images). Generate server-side filenames (UUID) — never use user-supplied names as paths. Serve user content from a different origin (user-content.example.com) so any XSS is isolated. Run uploads through ClamAV or a cloud scanning service for documents.

Question 28

Explain how to handle multipart/form-data requests in ExpressJS.

multipart/form-data is the only HTTP body format that interleaves text fields and files in one request. Neither express.json() nor express.urlencoded() can parse it — they consume the raw body and leave you with nothing usable. Multer (or busboy directly for streaming cases) is required.

import multer from "multer";

const upload = multer({ storage: multer.memoryStorage() });

// mixed: one avatar + multiple screenshots + text fields
app.post("/report",
  upload.fields([
    { name: "avatar",      maxCount: 1 },
    { name: "screenshots", maxCount: 5 },
  ]),
  (req, res) => {
    // text fields (title, body) on req.body
    // files on req.files.avatar[], req.files.screenshots[]
    res.json({
      title: req.body.title,
      avatar: req.files.avatar?.[0]?.originalname,
      screenshots: req.files.screenshots?.map((f) => f.originalname) ?? [],
    });
  }
);

Common pitfalls:

  • Body parsers conflict. Don't register express.json() before multer on the same route — it consumes the stream and multer sees an empty body.
  • Field order matters on the wire. Multipart parses sequentially. Put text fields before file fields in the form, so your fileFilter can access req.body values (e.g. req.body.userId) for per-user limits.
  • Streaming is better for huge files. For files > 100 MB, use busboy directly with a readable stream → S3 multipart upload, rather than buffering or writing to disk.

When to skip Express entirely: for very large or many concurrent uploads, pre-signed URLs bypass Node's event loop — the browser POSTs directly to S3/GCS/R2 using a time-limited signed URL your server issues. Your Node process handles the tiny "sign" request; the multi-GB bytes never touch it. For resumable uploads on flaky networks, tus (tus-js-client + tusd server or @tus/server middleware) is the standard.

Question 29

How do you implement WebSocket communication in an ExpressJS application?

Express doesn't speak WebSocket — the protocol starts with an HTTP Upgrade handshake that Express routes don't intercept. The standard pattern is to attach a WebSocket server to the same underlying http.Server Express runs on, so both share a port.

import express from "express";
import http from "node:http";
import { WebSocketServer } from "ws";

const app = express();
app.get("/health", (req, res) => res.send("ok"));

const server = http.createServer(app);
const wss = new WebSocketServer({ noServer: true });   // attach via manual upgrade

// authenticate on the upgrade handshake
server.on("upgrade", async (req, socket, head) => {
  try {
    const token = new URL(req.url, "http://x").searchParams.get("token");
    req.user = await verifyJwt(token);
  } catch {
    socket.write("HTTP/1.1 401 Unauthorized\r\n\r\n");
    socket.destroy();
    return;
  }
  wss.handleUpgrade(req, socket, head, (ws) => wss.emit("connection", ws, req));
});

wss.on("connection", (ws, req) => {
  ws.send(JSON.stringify({ type: "welcome", userId: req.user.id }));
  ws.on("message", (buf) => { /* handle */ });
});

server.listen(3000);                                   // HTTP and WS on :3000

Production essentials:

  • Authenticate on upgrade, before emitting the connection event. Anyone who reaches the handler has full access; reject at the handshake.
  • Heartbeats — ping/pong every 30s; terminate sockets that don't respond. NAT/proxies silently drop idle connections otherwise.
  • Backpressure — check ws.bufferedAmount before sending; if a slow consumer is buffering, drop the socket or throttle the producer.
  • Reverse proxy config — Nginx and ALBs need WebSocket support explicitly enabled (proxy_set_header Upgrade).

Scale-out: with multiple Node instances, a client on instance A can't send to a client on instance B. Use Redis pub/sub as a fan-out layer, or delegate to Socket.IO + @socket.io/redis-adapter which handles this out of the box. For high-scale real-time (100k+ connections per process), uWebSockets.js is an order of magnitude more efficient than ws. For hosted pub/sub, Ably, Pusher, and Supabase Realtime remove the ops burden.

Socket.IO is a higher-level protocol on top of WebSockets that adds rooms, automatic reconnection, namespaces, message acknowledgments, and HTTP long-polling fallback. It integrates with Express by attaching to the same HTTP server.

How can you use worker threads for CPU-intensive tasks in ExpressJS?

Node's event loop runs JavaScript on a single thread. CPU-heavy work (encryption, image processing, compression of large buffers, complex regex) blocks the loop, freezing every request. Worker threads (node:worker_threads, since Node 10.5) run JS on real OS threads — the correct escape hatch.

// worker.js — lives in its own file
import { parentPort, workerData } from "node:worker_threads";
import bcrypt from "bcrypt";

parentPort.on("message", async ({ id, password }) => {
  const hash = await bcrypt.hash(password, 12);
  parentPort.postMessage({ id, hash });
});

// main.js — maintain a small pool of workers
import { Worker } from "node:worker_threads";
import { availableParallelism } from "node:os";

class WorkerPool {
  constructor(size = availableParallelism() - 1) {
    this.workers = Array.from({ length: size }, () => new Worker("./worker.js"));
    this.idx = 0;
    this.pending = new Map();
    for (const w of this.workers) {
      w.on("message", ({ id, ...r }) => {
        this.pending.get(id)?.(r);
        this.pending.delete(id);
      });
    }
  }
  run(data) {
    return new Promise((resolve) => {
      const id = crypto.randomUUID();
      this.pending.set(id, resolve);
      this.workers[this.idx].postMessage({ id, ...data });
      this.idx = (this.idx + 1) % this.workers.length;
    });
  }
}

const pool = new WorkerPool();

app.post("/signup", async (req, res) => {
  const { hash } = await pool.run({ password: req.body.password });
  // ... save user with hash ...
  res.json({ ok: true });
});

Question 36

How do you handle graceful shutdown and cleanup in an ExpressJS application?

Graceful shutdown is: stop accepting new connections, finish in-flight requests, close resources (DB, Redis, queues), then exit. In container environments (Kubernetes, ECS), the runtime sends SIGTERM and gives you a grace window (default 30s in Kubernetes) before SIGKILL. Your job: use that window correctly.

import http from "node:http";

const server = http.createServer(app);
server.listen(PORT);

let shuttingDown = false;
app.get("/ready", (req, res) => {
  res.status(shuttingDown ? 503 : 200).send(shuttingDown ? "draining" : "ready");
});

async function shutdown(signal) {
  if (shuttingDown) return;
  shuttingDown = true;
  logger.info({ signal }, "shutdown initiated");

  // 1. fail readiness immediately → load balancer drains traffic
  //    (no code needed — handler above reads the flag)

  // 2. wait a beat so LB has time to remove us from rotation
  await new Promise((r) => setTimeout(r, 5_000));

  // 3. close the HTTP server (stops accepting new connections)
  await new Promise((resolve) => {
    server.close((err) => { if (err) logger.error(err); resolve(); });
    // Node 18.2+: force-close sockets that won't drain
    setTimeout(() => server.closeAllConnections?.(), 15_000).unref();
  });

  // 4. close dependencies
  await db.$disconnect();
  await redis.quit();
  await queue.close();

  logger.info("shutdown complete");
  process.exit(0);
}

process.on("SIGTERM", () => shutdown("SIGTERM"));
process.on("SIGINT",  () => shutdown("SIGINT"));

// never masquerade an unknown-state process as healthy
process.on("uncaughtException",  (err) => { logger.fatal(err); process.exit(1); });
process.on("unhandledRejection", (err) => { logger.fatal(err); process.exit(1); });

Non-obvious details:

  • Readiness fails first, server closes second. If you close the listener before fail-ready, the LB still sends requests that get refused (connection reset) during the removal window.
  • Keep-alive connections linger. server.close() waits for them to drain; closeAllConnections() (Node 18.2+) force-terminates after a timeout.
  • WebSocket/SSE connections block shutdown; send a reconnect signal and close them explicitly.
  • Background jobs — let BullMQ/inngest finish the active job; don't grab new ones while draining.

Libraries that handle edge cases: http-terminator and stoppable correctly handle keep-alive drain; terminus wraps the whole graceful-shutdown pattern with health-check integration. Always test shutdown locally with kill -TERM <pid> — the bug-finding rate is high.

let shuttingDown = false;
let ready = false;

// readiness — check dependencies, report degraded components
app.get("/ready", async (req, res) => {
  if (shuttingDown) return res.status(503).json({ status: "draining" });

  let healthy = true;
  await Promise.all([

    timed("redis", () => redis.ping()),

  const start = Date.now();
  return fn()
    .then(()  => ({ name, ms: Date.now() - start }))
    .catch((e) => Promise.reject({ name, err: e.message }));
}

Question 39

Describe how to use middleware for conditional request handling in ExpressJS.

Conditional middleware runs only when a request meets certain criteria — feature flags, route patterns, content type, user role, environment. Done right, it keeps common logic declarative; done wrong, it creates a maze where reasoning about order and coverage breaks down.

// 1. helper: conditional wrapper — runs middleware only when predicate passes
function when(predicate, mw) {
  return (req, res, next) => (predicate(req) ? mw(req, res, next) : next());
}

// 2. apply only in production
app.use(when(() => process.env.NODE_ENV === "production", helmet()));

// 3. apply only for authenticated users
app.use(when((req) => !!req.user, trackUsage));

// 4. apply only behind a feature flag
app.use(when(async (req) => await flags.on("newValidator", req.user?.id),
  newValidator));

// 5. apply only to API routes, not static ones
app.use(when((req) => req.path.startsWith("/api"), cors()));

// 6. apply only to methods with bodies
app.use(when((req) => ["POST","PUT","PATCH"].includes(req.method),
  express.json()));

// 7. per-route mounting is cleaner when the condition is a path
app.use("/admin", requireRole("admin"), adminRouter);

Guidance on conditions:

  • Prefer path-based mounting (app.use("/admin", mw)) over when predicates on path — Express's native routing is clearer.
  • Keep predicates pure and synchronous when possible. Async predicates work but add latency to every request.
  • Feature-flag middleware is an anti-pattern if the flag changes middleware ordering — two requests running under different flags behave differently at the framework level. Put flags inside the middleware.
  • Environment gates are fine but risky — "it works in dev, fails in prod" often traces here. Prefer the same middleware in both environments with different config.

Common use cases: dev-only request logging verbose mode, production-only security headers, conditional body parsers by content type, skipping auth on health checks and OPTIONS preflight, A/B testing new middleware with a flag before rolling it out. For complex predicates, CASL or Casbin policies express "who can do what when" more readably than stacked when() calls.

How do you handle API versioning in an ExpressJS application?

  • Document with OpenAPI — generate client SDKs from the spec. Swagger UI or Redoc for human-readable docs; openapi-typescript for typed clients.

Practical default for most teams: URL-path versioning (/api/v1) — it's boring, which is exactly what you want for versioning. Header-based is defensible but adds ceremony with no payoff for small consumers.

Question 41

Explain how to create a middleware for logging request and response data in ExpressJS.

Logging requests and responses together closes the observability loop: you see what came in and what went out, down to sizes and timing, with enough correlation data to trace through a distributed system. The pattern is one middleware that records on res.on("finish").

import pino from "pino";
import { AsyncLocalStorage } from "node:async_hooks";

const logger = pino({ redact: ["req.headers.authorization","req.headers.cookie"] });
const als = new AsyncLocalStorage();

function requestLogger(req, res, next) {
  const reqId = req.headers["x-request-id"] || crypto.randomUUID();
  req.id = reqId;
  res.setHeader("X-Request-Id", reqId);

  const start = process.hrtime.bigint();
  const reqBodySize = Number(req.headers["content-length"]) || 0;

  // capture response body size by wrapping res.write/res.end
  let resSize = 0;
  const origWrite = res.write.bind(res);
  const origEnd   = res.end.bind(res);
  res.write = (chunk, ...rest) => { if (chunk) resSize += Buffer.byteLength(chunk); return origWrite(chunk, ...rest); };
  res.end   = (chunk, ...rest) => { if (chunk) resSize += Buffer.byteLength(chunk); return origEnd(chunk, ...rest); };

  res.on("finish", () => {
    const durMs = Number(process.hrtime.bigint() - start) / 1e6;
    const level = res.statusCode >= 500 ? "error" : res.statusCode >= 400 ? "warn" : "info";
    logger[level]({
      reqId,
      req:  { method: req.method, url: req.originalUrl, bytes: reqBodySize,
              ua: req.get("user-agent"), ip: req.ip },
      res:  { status: res.statusCode, bytes: resSize },
      user: req.user?.id,
      durMs: +durMs.toFixed(2),
    });
  });

  als.run({ reqId, userId: req.user?.id }, next);
}

app.use(requestLogger);

What you should NOT log:

  • Request bodies wholesale. They contain passwords, tokens, PII, payment data. Log sizes and keys, not values.
  • Response bodies. Same reason, plus log-storage cost balloons fast.
  • Authorization, Cookie, Set-Cookie headers. Redact declaratively (pino's redact option) — one missed log line is a credential leak.
  • Query strings unredacted. ?token=xxx and ?email=xxx end up in logs.

Advanced: sample high-traffic endpoints (1 in 10-100 for /health, 1:1 for errors); emit logs as JSON on stdout in containers — the orchestrator ships them; use OpenTelemetry to propagate trace ids via traceparent headers so one request across many services threads together.

How do you implement a RESTful API with ExpressJS and MongoDB?

Building a RESTful API with Express and MongoDB is three layers stacked: routing (Express Router per resource), validation (Zod or similar at the edge), and data access (Mongoose schemas or the native driver). The architecture decisions that matter: how thick is each layer, where does error mapping happen, and how do you enforce ownership across endpoints.

}, { versionKey: false });

import { Product } from "../models/product.js";

    const filter = cursor ? { _id: { $lt: cursor } } : {};
    const docs = await Product.find(filter)
      .sort({ _id: -1 })
      .limit(Math.min(100, Number(limit)))
      .lean();                                   // plain JS objects — faster
    res.json({ data: docs, next: docs[docs.length-1]?._id });
  } catch (err) { next(err); }
});

  try {
    if (!mongoose.isValidObjectId(req.params.id)) return res.sendStatus(404);

});

  • Indexes on query fields. Without an index on userId, every per-user query scans the whole collection.

  • Validate ObjectId before querying — malformed ids throw otherwise.

Prisma (with MongoDB adapter, since 2023) brings TypeScript inference and migrations but lags Mongoose in advanced features. Pick Mongoose for rich Mongo apps, Prisma for type-safety-first teams, native driver for performance-critical services.

Describe the process of using Mongoose for schema validation in an ExpressJS application.

Mongoose schemas are the type system and validation layer for MongoDB documents in Node. Beyond the basics, mature schemas lean on middleware (hooks), custom validators, virtuals, and indexes declared on the schema so they're automatic across environments.

import mongoose from "mongoose";
const { Schema } = mongoose;

const UserSchema = new Schema({
  email: {
    type: String,
    required: [true, "email is required"],
    unique: true,                                  // creates a unique index
    lowercase: true,
    trim: true,
    validate: {
      validator: (v) => /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(v),
      message: "invalid email format",
    },
  },
  passwordHash: { type: String, required: true, select: false },  // never returned by default
  age:   { type: Number, min: 13, max: 120 },
  roles: { type: [String], enum: ["user","editor","admin"], default: ["user"] },
  profile: {
    name:      { type: String, maxlength: 100 },
    websites:  { type: [String], validate: [(arr) => arr.length <= 5, "max 5 websites"] },
  },
}, {
  timestamps: true,              // adds createdAt / updatedAt automatically
  toJSON: { virtuals: true, transform: (doc, ret) => { delete ret.__v; return ret; } },
});

// compound index for query patterns — e.g. list "editors created recently"
UserSchema.index({ roles: 1, createdAt: -1 });

// virtual — computed field, not stored
UserSchema.virtual("displayName").get(function () { return this.profile.name || this.email; });

// pre-save hook — hash the password before persist
UserSchema.pre("save", async function (next) {
  if (!this.isModified("passwordHash")) return next();
  this.passwordHash = await bcrypt.hash(this.passwordHash, 12);
  next();
});

// instance method — encapsulate domain logic
UserSchema.methods.verifyPassword = function (plain) {
  return bcrypt.compare(plain, this.passwordHash);
};

export const User = mongoose.model("User", UserSchema);

Where schema validation shines: schema rules run on every save(), create(), and (with runValidators: true) update(). You get a single source of truth — route handlers don't duplicate validation logic. Combine with Zod at the HTTP edge (client-friendly error messages, type inference) for defense in depth.

Pitfalls: unique: true doesn't validate per-request — it creates a DB index and relies on the duplicate-key error (handle code: 11000). findOneAndUpdate skips middleware unless you opt in. Subdocuments have their own validation scope. For TypeScript, use Mongoose's generic types (Model<IUser>) or switch to Typegoose/@typegoose/typegoose for class-based schemas with decorators.

  await tx.transfer.create({ data: { fromId, toId, amount } });
}, { isolationLevel: "Serializable" });               // highest level — retry on conflict

const client = await pool.connect();

  await client.query("COMMIT");
} catch (err) {
  await client.query("ROLLBACK");
  throw err;
} finally {
  client.release();
}
// MongoDB — requires replica set
const session = await mongoose.startSession();
session.startTransaction();
try {
  await Account.updateOne({ _id: fromId }, { $inc: { balance: -amount } }).session(session);
  await Account.updateOne({ _id: toId },   { $inc: { balance:  amount } }).session(session);
  await session.commitTransaction();

  await session.abortTransaction();
  throw err;
} finally {
  session.endSession();

Essential concepts:

  • Isolation levels. Default is usually Read Committed. For money/inventory use Serializable and handle retries on serialization failures.
  • Lock vs optimistic concurrency. SELECT ... FOR UPDATE serializes access but can deadlock. Optimistic (version column, retry on conflict) scales better if conflicts are rare.
  • Keep transactions short. Each lock blocks other writers. Never run HTTP calls or long computations inside a transaction.
  • Retries. Serializable transactions can fail with serialization errors — wrap in a retry loop with jitter.

Distributed transactions (across services) are usually a bad idea — two-phase commit has painful failure modes. Use sagas (compensating actions) with a durable workflow engine (Temporal, Inngest) or transactional outbox patterns instead.

How do you handle user registration and login with email verification in ExpressJS?

A production signup + login flow with email verification needs five coordinated pieces: secure password storage, email verification tokens, constant-time login (to prevent user enumeration), rate limiting, and session issuance. Skipping any one creates a classic vulnerability.

import bcrypt from "bcrypt";
import crypto from "node:crypto";

app.post("/auth/register", registerLimiter, async (req, res, next) => {
  try {
    const data = SignupSchema.parse(req.body);     // Zod validation

// verification

  await db.$transaction([
    db.user.update({ where: { email: rec.email }, data: { emailVerified: true } }),

// login — constant-time even for unknown users

  const { email, password } = req.body;

        where: { userId: user.id, usedAt: null }, data: { usedAt: new Date() },

  const { userId, token, newPassword } = req.body;

  const hash = crypto.createHash("sha256").update(token).digest("hex");
  const rec = await db.passwordResetToken.findFirst({
    where: { userId, tokenHash: hash, usedAt: null, expiresAt: { gt: new Date() } },
  });
  if (!rec) return res.status(400).json({ error: "Invalid or expired token" });

  const passwordHash = await bcrypt.hash(newPassword, 12);

  await db.$transaction([
    db.user.update({ where: { id: userId }, data: { passwordHash } }),
    db.passwordResetToken.update({ where: { id: rec.id }, data: { usedAt: new Date() } }),
    // invalidate all existing sessions — key step
    db.session.deleteMany({ where: { userId } }),
    db.refreshToken.updateMany({ where: { userId }, data: { revokedAt: new Date() } }),
  ]);

  await sendPasswordChangedNotification(userId);     // alert the user from a known-good email
  res.json({ success: true });
});

Security principles:

  • Hash tokens in storage. Email carries the raw token; DB stores only the hash — a DB leak doesn't give away valid reset links.
  • Invariant response for "email not found" — never signal which emails exist.
  • Short lifetime (15-60 min), single-use, one active token per user (invalidate older ones on new request).
  • Rate limit aggressively — resetLimiter caps requests per IP and per email.
  • Revoke all sessions on successful reset — otherwise an attacker with an existing session survives the reset.
  • Send a confirmation email to the user after the change — "your password was changed on X; if this wasn't you..." — gives a path to recover from compromised accounts.
  • Never auto-login after reset — force a fresh login so the user proves ownership.

Beyond passwords: passkeys (WebAuthn) sidestep reset entirely — users authenticate with device-bound cryptographic keys. For existing password systems, better-auth and Auth.js implement all the above correctly, saving you from reinventing a fiddly security-critical flow.

}

Custom rate limiting is worth writing only when the express-rate-limit defaults don't fit — e.g. you need per-endpoint policies, differential limits per plan tier, or sliding-window + token-bucket semantics. The mechanism is always: pick a key, increment atomically, check threshold, set response headers.

// token-bucket limiter backed by Redis — allows bursts, enforces sustained rate
function tokenBucketLimiter({ capacity, refillPerSec, key }) {
  const lua = `
    local bucket = KEYS[1]
    local now        = tonumber(ARGV[1])
    local capacity   = tonumber(ARGV[2])
    local refillRate = tonumber(ARGV[3])
    local data = redis.call("HMGET", bucket, "tokens", "ts")
    local tokens = tonumber(data[1]) or capacity
    local ts     = tonumber(data[2]) or now
    tokens = math.min(capacity, tokens + (now - ts) * refillRate)
    if tokens < 1 then
      redis.call("HMSET", bucket, "tokens", tokens, "ts", now)
      redis.call("EXPIRE", bucket, 3600)
      return {0, tokens}
    end
    tokens = tokens - 1
    redis.call("HMSET", bucket, "tokens", tokens, "ts", now)
    redis.call("EXPIRE", bucket, 3600)
    return {1, tokens}
  `;
  return async (req, res, next) => {
    const k = `rl:${key(req)}`;
    const [ok, remaining] = await redis.eval(lua, 1, k, Date.now() / 1000, capacity, refillPerSec);
    res.setHeader("X-RateLimit-Remaining", remaining);
    if (!ok) {
      res.setHeader("Retry-After", Math.ceil(1 / refillPerSec));
      return res.status(429).json({ error: "Rate limit exceeded" });
    }
    next();
  };
}

app.use(tokenBucketLimiter({
  capacity: 60, refillPerSec: 10,
  key: (req) => req.user?.id ?? req.ip,
}));

Why Lua: the check-and-update must be atomic — a naive GET → update → SET races under load. Redis runs each Lua script atomically. Alternatives: rate-limiter-flexible ships token-bucket/leaky-bucket/fixed/sliding-window out of the box with a cleaner API. For managed infra, use Cloudflare Rate Limiting at the edge — protects Node from traffic surges entirely.

Explain how to use the body-parser middleware for handling JSON and URL-encoded data in ExpressJS.

// base — verifies JWT, hydrates req.user
export function authenticate({ optional = false } = {}) {
  return async (req, res, next) => {
    const [scheme, token] = (req.headers.authorization || "").split(" ");
    if (scheme !== "Bearer" || !token) {
      return optional ? next() : res.sendStatus(401);
    }
    try {
      const payload = jwt.verify(token, ACCESS_SECRET, {
        issuer: "api.example.com", audience: "api",
      });
      // check revocation
      if (await redis.get(`jwt:blocklist:${payload.jti}`)) {
        return res.status(401).json({ error: "Token revoked" });
      }
      // hydrate full user for downstream handlers
      req.user = await db.user.findUnique({
        where: { id: payload.sub },
        select: { id: true, role: true, email: true, status: true },
      });
      if (!req.user || req.user.status === "suspended") {
        return res.status(403).json({ error: "Account unavailable" });
      }
      next();
    } catch (err) {
      const msg = err.name === "TokenExpiredError" ? "Token expired" : "Invalid token";
      res.status(401).json({ error: msg });
    }
  };
}

Question 62

How do you handle conditional GET requests using ETag and Last-Modified headers in ExpressJS?

Conditional GET is HTTP's native cache validation: the client asks "do you still have the same resource I saw last time?" via If-None-Match (ETag) or If-Modified-Since (Last-Modified). If nothing changed, the server returns 304 Not Modified with no body — saving bandwidth and CPU.

// manual ETag — compute a strong hash of the content
import crypto from "node:crypto";

app.get("/posts/:id", async (req, res) => {
  const post = await db.post.findUnique({ where: { id: req.params.id } });
  if (!post) return res.sendStatus(404);

  const body = JSON.stringify(post);
  const etag = `"${crypto.createHash("sha1").update(body).digest("base64")}"`;

  res.setHeader("ETag", etag);
  res.setHeader("Last-Modified", post.updatedAt.toUTCString());
  res.setHeader("Cache-Control", "public, max-age=0, must-revalidate");

  // compare against client's copy
  if (req.headers["if-none-match"] === etag) return res.sendStatus(304);
  if (req.headers["if-modified-since"] &&
      new Date(req.headers["if-modified-since"]) >= post.updatedAt) {
    return res.sendStatus(304);
  }
  res.type("application/json").send(body);
});

Express built-in behavior: app.set("etag", "strong") enables automatic ETag on res.send()/res.json(). The default is "weak" which is fine for most content. Express checks If-None-Match automatically when using res.send.

ETagLast-Modified
Hash of content — preciseTimestamp — 1-second granularity
Works for generated contentBest for static files
Costs a hash per responseCheap if you already have updatedAt

Combined with Cache-Control: max-age=60, stale-while-revalidate=300 lets the browser serve cached content for 60s with no round-trip, revalidate in the background, and fall back to stale during errors. This is the modern stack — ETag for correctness, Cache-Control for performance.

Question 63

Describe the process of implementing a content negotiation middleware in ExpressJS.

A content negotiation middleware centralizes response format selection so handlers don't repeat res.format() calls everywhere. The pattern: inspect Accept once, stash the chosen format on req, then handlers use a helper that serializes accordingly.

const SUPPORTED = ["application/json","application/xml","text/csv","text/html"];

app.use((req, res, next) => {
  req.preferredType = req.accepts(SUPPORTED) || "application/json";
  res.vary("Accept");                          // tell caches responses differ by Accept
  next();
});

// helper — handlers pass data, middleware decides format
app.use((req, res, next) => {
  res.respond = (data, status = 200) => {
    res.status(status);
    switch (req.preferredType) {
      case "application/json": return res.json(data);
      case "application/xml":  res.type("application/xml"); return res.send(toXml(data));
      case "text/csv":         res.type("text/csv");        return res.send(toCsv(data));
      case "text/html":        return res.render("resource", { data });
      default:                 return res.sendStatus(406);
    }
  };
  next();
});

app.get("/users/:id", async (req, res) => {
  const user = await db.user.findUnique({ where: { id: req.params.id } });
  if (!user) return res.sendStatus(404);
  res.respond(user);                            // serializes based on Accept
});

Why a middleware over per-handler res.format():

  • Consistency. One place decides what formats you support; routes can't drift.
  • Single source for Vary. Caches (CDN, reverse proxy) need this header to store variants — missing it causes wrong-format responses served from cache.
  • Testability. The helper is easy to mock in unit tests.

Alternatives to header-based negotiation: URL-based (/users/42.json vs /users/42.csv) is more explicit, cache-friendly, and easier to debug with curl. Reserve Accept-based negotiation for APIs that genuinely serve both browsers and machine clients from the same URL.

How do you handle large JSON payloads in an ExpressJS application?

"Large JSON" comes in two flavors that need different strategies: large single payloads (one 10MB request body) and large response streams (paginated dataset returned as JSON). The wrong approach OOMs your Node process; the right one uses constant memory.

10 MB request body (bulk import)Raise express.json({ limit: "20mb" }) only on the specific route; require auth; size-limit aggressively elsewhere.
100 MB request bodyReject JSON; use streaming NDJSON or CSV; or upload to S3 and process async.

// streaming NDJSON response — O(1) memory, arbitrary dataset size
app.get("/export/events.ndjson", async (req, res) => {
  res.setHeader("Content-Type", "application/x-ndjson");
  res.setHeader("Cache-Control", "no-store");

  let cursor = null;
  while (true) {
    const batch = await db.event.findMany({
      ...(cursor && { cursor: { id: cursor }, skip: 1 }),
      take: 500, orderBy: { id: "asc" },
    });
    if (batch.length === 0) break;
    for (const row of batch) {
      if (!res.write(JSON.stringify(row) + "\n")) {
        await new Promise((r) => res.once("drain", r));  // backpressure
      }
    }
    cursor = batch[batch.length - 1].id;
  }
  res.end();
});

Key techniques: watch res.write return value — when it's false, pause reading until drain fires (backpressure). For huge parsed input, use stream-json to parse incrementally rather than building one massive object. Consider MessagePack or Protocol Buffers for internal service-to-service traffic — 2-5x smaller, faster to parse. JSON is a terrible serialization format for bulk data; use it only when humans need to read it.

Question 65

Explain how to use the cors middleware to handle cross-origin requests in ExpressJS.

The cors middleware handles the two-phase CORS dance: preflight (OPTIONS with Access-Control-Request-Method headers) and the actual request (response includes Access-Control-Allow-Origin). One app.use(cors(options)) covers both.

import cors from "cors";

// function origin — full control over the allowlist
const corsOptions = {
  origin: (origin, cb) => {
    // non-browser clients (curl, server-to-server) have no Origin → allow
    if (!origin) return cb(null, true);
    const allowlist = ["https://app.example.com", "https://admin.example.com"];
    const matches = allowlist.includes(origin) || /\.internal\.example\.com$/.test(origin);
    cb(matches ? null : new Error(`CORS: ${origin} not allowed`), matches);
  },
  credentials:    true,
  methods:        ["GET","POST","PUT","PATCH","DELETE"],
  allowedHeaders: ["Content-Type","Authorization","X-Request-Id","Idempotency-Key"],
  exposedHeaders: ["X-Request-Id","X-RateLimit-Remaining"],   // expose custom resp headers to JS
  maxAge:         86400,                                       // cache preflight 24h
};

app.use(cors(corsOptions));
app.options("*", cors(corsOptions));                           // explicit preflight handler

Critical rules:

  • Register CORS before auth. Preflights don't carry auth headers — if auth comes first, they 401 and the browser aborts.
  • Never origin: "*" with credentials: true — browsers reject. Echo the origin from an allowlist instead.
  • Custom headers need allowedHeaders. X-Request-Id silently drops from preflights otherwise.
  • Custom response headers need exposedHeaders. Without it, fetch can't read them even though they're sent.
  • Cache preflights aggressively (maxAge: 86400) — otherwise every unique preflight doubles request counts.

For per-route policies (public docs at *, credentialed /api), mount cors(options) per path. Centralizing at the edge (Cloudflare Transform Rules) works but couples policy to infra — in-app CORS is easier to test and version-control.

import * as yup from "yup";

          error:  "Validation failed",
          issues: err.inner.map((e) => ({ field: e.path, message: e.message, type: e.type })),
        });

      next(err);
    }
  };
}

Nested routers model hierarchical resources (/users/:userId/posts/:postId/comments/:commentId) without exploding a single app.js. Express's Router is composable — routers can mount other routers.

Rules for healthy nesting:

  • Always use mergeParams: true on child routers — without it, child handlers can't see parent params.
  • Cap depth at 2-3 levels. /a/:x/b/:y/c/:z is a URL; deeper is a smell. Offer flat alternatives (/comments/:id) for direct access.

  • Param middleware (app.param("postId", loadPost)) runs once per request and attaches loaded resources to req, eliminating repeated DB lookups.

For large apps, consider a controller/service split where the router is pure routing and handlers delegate to services. Frameworks like NestJS enforce this structure; for vanilla Express, convention replaces code.

How do you implement a custom session store in ExpressJS?

A custom session store implements express-session's store interface — three methods: get(sid, cb), set(sid, session, cb), destroy(sid, cb). Optional but valuable: touch, length, all, clear. Building one manually is rare — the ecosystem has production-grade stores for every backend.

import session from "express-session";

class CustomStore extends session.Store {
  constructor(client) { super(); this.client = client; }

  async get(sid, cb) {
    try {
      const data = await this.client.get(`sess:${sid}`);
      cb(null, data ? JSON.parse(data) : null);
    } catch (err) { cb(err); }
  }

  async set(sid, sess, cb) {
    try {
      const ttl = sess.cookie?.maxAge ? Math.floor(sess.cookie.maxAge / 1000) : 86400;
      await this.client.setex(`sess:${sid}`, ttl, JSON.stringify(sess));
      cb(null);
    } catch (err) { cb(err); }
  }

  async destroy(sid, cb) {
    try { await this.client.del(`sess:${sid}`); cb(null); }
    catch (err) { cb(err); }
  }

  async touch(sid, sess, cb) {                              // reset TTL on each request
    try {
      const ttl = sess.cookie?.maxAge ? Math.floor(sess.cookie.maxAge / 1000) : 86400;
      await this.client.expire(`sess:${sid}`, ttl);
      cb(null);
    } catch (err) { cb(err); }
  }
}

How do you implement a middleware for handling file uploads with validation in ExpressJS?

const MAX_SIZE = 20 * 1024 * 1024;

const upload = multer({
  storage: multer.memoryStorage(),

  if (!req.file) return res.status(400).json({ error: "No file" });

  // 1. real-format detection — client-reported MIME can lie
  const detected = await fileTypeFromBuffer(req.file.buffer);
  if (!detected || !ALLOWED_MIMES.includes(detected.mime)) {
    return res.status(400).json({ error: `File content is ${detected?.mime ?? "unknown"}` });
  }

  // 2. business rules
  const userQuota = await getUserStorageUsed(req.user.sub);
  if (userQuota + req.file.size > QUOTA_BYTES) {
    return res.status(413).json({ error: "Storage quota exceeded" });
  }

  // 3. attach normalized info for handlers
  req.validatedFile = {
    ...req.file,
    detectedMime: detected.mime,
    detectedExt:  detected.ext,
  };
  next();
}

  async (req, res) => {

  }

Validation depth by file type:

  • Images: re-encode with sharp — rejects malformed files, strips EXIF, bounds dimensions (decompression-bomb defense).

  • Office docs: scan with ClamAV; convert to PDF for safer preview.
  • Archives: check compressed size before extracting — zip bombs are a thing (42KB → 4.5PB).

Never trust file.mimetype — it comes from the browser's filename extension. For production, scan with ClamAV or cloud malware services, re-encode images through a hardened library, serve from a separate domain, and set Content-Disposition: attachment on ambiguous types so browsers don't render them.

    });

    if (!text || text.length > 2000) return ack({ error: "Invalid" });

      data: { roomId, authorId: user.id, text, clientMsgId },

  });

  • Rate limit per socket (not per IP) to prevent a single user spamming.

// cache-aside with singleflight — prevents stampede when a hot key expires
const inFlight = new Map();

async function cached(key, ttlSec, loader) {
  const hit = await redis.get(key);
  if (hit) return hit === "NULL" ? null : JSON.parse(hit);

  // coalesce concurrent misses on the same key
  if (inFlight.has(key)) return inFlight.get(key);

  const p = (async () => {
    try {
      const fresh = await loader();
      // cache even null — prevents cache penetration DoS
      await redis.setex(key, ttlSec, fresh === null ? "NULL" : JSON.stringify(fresh));
      return fresh;
    } finally { inFlight.delete(key); }
  })();
  inFlight.set(key, p);
  return p;
}

    () => db.product.findUnique({ where: { id: req.params.id } }));
  if (!product) return res.sendStatus(404);
  res.set("Cache-Control", "public, max-age=30");
  res.json(product);

// invalidate on write

  const updated = await db.product.update({ where: { id: req.params.id }, data: req.body });
  await redis.del(`product:${req.params.id}`);
  res.json(updated);

How do you handle user authentication and authorization using JWT in ExpressJS?

JWT-based auth & authz splits cleanly into authentication (who you are — verified by JWT signature) and authorization (what you can do — role/permission checks on the verified identity). Middleware composition keeps these orthogonal.

// AUTHENTICATION — verify the token, hydrate req.user from DB
function authenticate() {
  return async (req, res, next) => {
    const [, token] = (req.headers.authorization || "").split(" ");
    if (!token) return res.sendStatus(401);
    try {
      const payload = jwt.verify(token, ACCESS_SECRET, {
        issuer: "api.example.com", audience: "api",
      });
      if (await redis.get(`jwt:blocklist:${payload.jti}`)) {
        return res.status(401).json({ error: "Token revoked" });
      }
      req.user = await db.user.findUnique({
        where: { id: payload.sub },
        select: { id: true, role: true, email: true, permissions: true, status: true },
      });
      if (!req.user || req.user.status === "suspended") return res.sendStatus(403);
      next();
    } catch (err) {
      const msg = err.name === "TokenExpiredError" ? "Token expired" : "Invalid token";
      res.status(401).json({ error: msg });
    }
  };
}

// AUTHORIZATION — role or permission checks
function requireRole(...roles) {
  return (req, res, next) =>
    roles.includes(req.user?.role) ? next() : res.sendStatus(403);
}
function requirePermission(perm) {
  return (req, res, next) =>
    req.user?.permissions?.includes(perm) || req.user?.role === "admin"
      ? next() : res.sendStatus(403);
}

// resource-scoped authz — "you can edit only your own post"
async function canEditPost(req, res, next) {
  const post = await db.post.findUnique({ where: { id: req.params.id } });
  if (!post) return res.sendStatus(404);
  if (post.authorId !== req.user.sub && req.user.role !== "admin") return res.sendStatus(403);
  req.post = post; next();
}

Token invalidation: short access TTL (15 min) + refresh rotation + Redis blocklist for immediate revocation. Question 88

How do you handle role-based access control (RBAC) in an ExpressJS application?

RBAC gives you three dials: roles (coarse buckets), permissions (fine-grained actions), and resource scope (who owns what). Simple apps need only roles; complex apps map roles to permissions and check permissions on each route; enterprise apps add ABAC (attribute-based) on top.

// role → permissions (loaded from DB or YAML at boot)
const ROLE_PERMS = {
  viewer: ["post:read"],
  editor: ["post:read", "post:create", "post:update:own"],
  moderator: ["post:read", "post:create", "post:update:*", "post:delete:*"],
  admin: ["*"],
};

function hasPermission(user, perm) {
  const perms = ROLE_PERMS[user.role] ?? [];
  if (perms.includes("*")) return true;
  return perms.some((p) => p === perm || p === perm.split(":").slice(0,-1).join(":") + ":*");
}

function requirePermission(perm) {
  return (req, res, next) =>
    hasPermission(req.user, perm) ? next() : res.sendStatus(403);
}

// resource-scoped "own" check — editor can update only their own posts
async function canUpdatePost(req, res, next) {
  const post = await db.post.findUnique({ where: { id: req.params.id } });
  if (!post) return res.sendStatus(404);
  if (hasPermission(req.user, "post:update:*")) { req.post = post; return next(); }
  if (hasPermission(req.user, "post:update:own") && post.authorId === req.user.sub) {
    req.post = post; return next();
  }
  return res.sendStatus(403);
}

app.patch("/posts/:id", authenticate(), canUpdatePost, updatePostHandler);

When to graduate beyond this:

  • Per-field permissions: editors can change title but not status. Use CASL: ability.can("update", post, "title").
  • Attribute-based rules: "users in org X can access org X's data"; "after business hours, only admins". Use Casbin with RBAC-with-domains or ABAC policies.
  • Centralized policy engine: microservices sharing authz rules → Open Policy Agent (OPA) with policies as Rego code.

Non-negotiables: always enforce server-side (client-side checks are UX, not security); audit-log every denial with user/action/resource (leading indicator of probing); start strict (deny by default) and add permissions as needed, not the reverse.

Question 91

Describe the process of creating a custom middleware for API request validation in ExpressJS.

A custom API request validation middleware generalizes the per-route pattern into a reusable factory. The goal: one consistent shape for errors, one place to evolve validation rules, and no per-route boilerplate.

import { z } from "zod";

// factory that validates any combination of body/query/params/headers
export function validate(schemas) {
  return async (req, res, next) => {
    const results = {};
    const issues = [];

    for (const prop of ["body", "query", "params", "headers"]) {
      if (!schemas[prop]) continue;
      const result = schemas[prop].safeParse(req[prop]);
      if (!result.success) {
        for (const issue of result.error.issues) {
          issues.push({
            location: prop,
            field:    issue.path.join("."),
            message:  issue.message,
            code:     issue.code,
          });
        }
      } else {
        results[prop] = result.data;
      }
    }
    if (issues.length) {
      return res.status(400).json({
        error: "Validation failed", issues, reqId: req.id,
      });
    }
    // stash cleaned versions — don't overwrite read-only req.query in Express 5
    if (results.body)   req.body   = results.body;
    if (results.params) req.params = results.params;
    req.validated = results;
    next();
  };
}

// use it
app.post("/posts/:id/comments",
  validate({
    params: z.object({ id: z.string().uuid() }),
    body:   z.object({
      text: z.string().trim().min(1).max(2000),
      replyTo: z.string().uuid().optional(),
    }).strict(),
    query:  z.object({ notify: z.coerce.boolean().optional() }),
  }),
  createCommentHandler
);

Design decisions worth articulating:

  • One middleware, all locations. Cuts boilerplate — handlers declare schemas for body/query/params once.
  • Aggregated errors. Return all issues at once with location + field + message — better UX than "fix, retry".
  • Reassign cleaned values. Downstream handlers see coerced/stripped data, not raw input.
  • Express 5 read-only query. req.query is a getter; write to req.validated.query instead.
  • Stable error shape. { error, issues[{location,field,message,code}], reqId } — clients parse this once.

For multi-framework shops, schema libraries like TypeBox (JSON Schema + TS inference) give you validators that work in Express, Fastify, and client-side form code. Consistent schemas across layers eliminate a huge class of "server accepts what client rejects" bugs.

Three hazards explained:

  1. Depth DoS. {a:{a:{a:{...(x 1000)}}}} in 100KB is cheap to send, expensive to validate recursively. Mitigate: cap body size; most schemas bounded naturally by .strict().
  2. Prototype pollution. Payloads like {"__proto__":{"admin":true}} used to mutate Object.prototype. Node 17+ fixed JSON.parse; still avoid Object.assign-style deep merges on raw input. Zod's .strict() rejects __proto__ as an unknown key.
  3. Mass assignment. Without .strict(), a client sending {email:"x", role:"admin"} to PATCH /profile can escalate privileges if your ORM passes through unknown fields. Always use strict schemas.

For deeply nested arrays, use streaming JSON parsers (stream-json) instead of loading everything into memory. Process line items one at a time — constant memory regardless of order size.

How do you implement a middleware for input validation using AJV in ExpressJS?

AJV (Another JSON Schema Validator) is the fastest JSON Schema validator on Node — compiles schemas to generated JavaScript for near-native performance. Use it when JSON Schema is your contract format or raw speed matters.

  type: "object",
  additionalProperties: false,
  required: ["email", "password", "name"],

    name:     { type: "string", minLength: 1, maxLength: 100 },
    age:      { type: "integer", minimum: 13, maximum: 120 },

  return (req, res, next) => {

          field:   e.instancePath.replace(/^\//, "") || e.params?.missingProperty,
          message: e.message,
          schemaPath: e.schemaPath,
        })),
      });
    }
    next();
  };
}

How do you handle real-time data updates using Server-Sent Events (SSE) in ExpressJS?

Server-Sent Events (SSE) is the simpler cousin of WebSockets: server-to-client only, plain HTTP, auto-reconnect in browsers, no handshake protocol. For notifications, activity feeds, live dashboards — anywhere you don't need client → server messages — SSE beats WebSockets on operational simplicity.

app.get("/events", requireAuth, (req, res) => {
  res.set({
    "Content-Type":      "text/event-stream",
    "Cache-Control":     "no-cache, no-transform",
    "Connection":        "keep-alive",
    "X-Accel-Buffering": "no",                       // disable Nginx response buffering
  });
  res.flushHeaders();

  const send = (event, data, id) => {
    if (id)    res.write(`id: ${id}\n`);
    if (event) res.write(`event: ${event}\n`);
    res.write(`data: ${JSON.stringify(data)}\n\n`);  // double-newline terminates event
  };

  // immediate "hello"
  send("connected", { userId: req.user.sub });

  // push updates from a pub/sub channel
  const sub = redis.duplicate();
  sub.subscribe(`user:${req.user.sub}:events`);
  sub.on("message", (_, msg) => {
    const event = JSON.parse(msg);
    send(event.type, event.data, event.id);
  });

  // heartbeat every 20s — keeps proxies from killing idle conns
  const heart = setInterval(() => res.write(":heartbeat\n\n"), 20_000);

  // cleanup on disconnect
  req.on("close", () => {
    clearInterval(heart);
    sub.unsubscribe().then(() => sub.quit());
    res.end();
  });
});

Question 97

Explain how to use the serve-favicon middleware to serve a favicon in ExpressJS.

serve-favicon is a tiny middleware that serves a favicon (favicon.ico) efficiently — it loads the file into memory once, skips the Express routing pipeline, sets long-lived cache headers, and short-circuits the request before morgan/logging can spam your logs with favicon hits.

import favicon from "serve-favicon";
import path from "node:path";
import { fileURLToPath } from "node:url";

const __dirname = path.dirname(fileURLToPath(import.meta.url));

// ideally register BEFORE logging middleware
app.use(favicon(path.join(__dirname, "public", "favicon.ico"), {
  maxAge: 7 * 24 * 60 * 60 * 1000,                   // 7 days (in ms) — aggressive cache
}));

app.use(morgan("combined"));
// ... routes ...

What it really saves:

  • Log noise. Browsers request /favicon.ico on every page load; without serve-favicon, each hit generates an access-log entry and counts against rate limits.
  • Disk I/O. The file is loaded into memory once at startup, not on every request.
  • Pipeline cost. Short-circuits before routing, validation, auth middleware runs.

2026 reality check: most production apps serve static assets (including favicons) via a CDN (Cloudflare, CloudFront) or reverse proxy (Nginx, Caddy). In that world, serve-favicon is redundant — the CDN caches the favicon at the edge, and Node never sees the request. The middleware earns its place in:

  • Standalone Express apps without a CDN (dev, intranet tools).
  • Monoliths that serve HTML + API from the same process.
  • Apps where quickly reducing favicon log noise matters and you can't change deployment topology.

Modern alternative: express.static with maxAge: "7d" on a public folder covers favicons alongside other assets. One middleware, one cache policy, one source of truth. For really production-grade setups, use <link rel="icon" href="https://cdn.example.com/favicon.ico"> in your HTML and let the CDN handle everything.

Question 98

How do you implement a custom request timeout middleware in ExpressJS?

A request-timeout middleware bounds how long any request can take, protecting against slow upstreams, pathological queries, or infinite-loop bugs. Express has no built-in timeout; Node's default is infinite once a response has started.

// per-request timeout with proper cleanup
function requestTimeout(ms) {
  return (req, res, next) => {
    // AbortController lets downstream work cancel itself when we time out
    const ac = new AbortController();
    req.signal = ac.signal;

    const timer = setTimeout(() => {
      ac.abort();
      if (!res.headersSent) {
        res.status(503).json({ error: "Request timed out", reqId: req.id });
      } else {
        // response started streaming — all we can do is destroy the socket
        res.destroy(new Error("Request timeout"));
      }
    }, ms);

    // clean up — whichever finishes first
    const clear = () => { clearTimeout(timer); };
    res.on("finish", clear);
    res.on("close",  clear);

    next();
  };
}

// apply globally, or per-route with tighter budgets
app.use(requestTimeout(30_000));                   // 30s default

// expensive routes get their own, shorter budget
app.get("/search", requestTimeout(5_000), searchHandler);

// downstream handlers can use req.signal to cancel upstreams
app.get("/upstream", async (req, res, next) => {
  try {
    const r = await fetch("https://slow.example.com", { signal: req.signal });
    res.json(await r.json());
  } catch (err) {
    if (err.name === "AbortError") return; // response already sent by timeout handler
    next(err);
  }
});

app.get("/posts", async (req, res) => {
  const limit = Math.min(100, Number(req.query.limit) || 20);

  const { rows } = await pool.query(
    `SELECT id, title, tags, created_at
       FROM posts
      WHERE ($1::text IS NULL OR $1 = ANY(tags))
      ORDER BY id DESC
      LIMIT $2`,
    [tag, limit]
  );
  res.json({ data: rows });

app.post("/orders", async (req, res, next) => {
  const client = await pool.connect();
  try {
    await client.query("BEGIN");
    const { rows: [order] } = await client.query(
      "INSERT INTO orders(customer_id, total) VALUES($1,$2) RETURNING *",
      [req.body.customerId, req.body.total]

      await client.query(
        "INSERT INTO order_items(order_id, sku, qty) VALUES($1,$2,$3)",

    res.status(201).json(order);
  } catch (err) {
    await client.query("ROLLBACK"); next(err);
  } finally {
    client.release();
  }
});
