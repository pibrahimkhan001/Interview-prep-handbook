Question 1

What is an API?

An API (Application Programming Interface) is a contract that lets two pieces of software talk to each other. You send a request describing what you want; the API sends back a response with data or confirmation. The rules of how to ask — URLs, parameters, data formats — are the API's "interface."

# ask GitHub for info about a user
GET https://api.github.com/users/torvalds

# response (JSON)
{
  "login": "torvalds",
  "name":  "Linus Torvalds",
  "company": "Linux Foundation",
  "public_repos": 9
}

Why APIs matter: they let your mobile app pull data from a server without owning the database; let one company's software integrate with another's (Stripe for payments, Twilio for SMS); and let teams split a large system into independent services that talk via well-defined contracts. In 2026, the vast majority of web traffic is machine-to-machine API calls, not humans visiting web pages.

When developers say "an API," they usually mean a web API — an HTTP-based service you call over the internet.

Question 4

What is a RESTful API?

A RESTful API follows the principles of REST (Representational State Transfer), a design style for HTTP APIs introduced by Roy Fielding in 2000. The key idea: treat everything as a resource that has a URL, and use standard HTTP methods (GET/POST/PUT/DELETE) to act on it.

GET    /users          # list users
GET    /users/42       # fetch user 42
POST   /users          # create a new user
PUT    /users/42       # replace user 42
PATCH  /users/42       # update part of user 42
DELETE /users/42       # delete user 42

Core properties of a RESTful API:

  • Resource-based URLs — nouns, not verbs (/users, not /getAllUsers).
  • Standard HTTP methods carry meaning (GET reads, POST creates, DELETE removes).
  • Stateless — each request carries everything it needs (auth token, parameters); the server doesn't remember previous calls.
  • Standard status codes (200 OK, 404 Not Found, 500 Server Error) tell the client what happened.
  • Representation — usually JSON; the same resource could theoretically be sent as XML, CSV, HTML.

Most "REST" APIs in the wild don't follow every REST rule (HATEOAS especially is rare), but they share enough conventions that developers can pick any one up quickly. That consistency is REST's real gift.

Question 5

What is JSON, and why is it commonly used in APIs?

JSON (JavaScript Object Notation) is a text format for representing structured data — numbers, strings, booleans, arrays, and objects. It was invented by Douglas Crockford around 2001 and took over from XML as the dominant API data format.

{
  "id": 42,
  "name": "Ada Lovelace",
  "active": true,
  "roles": ["admin", "editor"],
  "address": {
    "city": "London",
    "country": "UK"
  }
}

Why APIs love JSON:

  • Compact. Much smaller than equivalent XML — less bandwidth, faster parsing.
  • Human-readable. Easy to eyeball and debug with curl or DevTools.
  • Native to JavaScript. JSON.parse() / JSON.stringify() are built in; no library needed in browsers.
  • Universal. Every language has a parser; no one has to learn a new format.
  • Maps directly to data structures — objects become dictionaries/hashmaps, arrays become lists.

Limitations: no comments, no dates (just strings), no binary data (base64-encode it), and no schema (pair with JSON Schema or OpenAPI for that). For these reasons, some modern APIs use MessagePack or Protocol Buffers internally — but JSON remains the default for public APIs because of its universality.

Question 7

What is an endpoint in an API?

An endpoint is a specific URL where an API exposes a piece of functionality. Think of an endpoint as an "address" you send requests to; the API routes each URL + method combination to different code.

// a few endpoints from GitHub's REST API
GET  https://api.github.com/users/{username}          # fetch a user
GET  https://api.github.com/users/{username}/repos    # list their repos
POST https://api.github.com/repos/{owner}/{repo}/issues   # create an issue
PATCH https://api.github.com/user                     # update your profile

Key pieces of an endpoint:

  • Base URL — https://api.github.com — where the API lives.
  • Path — /users/{username} — identifies the resource. Placeholders like {username} become real values (/users/torvalds).
  • Method — GET, POST, PUT, etc. — describes the intent (read, create, update, delete).
  • Query parameters — optional filters after ?: /users?sort=newest&limit=10.

The same path can mean different things under different methods: GET /users/42 fetches user 42, DELETE /users/42 removes them. The method is as much part of the endpoint as the URL. When API documentation lists endpoints, it usually shows method + path together: DELETE /users/:id.

Question 8

What is an HTTP request?

An HTTP request is a message sent from a client (browser, mobile app, another server) to a server asking it to do something. Every API call is an HTTP request.

POST /users HTTP/1.1            # 1. request line — method + path + protocol version
Host: api.example.com           # 2. headers — metadata about the request
Authorization: Bearer abc123
Content-Type: application/json
Content-Length: 47

{ "name": "Ada", "email": "ada@example.com" }    # 3. body — optional payload

A request has three parts:

  • Request line — the HTTP method (GET, POST, etc.), the path, and the protocol version.
  • Headers — key/value metadata: content type, authentication, client info (User-Agent), what formats you accept, cache hints.
  • Body — optional data payload. GET/DELETE usually have no body; POST/PUT/PATCH typically carry one (JSON, form data, files).

The server reads the request, figures out what's being asked, does the work (reads a database, calls another service, etc.), and sends back an HTTP response with a status code (200, 404, etc.), its own headers, and an optional body containing the result. This request/response cycle is the fundamental building block of all HTTP APIs.

Question 9

What are the different types of HTTP methods?

HTTP defines a small set of methods (also called "verbs") that describe the intent of a request. Each one has a conventional meaning that REST APIs follow closely.

MethodIntentHas body?Safe?Idempotent?
GETRead a resourceNoYesYes
POSTCreate (or generic action)YesNoNo
PUTReplace a resource entirelyYesNoYes
PATCHUpdate part of a resourceYesNoUsually
DELETERemove a resourceNoNoYes
HEADLike GET but headers only, no bodyNoYesYes
OPTIONSAsk what's allowed (used for CORS preflight)NoYesYes

Safe means the method shouldn't change server state (GET should never create or delete). Idempotent means calling it multiple times has the same effect as calling it once — critical for retries on flaky networks (safe to retry PUT / DELETE; risky to retry POST without an idempotency key).

You'll see GET, POST, PUT, PATCH, and DELETE in ~95% of REST APIs. The others show up for specific purposes (CORS preflight, efficient probing). Picking the right verb makes your API predictable — and every HTTP tool in the world "just works" with it.

Question 10

Explain the difference between GET and POST methods.

GET reads data; POST submits data to create or trigger something. They're the two most common HTTP methods and the easiest to mix up.

AspectGETPOST
PurposeRetrieve informationCreate resources or submit data
Has a body?No (data goes in URL)Yes (data goes in body)
Idempotent?Yes — same call = same resultNo — each call may create a new thing

// GET — search for users
GET /users?name=ada&active=true

// POST — create a new user

Content-Type: application/json
{ "name": "Ada", "email": "ada@example.com" }

Rules of thumb: if you're just fetching something, use GET. If the URL would need to carry sensitive data or the action changes server state (creating, purchasing, sending), use POST. Never use GET for destructive actions — browsers and crawlers prefetch GET URLs, which can accidentally trigger them.

Question 11

What is the purpose of the PUT method?

PUT is used to replace an entire resource at a known URL. The client sends the complete new state, and the server either updates the existing resource or creates it at that URL if it didn't exist (sometimes called "upsert").

// PUT — replace user 42 entirely
PUT /users/42
Content-Type: application/json

{
  "name":  "Ada Lovelace",
  "email": "ada@example.com",
  "role":  "admin"
}
// response: 200 OK (updated) or 201 Created (just created)

Key properties:

  • Full replacement. You send the whole resource, not just the fields you're changing. Missing fields are set to their defaults or null.
  • Idempotent. Sending the same PUT twice produces the same state. Safe to retry on network failure.
  • Works on a specific URL. The client decides the resource's URL (/users/42), unlike POST where the server picks.

PUT vs PATCH: if you want to change just one field (e.g., a user's email), PATCH is usually better — it sends only the diff, avoiding the risk of accidentally wiping fields you forgot to include. Use PUT when you have the whole resource in hand (edit form with all fields, or config updates).

Question 12

What does the DELETE method do?

DELETE removes a resource. Once the server processes a successful DELETE, future GETs to that URL typically return 404.

DELETE /users/42

// common responses
204 No Content       # deleted, nothing to return (the conventional response)
200 OK               # deleted; body contains the deleted resource
404 Not Found        # already deleted (or never existed)
403 Forbidden        # you're not allowed to delete this

Key properties:

  • Idempotent. Deleting an already-deleted resource should still be safe — most APIs return 204 on the second call (or 404 if "it must exist") — never an error that breaks retries.
  • Usually no body. The URL alone identifies what to delete.
  • Often a soft delete. Real-world APIs usually set a deleted_at flag rather than wiping the row — enables recovery, audit, and compliance (GDPR's "right to be forgotten" still requires a proper purge process).

Safety in practice: because DELETE is destructive and irreversible for the user, APIs typically require authentication, log every call, and may ask for extra confirmation (a password re-entry, an "Are you sure?" flow on the client). Some APIs implement DELETE as a two-step flow: first PATCH to mark for deletion, then a cleanup job removes after a recovery window.

Question 13

What are status codes in HTTP?

HTTP status codes are 3-digit numbers in every response that tell the client what happened. They're grouped into five classes based on the first digit.

ClassMeaningExamples
1xxInformational (rare)100 Continue, 101 Switching Protocols
2xxSuccess200 OK, 201 Created, 204 No Content
3xxRedirection301 Moved Permanently, 302 Found, 304 Not Modified
4xxClient error — the request was bad400 Bad Request, 401 Unauthorized, 404 Not Found, 429 Too Many Requests
5xxServer error — the server failed500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable

Why status codes matter: they let generic tools (browsers, load balancers, monitoring, retry libraries) make decisions without parsing your response body. A 429 means "back off and retry"; a 500 might trigger an alert; a 404 can be cached; a 401 should redirect to login. Using the wrong code (like returning 200 with { "error": "..." }) breaks this ecosystem.

You'll memorize 200, 201, 204, 301, 400, 401, 403, 404, 429, 500, 502, 503 — those cover 95% of real traffic. MDN's HTTP status reference is the definitive source for the rest.

Question 14

What does a 200 status code signify?

200 OK is the most common success response. It means the request was processed successfully and a response body follows.

GET /users/42 HTTP/1.1

HTTP/1.1 200 OK
Content-Type: application/json

{ "id": 42, "name": "Ada Lovelace", "role": "admin" }

When to return 200:

  • GET that successfully retrieves a resource (with the resource in the body).
  • PUT / PATCH that updates an existing resource (often with the updated version in the body).
  • POST for actions that aren't "create" — like /orders/42/refund or /sessions/logout.
  • DELETE when you want to return the deleted resource (use 204 if the body is empty).

Related 2xx codes:

  • 201 Created — specifically for successful POST that created a new resource; include a Location header pointing to the new URL.
  • 202 Accepted — request valid, work queued asynchronously; follow-up needed to check result.
  • 204 No Content — success, no body to return (typical for DELETE).

Returning 200 for every response (even failures) is a common anti-pattern — it defeats status-based tooling. Match the code to what actually happened.

Question 15

What does a 404 status code mean?

404 Not Found means the server understood the request fine but there's nothing at that URL. It's the single most famous error code — the web page you hit when a link is broken.

GET /users/9999 HTTP/1.1

HTTP/1.1 404 Not Found
Content-Type: application/json

{ "error": "User not found", "id": 9999 }

When to return 404:

  • The specific resource doesn't exist (/users/9999 when user 9999 was never created or was deleted).
  • The URL doesn't match any route in your API.
  • (Sometimes) the resource exists but the caller shouldn't know about it — some APIs return 404 instead of 403 to avoid revealing that resources exist.

Don't confuse 404 with:

  • 400 Bad Request — the URL is syntactically wrong or parameters are invalid.
  • 410 Gone — the resource used to exist and definitely won't come back. Rare; used when you want crawlers to stop indexing.
  • 204 No Content — the resource exists but has no body to return (e.g., after a successful DELETE).

For a good user experience, include a helpful error message explaining what was missing — not just a blank 404 page.

Question 16

What does a 500 status code indicate?

500 Internal Server Error is the generic "something went wrong on our side" response. The client's request was syntactically valid, but the server crashed, had a bug, or hit an unexpected error while processing it.

POST /orders HTTP/1.1
Content-Type: application/json

{ "productId": 42, "quantity": 2 }

HTTP/1.1 500 Internal Server Error
Content-Type: application/json

{ "error": "Something went wrong", "reqId": "req_abc123" }

When 500 happens:

  • Unhandled exception in your code (null pointer, type error, unexpected input).
  • Database connection lost.
  • Out-of-memory, disk full, or process crash.
  • A dependency (payment gateway, email service) returned an unexpected response that your code didn't handle.

Related 5xx codes that are more specific:

  • 502 Bad Gateway — your server tried to call another server, which failed or returned invalid data.
  • 503 Service Unavailable — server is temporarily down (maintenance, overloaded).
  • 504 Gateway Timeout — your server waited too long for an upstream response.

Best practices: never leak stack traces or internal details in 500 responses — attackers use them to probe your system. Log everything server-side with a request id, return a generic message + that id to the user. Every 500 should trigger an alert; they're bugs, not normal.

Question 17

What is an API key?

An API key is a long, random string that identifies a specific client (app or developer account) calling an API. The client sends the key with each request, and the server uses it to authenticate and track usage.

// usually sent as a header
GET /v1/weather?city=London
X-API-Key: sk_live_4a8b3c2d9e7f1a6b5c4d3e2f

// or as a query parameter (less secure)
GET /v1/weather?city=London&api_key=sk_live_4a8b3c2d9e7f1a6b5c4d3e2f

What API keys are good for:

  • Identifying the client — "which app is calling?" so you can meter, log, or bill per customer.
  • Rate limiting — each key has its own quota.
  • Revocation — if a key leaks, you rotate it without affecting other clients.

Limitations — why API keys aren't "real" auth:

  • A leaked key gives full access to anyone who grabs it. Unlike passwords, keys are usually stored in app code/env files, where they can leak via git commits, logs, or screenshots.
  • They identify the app, not the user. Use OAuth / JWT for user-specific authorization.

Best practices: always send keys in headers (not URLs — URLs end up in logs), always over HTTPS, rotate periodically, scope to specific permissions ("read-only key" vs "full-access key"), and monitor for anomalous usage. Most modern APIs (Stripe, OpenAI) use API keys for server-to-server auth, OAuth/JWT for user-facing flows.

Question 18

What is OAuth?

OAuth 2.0 (and its 2026 update, OAuth 2.1) is a standard protocol for delegated authorization — letting one app act on your behalf with another service without sharing your password.

Example: "Sign in with Google" or "Connect your GitHub." You click "Allow," and the app gets an access token scoped to specific permissions (read your email, post to your feed). The app never sees your password.

// 1. User clicks "Sign in with Google" — redirected to Google
GET https://accounts.google.com/o/oauth2/v2/auth?
  client_id=MY_APP&
  redirect_uri=https://myapp.com/callback&
  scope=email+profile&
  response_type=code

// 2. User approves — Google redirects back with a code
GET https://myapp.com/callback?code=ABC123

// 3. Server exchanges code for a token
POST https://oauth2.googleapis.com/token
// returns: { "access_token": "ya29.a0...", "expires_in": 3600 }

// 4. App calls Google APIs with the token
GET https://www.googleapis.com/userinfo/v2/me
Authorization: Bearer ya29.a0...

Key concepts:

  • Access token — short-lived credential (usually 1 hour) sent with API calls.
  • Refresh token — longer-lived credential used to get new access tokens.
  • Scopes — what the app is allowed to do (read_email, post_tweets).
  • PKCE — now mandatory in OAuth 2.1 for all clients; prevents code-interception attacks.

OAuth powers almost every "Sign in with X" button online. Libraries like Auth.js or better-auth implement the full flow for you — don't roll your own; the spec has many footguns.

Question 19

What is a token, and how is it used in API authentication?

A token is a credential (usually a long opaque or signed string) that a client presents with each API request to prove who they are. It replaces sending a username/password on every call.

// 1. user logs in once with credentials
POST /auth/login
{ "email": "ada@example.com", "password": "hunter2" }

// 2. server returns a token
{ "accessToken": "eyJhbGc...", "expiresIn": 900 }

// 3. client sends token on every subsequent API call
GET /me
Authorization: Bearer eyJhbGc...

Why tokens over passwords on every request:

  • Security. Password is entered once (and ideally stored nowhere); tokens are scoped and time-limited.
  • Revocable. If a token leaks, you invalidate just that token, not the user's password.
  • Stateless servers. Tokens (especially JWTs) can carry identity claims, so servers don't need a database lookup for every request.
  • Scoped. A token can grant narrow permissions (read-only, one specific API).

Common token types:

  • JWT (JSON Web Token) — a signed, self-describing token that the server verifies without a DB lookup.
  • Opaque token — a random string; the server looks it up in Redis/DB to find out who it belongs to.
  • API key — a long-lived token for server-to-server use (see Q17).

Always send tokens over HTTPS, store them carefully (HttpOnly cookies or secure storage, never localStorage for session tokens), and keep expiry short (15 min) with refresh tokens for longer sessions.

Question 20

Explain the concept of rate limiting in APIs.

Rate limiting caps how many requests a client can make in a given time window. It protects the API from abuse, accidental infinite loops, and traffic spikes that would otherwise overwhelm the server.

// response when you've hit the limit
HTTP/1.1 429 Too Many Requests
RateLimit-Limit:     100
RateLimit-Remaining: 0
RateLimit-Reset:     60              # seconds until reset
Retry-After:         60

{ "error": "Rate limit exceeded. Retry in 60 seconds." }

Typical rate-limit schemes:

  • Per IP — good for public, unauthenticated endpoints (login, signup).
  • Per API key / user — fair sharing; free users might get 100 req/hour, paid plans 10,000.
  • Per endpoint — tighter limits on expensive operations (search, AI calls).

What clients should do when they hit 429:

  • Read the Retry-After header and wait that long before retrying.
  • Use exponential backoff with jitter — don't hammer the server the instant the window resets.
  • Cache responses locally where possible to reduce call volume.

Why rate limits matter: without them, a single buggy client (or a DDoS attacker) can take your API down for everyone. The standard headers (RateLimit-Limit, RateLimit-Remaining, RateLimit-Reset) are defined in RFC 9331 and let well-behaved clients pace themselves automatically.

Question 21

What is API versioning, and why is it important?

API versioning means releasing a new version of your API alongside the old one, so existing clients keep working when you change the interface. Without versioning, any change — adding a required field, renaming an endpoint — breaks every app that calls you.

// three common versioning strategies
GET /v1/users/42                                # URL path — easiest, most visible
GET /users/42       Accept: application/vnd.myapi.v1+json   # header — cleaner URL
GET /users/42?version=1                         # query param — discouraged

Why it's important:

  • Client stability. Mobile apps can take weeks or months to update. You can't force everyone to upgrade at once.
  • Freedom to evolve. You can fix design mistakes or add new features without fear of breaking old clients.
  • Clear deprecation. You announce "v1 will be removed on 2027-01-01"; clients migrate on their own schedule.

When to bump the version:

  • Breaking change — new required field, renamed response key, removed endpoint, changed data type. This goes in a new version.
  • Non-breaking change — adding optional fields, new endpoints, new status codes. No version bump needed; clients keep working.

Practical tip: most mature APIs (GitHub, Stripe) use URL-based versioning for its simplicity. Run at most 2-3 versions in parallel; maintaining more is a support nightmare. Communicate deprecations clearly with headers (Sunset, Deprecation) and email warnings.

Question 22

What is the purpose of headers in an HTTP request?

HTTP headers are key-value metadata attached to every request and response. They carry information that doesn't fit the body — content type, authentication, caching hints, client identification, and more.

POST /orders HTTP/1.1
Host:            api.example.com         # which server to send this to
Authorization:   Bearer eyJhbGc...        # who's calling
Content-Type:    application/json         # what format the body is in
Content-Length:  87                       # how many bytes the body is
Accept:          application/json         # what format I want back
User-Agent:      my-app/1.0               # identifies the client software
X-Request-Id:    req_abc123               # correlation id for tracing

{ "productId": 42, "quantity": 2 }

Common request headers:

  • Authorization — credentials (token, API key, Basic auth).
  • Content-Type — format of the body (JSON, form-data, XML).
  • Accept — what the client wants back (application/json, text/html).
  • User-Agent — identifies the client software; servers sometimes use it for analytics or routing.
  • Cookie — session or tracking data from the browser.

Common response headers: Content-Type, Cache-Control (caching rules), Set-Cookie, Location (for redirects), and custom headers like X-RateLimit-Remaining.

Headers are case-insensitive (Content-Type and content-type are the same) and can't contain newlines — they're a flat list of short strings, not a place for structured data.

Question 23

What is the Accept header used for?

The Accept header tells the server what response formats the client can understand. It's how one URL can serve different formats to different clients — JSON for your mobile app, HTML for your browser, CSV for a spreadsheet tool.

// same URL, three clients, three formats
GET /reports/sales
Accept: application/json        →   { "total": 12000, ... }

GET /reports/sales
Accept: text/csv                →   date,total\n2026-04-01,12000\n

GET /reports/sales
Accept: text/html               →   <h1>Sales report</h1><p>...</p>

How it works: the server picks the best match from the client's preferences and returns the resource in that format. If the server can't satisfy the request, it returns 406 Not Acceptable.

Quality values: you can rank preferences with q weights:

Accept: application/json, text/html;q=0.9, */*;q=0.1
// "I strongly prefer JSON, accept HTML, and anything else as last resort"

Other Accept-* headers serve similar purposes:

  • Accept-Language — what languages the client prefers (en-US, de;q=0.8).
  • Accept-Encoding — what compression the client supports (gzip, br).
  • Accept-Charset — what character encoding (mostly obsolete; UTF-8 is standard).

Most REST APIs today default to JSON regardless of Accept, but supporting multiple formats is what content negotiation is for — remember to also set Vary: Accept in responses so caches store the right variant per format.

Question 24

What is the Content-Type header?

The Content-Type header tells the other side what format the body is in. It appears both in requests (describing what the client is sending) and responses (describing what the server is returning).

// client sends JSON
POST /users
Content-Type: application/json
{ "name": "Ada" }

// client sends HTML form data
POST /login
Content-Type: application/x-www-form-urlencoded
email=ada@example.com&password=hunter2

// client uploads a file
POST /photos
Content-Type: multipart/form-data; boundary=----abc
...

// server returns JSON
HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8
{ "id": 42, "name": "Ada" }

Common Content-Types:

  • application/json — JSON data; by far the most common for APIs.
  • application/x-www-form-urlencoded — classic HTML form submissions.
  • multipart/form-data — forms with file uploads.
  • text/html — web pages.
  • text/plain — plain text.
  • application/xml — XML data.
  • image/png, application/pdf, etc. — binary files.

Why it matters: without Content-Type, the receiver can't parse the body. If you forget it on a JSON POST request, Express's express.json() middleware skips the body and req.body is undefined. Setting a wrong Content-Type is a classic "my API doesn't work" debugging moment. Always include charset=utf-8 for text-based formats to avoid encoding surprises.

Question 25

Explain the difference between query parameters and path parameters.

Both query parameters and path parameters pass data to an API via the URL, but they have different purposes and conventions.

GET /users/42/posts/7
           ↑          ↑
      userId=42    postId=7

// query parameters — filter, sort, paginate
GET /users?role=admin&sort=newest&limit=20

Question 26

What is CORS, and why is it important in APIs?

CORS (Cross-Origin Resource Sharing) is a browser security feature that controls whether JavaScript on one domain can call an API on a different domain. Without CORS, your web app at app.example.com can't call your API at api.example.com — the browser blocks the response.

// browser on https://app.example.com runs:
fetch("https://api.example.com/users")
// → browser sends OPTIONS preflight first to check permission

// API must respond with CORS headers
Access-Control-Allow-Origin:      https://app.example.com
Access-Control-Allow-Methods:     GET, POST
Access-Control-Allow-Headers:     Authorization, Content-Type
Access-Control-Allow-Credentials: true

What CORS protects against: a malicious site (evil.com) trying to use your logged-in session to call bank.com/transfer. Browsers send the victim's cookies automatically, but CORS blocks the response from being read by the attacker's JavaScript.

Key concepts:

  • Origin = protocol + domain + port. http://example.com and https://example.com are different origins.
  • Simple requests (GET/POST with plain content types) are sent directly.
  • Preflight — for anything fancier (custom headers, JSON POST, PUT/DELETE), the browser sends an OPTIONS request first to check permission.
  • CORS is a browser thing. Server-to-server calls and mobile apps ignore it entirely.

To enable CORS on your API, respond with Access-Control-Allow-Origin. In Express, use the cors middleware with an allowlist — never * with credentials.

Question 27

What is the purpose of the OPTIONS HTTP method?

OPTIONS asks the server "what can I do with this URL?" without actually doing it. In practice, its main use today is the CORS preflight — browsers automatically send OPTIONS before certain cross-origin requests to check if the real request is allowed.

// browser-generated preflight — happens before your actual fetch
OPTIONS /users HTTP/1.1
Origin: https://app.example.com
Access-Control-Request-Method:  POST
Access-Control-Request-Headers: Authorization, Content-Type

// server's preflight response — lists what's permitted
HTTP/1.1 204 No Content
Access-Control-Allow-Origin:  https://app.example.com
Access-Control-Allow-Methods: GET, POST, DELETE
Access-Control-Allow-Headers: Authorization, Content-Type
Access-Control-Max-Age:       86400             # cache this answer 24h

When OPTIONS fires:

  • Cross-origin request with a non-simple method (PUT, DELETE, PATCH).
  • Cross-origin request with a custom header (Authorization, X-Request-Id).
  • Cross-origin POST with a JSON body (Content-Type: application/json).

Other (rare) OPTIONS uses: discovering what methods a URL supports (Allow: GET, POST, DELETE in the response). Mostly legacy.

Performance tip: set Access-Control-Max-Age (up to 24h) so browsers cache the preflight answer. Without it, every cross-origin API call triggers two network round-trips — half your latency is preflights.

Question 28

What is an API gateway?

An API gateway sits in front of one or more backend services and acts as the single entry point for all API traffic. Clients talk to the gateway; the gateway routes, authenticates, rate-limits, and aggregates before passing the request to the right service.

                        ┌─────────────────┐
                        │   API Gateway   │
client  ───────▶        │                 │
                        │ • auth          │  ──▶ user-service
                        │ • rate limit    │  ──▶ orders-service
                        │ • routing       │  ──▶ payments-service
                        │ • logging       │  ──▶ notifications-service
                        │ • caching       │
                        └─────────────────┘

What gateways typically do:

  • Route requests to the right backend service based on path (/users/* → user service).
  • Authenticate once at the edge; pass a verified identity downstream.
  • Rate limit across services to prevent one client from hogging everything.
  • TLS termination — backend services can run plain HTTP internally.
  • Aggregate — combine responses from multiple services into one (BFF pattern).
  • Log & monitor — one place to collect metrics and traces.

Common gateway software: Kong, KrakenD, Traefik, Envoy, cloud-native options like AWS API Gateway, Google Cloud Endpoints, and Azure API Management. For simpler needs, Nginx or a thin Node/Go service handles 80% of what a gateway does.

Question 29

Explain the concept of idempotency in APIs.

Idempotency means making the same request multiple times has the same effect as making it once. Idempotent operations are safe to retry on network failures — a huge deal on flaky mobile connections.

MethodIdempotent?Why
GETYesReading doesn't change state
PUTYesReplacing with the same data = same state
DELETEYesDeleting an already-deleted thing = still deleted
POSTNo (default)"Create a user" twice = two users
PATCHDepends{ "counter": { "$inc": 1 } } is not idempotent

Making POST idempotent — the Idempotency-Key pattern:

POST /payments HTTP/1.1
Idempotency-Key: charge_2026_04_19_abc123       # client-generated unique id
{ "amount": 100, "currency": "USD" }

// server stores the result of each key for 24h
// retrying with the same key returns the SAME response — no duplicate charge

Why it matters in real life: users double-tap buttons; mobile networks drop mid-request; proxies retry automatically. Without idempotency, each retry might create a duplicate order, double-charge a card, or send the same email twice. Stripe, Shopify, and other payments APIs require idempotency keys for this reason. On your own API, opt-in via the Idempotency-Key header for any write that's expensive to duplicate.

Question 30

What is HATEOAS in RESTful APIs?

HATEOAS (Hypermedia As The Engine Of Application State) is the "full REST" principle: responses should include links that tell the client what it can do next, so the client doesn't hardcode URLs.

// plain REST response — client has to know /orders/42/cancel
{
  "id": 42,
  "status": "pending",
  "total": 99.99
}

// HATEOAS — response includes next-action links
{
  "id": 42,
  "status": "pending",
  "total": 99.99,
  "_links": {
    "self":   { "href": "/orders/42" },
    "cancel": { "href": "/orders/42/cancel", "method": "POST" },
    "pay":    { "href": "/orders/42/pay",    "method": "POST" },
    "items":  { "href": "/orders/42/items" }
  }
}

The theoretical benefit: the client doesn't need to know your URL structure. It fetches the root /, gets links, follows them. When URLs change, nothing breaks. It's how the web works — you don't hand-type URLs into your browser; you click links.

Reality in 2026: HATEOAS is rare. Most "RESTful" APIs skip it because:

  • It doubles response size with metadata most clients ignore.
  • Developers prefer typed API clients (OpenAPI-generated SDKs) that hardcode routes with compile-time safety.
  • The benefit (evolve URLs without breaking clients) is handled by versioning instead.

You'll see HATEOAS in enterprise systems, hypermedia-focused frameworks (HTMX), and APIs that need deep discoverability (browser-rendered admin UIs). For most web APIs, it's optional.

Question 31

What is the difference between synchronous and asynchronous API calls?

The difference is whether the caller waits for the result (synchronous) or gets a ticket and checks back later (asynchronous).

SynchronousAsynchronous
Client waits for the full responseClient gets an immediate "accepted" reply
Simple to code and understandRequires polling, webhooks, or WebSocket updates
Times out on long-running workSuits long-running tasks (PDF gen, AI inference)
Returns 200 with the resultReturns 202 Accepted with a status URL
// synchronous — client blocks until done
POST /users
{ "name": "Ada" }
→ HTTP/1.1 201 Created
  { "id": 42, "name": "Ada" }

// asynchronous — server queues the job, returns a handle
POST /reports/generate
{ "type": "annual", "year": 2026 }
→ HTTP/1.1 202 Accepted
  Location: /jobs/xyz
  { "jobId": "xyz", "status": "queued" }

// client polls (or gets notified via webhook)
GET /jobs/xyz
→ { "status": "done", "result": "/reports/123.pdf" }

When to use async: any operation that could take more than a few seconds — generating PDFs, transcoding video, running ML models, bulk imports. Sync calls fail over at load-balancer timeouts (30-60s), so async protects your API from cascading failures.

Delivery patterns: polling (client checks periodically), webhooks (server calls the client back), SSE/WebSocket (live push). Pick based on how often the status changes and who has the easier infrastructure.

Question 32

What is a webhook?

A webhook is a reverse API call — instead of polling, you give a provider a URL and they POST to it when something happens. It flips the request direction: your server becomes the "client" and the third-party service becomes the "caller."

// 1. you register a webhook with Stripe
POST https://api.stripe.com/v1/webhook_endpoints
{ "url": "https://myapp.com/webhook/stripe",
  "events": ["payment.succeeded", "payment.failed"] }

// 2. later — a customer pays — Stripe POSTs to your URL
POST /webhook/stripe HTTP/1.1
Stripe-Signature: t=1234567,v1=abc123...
{ "type": "payment.succeeded", "data": { ... } }

// 3. your server processes the event (mark order paid, email receipt, etc.)

Why webhooks beat polling: instead of asking "did anything happen?" every 30 seconds, you're notified instantly when something happens. Lower latency, lower load, real-time UX.

Handling webhooks correctly:

  • Verify signatures. Attackers can POST fake webhooks to your public endpoint. Providers sign payloads with a secret; you verify before trusting.
  • Return 200 fast. Providers retry if you respond slowly or with an error. Queue work asynchronously; ack the webhook immediately.
  • Dedupe by event ID. Providers retry on failure and can send the same event twice. Store processed IDs.
  • Be idempotent. Processing the same event twice should produce the same result.

Every modern SaaS (Stripe, GitHub, Slack, Shopify) offers webhooks. It's how integrations connect reliably — like an API-to-API push notification.

Question 33

Explain the concept of pagination in APIs.

Pagination is how APIs return manageable chunks of a large collection instead of dumping thousands of items in one response. Without pagination, fetching "all users" on a site with 10 million users would crash both the server and the client.

StyleURL looks likeBest for
Offset / page?page=2&limit=20Small datasets, "jump to page" UIs
Cursor / keyset?cursor=eyJpZCI6MTAwfQLarge datasets, infinite scroll
Time-based?before=2026-01-01&limit=50Feeds, activity streams
// offset pagination response
{
  "data": [ { "id": 21, ... }, { "id": 22, ... }, ... ],
  "meta": {
    "page":       2,
    "limit":      20,
    "total":      10_000,
    "totalPages": 500
  }
}

// cursor pagination response
{
  "data":       [ ... ],
  "nextCursor": "eyJpZCI6MTAwfQ",        // opaque — clients pass it back as-is
  "hasMore":    true
}

Offset vs cursor trade-off: offset is intuitive (?page=5) but slows down on huge datasets (the DB scans skipped rows). Cursor is fast and stable (insertions don't shift pages) but you can't "jump to page 50" — only next/previous.

Best practices: always cap limit (typically 100); include hasMore or nextCursor so clients don't guess when to stop; document defaults clearly. GitHub and Stripe use cursor pagination — a strong signal for APIs at scale.

Question 34

What is the difference between an API endpoint and a resource?

In REST, a resource is the thing — a user, an order, a photo — that your API exposes. An endpoint is a specific URL+method combination that acts on a resource.

// resource: "user"
// endpoints:
GET    /users        # list users
POST   /users        # create a user
GET    /users/42     # fetch user 42
PUT    /users/42     # replace user 42
DELETE /users/42     # delete user 42
ResourceEndpoint
Abstract concept ("a user")Concrete URL + method (GET /users/42)
One per type of thingMany per resource (read, create, update, delete, search, etc.)
Modeled as a nounMethod + URL combined

Think of it like files on a filesystem: the resource is the file; the endpoints are the operations you can perform on it (read, write, delete). The same file has multiple operations; two different files have their own operations.

In API documentation, resources are usually top-level sections ("Users," "Orders," "Payments"); endpoints are listed underneath as the specific requests you can make against that resource. A good API design starts by identifying resources first, then deciding what endpoints each resource needs.

RESTful APIs are stateless, so every request must carry its own authentication — there's no "session" the server remembers. The client proves identity on each call, usually via an HTTP header.

Question 36

What is an API rate limit, and how can it be implemented?

An API rate limit is a cap on how many requests a single client can make within a time window. It protects the API from abuse (bots, scrapers), accidental overuse (buggy retry loops), and fair sharing (one client can't hog the server).

// common server response when the limit is hit
HTTP/1.1 429 Too Many Requests
RateLimit-Limit:     1000                # allowed per window
RateLimit-Remaining: 0                   # left in current window
RateLimit-Reset:     60                  # seconds until reset
Retry-After:         60

{ "error": "Rate limit exceeded. Retry in 60 seconds." }

How you can implement one in Express:

import rateLimit from "express-rate-limit";

const limiter = rateLimit({
  windowMs: 15 * 60 * 1000,              // 15 minutes
  max:      100,                          // 100 requests per IP per window
  standardHeaders: true,
  keyGenerator: (req) => req.user?.id ?? req.ip,
});
app.use("/api", limiter);

Implementation options:

  • In-memory counter — fine for single-server apps.
  • Redis counter — needed for multi-server deployments so counts are shared.
  • Edge rate limiting (Cloudflare, AWS WAF) — stops abusive traffic before it reaches your servers.

Common algorithms: fixed window (simple but bursty at boundaries), sliding window (smoother), token bucket (allows bursts up to a cap), leaky bucket (smooths traffic). For most APIs, express-rate-limit + Redis is enough; reach for more complex algorithms only when you hit their limits.

Question 38

Public APIs are open to any developer who signs up for a key. Private (internal) APIs are only used inside one company.

Explain the concept of API mocking.

API mocking means simulating an API's responses without the real backend. You hit a fake server that returns canned data, so you can build and test against it before the real API exists, or when it's slow, expensive, or unreliable.

Question 43

What are the advantages of using GraphQL over REST?

GraphQL's main wins come from solving two specific problems of REST APIs: over-fetching (you get more data than you need) and under-fetching (you need multiple requests to piece together one view).

Concrete benefits:

  • Ask for exactly what you need. A mobile screen showing just user name + avatar gets just those fields — no wasted bandwidth on the 30 other fields on the user record.
  • One request, many resources. Fetching a user plus their latest posts plus the post authors' avatars is one query, not three round-trips.
  • Strongly typed schema. The GraphQL schema is mandatory (unlike OpenAPI which is optional). Every field has a type; clients can auto-generate type-safe code (graphql-codegen).
  • Evolves without versioning. Deprecate fields with @deprecated instead of releasing v2. Old queries keep working.
  • Introspection. Clients can ask the server "what types and fields exist?" Tools like GraphiQL and Apollo Studio make exploration easy.
  • Subscriptions built-in for real-time updates over WebSockets.

Where REST still wins:

  • HTTP caching — REST with GET + Cache-Control works with CDNs; GraphQL POSTs don't.
  • File uploads — not part of spec; needs extensions.
  • Simpler — for CRUD APIs, GraphQL is extra machinery.
  • Public APIs — REST is the universal skill; GraphQL requires client adoption.
  • N+1 queries — you must use DataLoader; easy to create performance nightmares otherwise.

In 2026, many teams run both: REST for third-party integration, GraphQL for their own frontend. tRPC (for all-TypeScript stacks) gives GraphQL's type-safety without the schema overhead.

Question 44

What is an API gateway, and what are its benefits?

An API gateway is a server that sits between clients and your backend services, acting as the single entry point for all API traffic. Every request flows through it before reaching the real service.

client  ──▶  API Gateway  ──▶  user-service
                           ──▶  orders-service
                           ──▶  payments-service

What the gateway handles (so your services don't have to):

  • Routing. /api/users/* → user service, /api/orders/* → orders service.
  • Authentication. Validate tokens once at the edge; pass a verified identity to downstream services.
  • Rate limiting. One place to enforce limits across all services.
  • TLS termination. Services behind the gateway run plain HTTP internally.
  • Logging & monitoring. One consistent place to emit structured logs, traces, and metrics.
  • Request/response transformation. Aggregate multiple backend calls into one client response (BFF pattern), or translate formats (gRPC ↔ REST).
  • Caching. Cache GET responses at the gateway to reduce load on backends.

Benefits:

  • Decoupling. Clients don't know your internal service topology; change services without client rewrites.
  • Cross-cutting concerns in one place. Update rate limiting once, not across 12 services.
  • Security surface shrinkage. Backend services aren't directly exposed to the internet.

Popular options: Kong, KrakenD, Traefik, Envoy, and cloud-native services like AWS API Gateway, Google Cloud Endpoints, and Azure API Management. For simpler needs, Nginx handles 80% of what a gateway does.

Question 45

How do you handle errors in APIs?

Good API error handling has three parts: return the right status code, return a structured error body, and never leak internals.

// consistent error shape across ALL endpoints
{
  "error": {
    "code":    "USER_NOT_FOUND",
    "message": "No user with id 9999",
    "details": [
      { "field": "id", "issue": "invalid" }
    ]
  },
  "reqId": "req_abc123"        // correlation id for support tickets
}

Match the status to what happened:

  • 400 Bad Request — invalid input (malformed JSON, missing fields, bad types).
  • 401 Unauthorized — no or invalid credentials.
  • 403 Forbidden — authenticated but not allowed.
  • 404 Not Found — resource doesn't exist.
  • 409 Conflict — duplicate, stale update, already-exists.
  • 422 Unprocessable Entity — valid syntax but semantic validation failed.
  • 429 Too Many Requests — rate-limited.
  • 500 Internal Server Error — bug or crash in your code.
  • 502/503/504 — upstream dependency failed or is down.

Critical rules:

  • Never return 200 with { "error": ... }. Breaks every HTTP tool (monitoring, retries, caches).
  • Include a machine-readable code — clients can switch on USER_NOT_FOUND; they can't reliably parse human prose.
  • Never leak stack traces or SQL errors in production responses — attackers use them to map your system.
  • Always include a request id — users quote it in support tickets; you use it to find logs.
  • Validation errors should list all issues at once, not one at a time (better UX).

Standard format reference: RFC 7807 application/problem+json defines a consistent error shape. Many APIs adopt a light version of it.

Question 46

What is an API client?

An API client is any piece of software that calls an API. That's a broad definition — a browser, a mobile app, a Python script, another server, even a CLI tool like curl.

// a raw HTTP client — works everywhere
curl -H "Authorization: Bearer abc123" https://api.example.com/users/42

// a browser calling with fetch
const res = await fetch("https://api.example.com/users/42", {
  headers: { "Authorization": "Bearer abc123" }
});

// a dedicated SDK — typed, handles auth, retries, pagination
import { OpenAI } from "openai";
const client = new OpenAI({ apiKey });
const response = await client.chat.completions.create({ ... });

Kinds of API clients, from low-level to high-level:

  • Raw HTTP libraries (fetch, axios, requests in Python) — give you full control, need you to handle everything (auth, retries, pagination).
  • Generated clients (from OpenAPI, GraphQL schema) — typed, automatically in sync with the API.
  • Official SDKs (Stripe, OpenAI, AWS) — polished, handle rate limits, retries, pagination, and edge cases.
  • GUI clients (Postman, Insomnia, Bruno) — for manual exploration and testing.

What a good client library does for you: authentication, automatic retries with backoff, request/response serialization, timeouts, rate-limit awareness, pagination helpers, typed responses. Writing these from scratch is fiddly — prefer the official SDK when one exists; for smaller APIs, generate a client from the OpenAPI spec with openapi-typescript or Orval.

Question 47

What is a response body in an HTTP response?

The response body is the optional data payload in an HTTP response — what follows the status line and headers. For APIs, this is usually the JSON you care about; for websites, it's the HTML page.

HTTP/1.1 200 OK                    ← status line
Content-Type: application/json     ← headers
Content-Length: 42
Date: Sun, 19 Apr 2026 12:00:00 GMT

{ "id": 42, "name": "Ada Lovelace" }   ← body

What bodies carry, by status class:

  • 2xx (success) — the requested resource, the created resource, or nothing (204 No Content).
  • 3xx (redirect) — usually empty; the Location header carries the real info.
  • 4xx (client error) — an error object explaining what was wrong.
  • 5xx (server error) — a generic error (don't leak internals).

Not every response has a body:

  • 204 No Content — success with no data (common after DELETE).
  • 304 Not Modified — cached copy still valid; body elided.
  • HEAD responses — headers only, by design.

Body format is declared in the Content-Type header (application/json, text/html, image/png, etc.). Binary responses (images, PDFs) send raw bytes, not JSON. Very large bodies can be streamed — the client processes chunks as they arrive without buffering the whole thing.

Question 48

Explain the difference between SOAP and REST in terms of state.

The key difference is statelessness. REST is strictly stateless by design; SOAP can (and often does) maintain state across calls.

AspectRESTSOAP
Server-side session stateNone — each request is independentCan track sessions, transactions, conversations
Every request self-contained?Yes — carries all auth & contextNot necessarily — may rely on prior state
ScalabilityEasy — any server can handle any requestHarder — needs sticky sessions or shared state
CachingEasy — GET responses are cacheableHard — stateful responses can't be cached

GET /orders/42
Authorization: Bearer abc123

GET /orders/42/items
Authorization: Bearer abc123          ← sent again

// SOAP (stateful example) — session initiated once
POST /BankService — login → session id

Why REST chose stateless: it scales trivially. Put 10 app servers behind a load balancer, any can handle any request — no sticky sessions needed. SOAP's statefulness creates complexity (session replication, affinity, failover) that REST avoids entirely.

Trade-off: REST forces clients to send auth + context on every call (slightly more bandwidth); SOAP's stateful transactions are cleaner to express in code but much harder to scale. REST's simplicity won for modern web.

Question 49

What is an SDK in the context of APIs?

An SDK (Software Development Kit) is a language-specific library that wraps an API. Instead of crafting HTTP requests by hand, you call typed methods that feel native to your programming language.

// without an SDK — raw HTTP

  method: "POST",
  headers: {
    "Authorization": "Bearer sk_test_...",
    "Content-Type":  "application/x-www-form-urlencoded",
  },
  body: new URLSearchParams({ amount: "1000", currency: "usd", source: "tok_..." }),
});

if (!res.ok) throw new Error(charge.error.message);

  amount: 1000, currency: "usd", source: "tok_...",
});

What SDKs do beyond HTTP:

  • Authentication handling — inject tokens automatically.
  • Retries with exponential backoff — handle transient failures.
  • Rate-limit awareness — respect Retry-After.
  • Pagination helpers — auto-iterate through pages.
  • Type safety — compile-time checks for request/response shapes.
  • Error classes — typed exceptions (RateLimitError, AuthError) instead of raw HTTP codes.
  • Idempotency — auto-generate keys for retries.

SDK vs API: the API is the HTTP contract (what requests the server accepts); the SDK is one language's convenient wrapper around it. Multiple SDKs (JS, Python, Go, etc.) all talk to the same API. Many companies generate their SDKs from OpenAPI specs with openapi-generator — one source, many languages.

Question 52

What is Postman, and how is it used?

Postman is the most popular GUI for testing and exploring APIs. You type in a URL, headers, and body; click Send; see the response. It's much friendlier than raw curl for manual testing and saves your work as reusable "collections."

Key features:

  • Requests. Build any HTTP request with a form UI — method, URL, headers, body, params.
  • Collections. Group related requests (e.g., "User API," "Orders API") and share with your team.
  • Environments. Variables for dev/staging/prod ({{baseUrl}}, {{apiKey}}) so you can switch without editing URLs.
  • Tests. Write JS snippets that assert on responses (pm.expect(pm.response.code).to.equal(200)).
  • Collection Runner. Run a whole collection against an environment — lightweight integration testing.
  • Mock servers. Host fake responses from your collection.
  • Documentation. Auto-generate browsable docs from a collection.
  • Code generation. Copy a request as curl, fetch, Python, etc.

Common usage pattern:

  1. Add an auth request that saves the returned token: pm.environment.set("token", pm.response.json().accessToken).
  2. Other requests reference {{token}} in the Authorization header.
  3. Chain: run auth → run protected endpoints → verify responses.

Alternatives in 2026: Bruno (file-based, git-friendly, offline-first), Insomnia, Hoppscotch (open-source, web-based). Postman's push toward cloud accounts has driven many teams to Bruno; pick based on whether your requests need to live in git alongside code.

Question 54

What does the 401 status code indicate?

401 Unauthorized means "you haven't proven who you are." Despite the confusing name, it's really about authentication, not authorization.

// no token at all
GET /api/me

HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer realm="api"
{ "error": "Authentication required" }

// invalid or expired token
GET /api/me
Authorization: Bearer abc.def.xyz

HTTP/1.1 401 Unauthorized
{ "error": "Token expired" }

When 401 is the right answer:

  • No Authorization header on a protected endpoint.
  • Invalid signature on a JWT.
  • Expired token.
  • Revoked session.
  • Wrong API key.

Don't confuse 401 with:

  • 403 Forbidden — you're authenticated, but not allowed to do this action. "I know who you are; you can't do that."
  • 400 Bad Request — the request was malformed, independent of auth.
  • 404 Not Found — deliberately used by some APIs instead of 403, to avoid hinting that protected resources exist.

How clients should handle 401:

  • Try refreshing the access token (if you have a refresh token) and retry the request.
  • If the refresh also fails, clear credentials and redirect to login.
  • Never retry indefinitely — the creds are dead until the user intervenes.

HTTP's spec naming is historically messy: "401 Unauthorized" should really be called "Unauthenticated," but we're stuck with the name.

Question 55

What does the 403 status code mean?

403 Forbidden means "you're authenticated, but you're not allowed to do this." The server recognized you; it's refusing because the specific action isn't permitted for your account.

// you're logged in, but you tried to delete someone else's account
DELETE /users/9999
Authorization: Bearer <my valid token, user id 42>

HTTP/1.1 403 Forbidden
{
  "error": "You don't have permission to delete this user",
  "code": "INSUFFICIENT_PRIVILEGES"
}

Common 403 scenarios:

  • Wrong role. A regular user trying to hit an admin-only endpoint.
  • Wrong resource owner. Trying to edit another user's post.
  • Missing permission/scope. OAuth token doesn't have the required scope.
  • Geographic restrictions. "Sorry, not available in your region."
  • IP allowlist. Internal admin API blocks traffic from outside the office.

401 vs 403 in one sentence:

  • 401 — "I don't know who you are; please authenticate." → Client should send credentials (or redirect to login).
  • 403 — "I know who you are, but no." → Client should show a "not permitted" UI; re-authenticating won't help.

Security consideration: for very sensitive resources (e.g., accessing other tenants' data), some APIs return 404 instead of 403 to avoid confirming that the resource exists. This is called obscurity through status codes and tradeoffs against clarity, but it's common in strict environments.

Question 56

What is a RESTful resource?

In REST, a resource is the core concept — it's any "thing" in your system that has a URL and can be acted on: a user, an order, a photo, a comment. Resources are modeled as nouns, never verbs.

// good — nouns
GET    /users
POST   /orders
DELETE /photos/42

// bad — verbs in the URL
GET    /getAllUsers        ← let the HTTP method describe the action
POST   /createOrder
POST   /deletePhotoById/42

Properties of a RESTful resource:

  • Identified by a URL. Every resource has a unique URI: /users/42.
  • Uniform interface. The same HTTP methods apply across all resources (GET reads, DELETE removes) — predictability across the API.
  • Multiple representations. The same resource can be returned as JSON, XML, or HTML depending on Accept.
  • Can have sub-resources. /users/42/posts is the user's posts resource.

Resource vs endpoint: the resource is abstract ("a user"); an endpoint is a specific URL+method combination for acting on it (GET /users/42 fetches, DELETE /users/42 removes). Multiple endpoints belong to one resource.

Resource modeling tips: use plural nouns for collections (/users, not /user); use nesting sparingly (2 levels max: /users/:id/posts is fine, deeper gets painful); for non-CRUD operations that don't map to a verb cleanly, create a resource representing the action: /orders/42/refund, /emails/verify.

Question 57

What is an API contract?

An API contract is the formal agreement describing exactly what the API accepts and returns: endpoints, parameter types, status codes, response shapes, auth requirements. It's the shared "truth" between backend and frontend teams.

# an OpenAPI contract — machine-readable, language-agnostic
paths:
  /users/{id}:
    get:
      parameters:
        - name: id
          in: path
          required: true
          schema: { type: string, format: uuid }
      responses:
        "200":
          content:
            application/json:
              schema:
                type: object
                properties:
                  id:    { type: string }
                  email: { type: string, format: email }
                  name:  { type: string }
                required: [id, email, name]
        "404":
          description: User not found

Why contracts matter:

  • Parallel development. Frontend and backend teams build simultaneously — frontend against a mock, backend against the spec. They integrate painlessly if both honor the contract.
  • Generated clients. Type-safe SDKs in JS, Python, Go, etc. — generated from the same source.
  • Contract testing. Tools like Pact and Dredd verify that the real API matches the spec, catching drift.
  • Shared documentation. The spec doubles as docs — Swagger UI renders it browsably.

Common contract formats: OpenAPI (the REST standard), GraphQL schema, Protocol Buffers (for gRPC). Most modern teams use OpenAPI for REST APIs and commit the YAML alongside the code. When the contract changes, review becomes a code review — everyone sees the diff.

The key insight: the contract is the product. Code implements it; tests verify it; clients consume it. Without a contract, every API change is an integration surprise.

Question 58

What is the difference between an API and a web service?

"API" and "web service" overlap heavily, but they're not the same thing. Every web service is a kind of API, but not every API is a web service.

Think of it this way: "API" is the general concept — any interface one piece of software exposes to another. "Web service" is a specific kind of API: one that lives on a server and is called over a network.

// a library API — function calls, no network

// an OS API — system calls

Question 59

Explain the concept of RESTful architecture.

REST (Representational State Transfer) is an architectural style for designing networked applications. Roy Fielding described it in his 2000 PhD dissertation, and it became the dominant way to build web APIs.

The six architectural constraints:

  • Client-server. Clear separation of concerns: clients handle UI, servers handle data and logic. Either can evolve independently.
  • Stateless. Each request carries all information needed to process it. The server doesn't remember previous calls.
  • Cacheable. Responses declare their cacheability (Cache-Control, ETag). Clients and intermediaries cache where allowed.
  • Layered system. Clients can't tell whether they're talking to the origin server or a proxy/cache/load balancer. Enables scaling.
  • Uniform interface. Resources have URLs, actions use standard HTTP methods, responses use standard formats and status codes. Consistency across the whole API.
  • Code on demand (optional). Server can send executable code (JavaScript) to clients. Rarely used in practice.
// RESTful example — each request is self-contained
GET /orders/42 HTTP/1.1
Host: api.example.com
Authorization: Bearer abc123           # carries auth itself
Accept: application/json               # tells server the representation wanted

// server response includes caching directive
HTTP/1.1 200 OK
ETag: "abc123"
Cache-Control: max-age=60
Content-Type: application/json

Why REST won: it uses the existing web infrastructure — HTTP, URLs, standard methods, caches, load balancers. No new protocols, no custom tools. Any developer who knows HTTP can pick up any REST API in minutes. That accessibility is REST's biggest strength, and why it's the default choice for public APIs in 2026.

GET    /users/42/posts                  # sub-collection
POST   /users                           # create
GET    /users/42/posts?limit=20         # with query params
GET    /users/search?q=ada              # action via URL

Parts of a URI:

  • Scheme — https:// (always HTTPS in production).
  • Host — api.example.com (the server).
  • Version — /v1 (optional, URL-based versioning).
  • Path — /users/42 (identifies the resource).
  • Query string — ?include=posts&sort=newest (filters/options).
  • Fragment — #section (client-side only; APIs don't use this).

Design principles:

  • Use nouns, not verbs. /users, not /getUsers.
  • Plurals for collections. /users, not /user.
  • Lowercase with hyphens. /user-profiles, not /UserProfiles.
  • Short, hierarchical paths. /posts/42/comments is fine; /a/b/c/d/e is not.
  • Stable identifiers. Once published, don't change URIs — external clients break.

A good URI tells the reader what resource you're dealing with just by looking at it.

Question 61

What is the purpose of the PATCH method?

PATCH is for partial updates — you send only the fields you want to change, and the server leaves the rest alone. Contrast with PUT, which replaces the whole resource.

// existing user record
{ "id": 42, "name": "Ada", "email": "ada@old.com", "age": 36, "role": "user" }

// PATCH — update just the email
PATCH /users/42
Content-Type: application/json
{ "email": "ada@new.com" }

// result — only email changed; everything else preserved
{ "id": 42, "name": "Ada", "email": "ada@new.com", "age": 36, "role": "user" }

// PUT — would replace EVERYTHING
PUT /users/42
{ "email": "ada@new.com" }
// result: other fields wiped or set to defaults!

When to use PATCH:

  • Updating one field (change password, toggle a preference).
  • Bandwidth-constrained clients (mobile apps).
  • Avoiding accidental overwrites of fields you didn't mean to touch.
  • Concurrent editors — only the fields each user changed get sent.

PATCH body formats:

  • Simple JSON (most common) — just the fields to change: { "email": "new@example.com" }.
  • JSON Merge Patch (RFC 7396) — same as above, but null means "delete this field."
  • JSON Patch (RFC 6902) — explicit operations: [{"op":"replace","path":"/email","value":"new@..."}]. More verbose but supports precise changes like array inserts.

Idempotency: PATCH usually is idempotent (sending the same patch twice = same state), but not always ({ "counter": { "$inc": 1 } } isn't). Document clearly which endpoints are safe to retry.

Question 62

What is API throttling?

Throttling is a form of rate limiting that focuses on short-window burst control — "no more than X requests per second" — rather than long-window quotas ("1000 requests per day"). Its goal is to smooth traffic and protect downstream systems from sudden spikes.

// throttled — 10 req/sec max
// request 1-10 in second N: accepted
// request 11 in second N:   HTTP 429 Too Many Requests
//                           Retry-After: 1

// vs rate limit — 1000 req/hour
// any 1000 requests anywhere in the hour are accepted
// the 1001st is rejected until the hour window resets
ThrottlingRate limiting
Short windows (seconds)Long windows (minutes/hours/days)
Caps burst rateCaps total volume
Protects server from overloadProtects from abuse, enforces quotas
Typical: 10-100 req/secTypical: 1000 req/hour

Implementation algorithms:

  • Token bucket — allows bursts up to the bucket size; refills at a steady rate. Feels natural to users.
  • Leaky bucket — queues requests and drains at a fixed rate. Smooths spikes at the cost of latency.
  • Sliding window — counts requests in the trailing N seconds. Accurate but more expensive.

In practice, you want both. A per-second throttle catches runaway loops; a per-hour rate limit catches abuse. Stack them: max 10 req/sec AND max 1000 req/hour. For serious protection, combine edge throttling (Cloudflare, AWS WAF) with app-level limits (express-rate-limit + Redis).

Question 63

What is API versioning, and what are the common strategies for it?

API versioning lets you evolve your API without breaking existing clients. Once an API is published, any breaking change — renamed field, removed endpoint, type change — needs a new version so old clients keep working.

Common versioning strategies:

Question 65

What is a JSON Web Token (JWT)?

A JWT (JSON Web Token) is a compact, URL-safe string that carries signed claims about a user or session. Clients typically get one at login and include it on every subsequent request.

// a JWT has three parts, separated by dots
eyJhbGciOiJIUzI1NiJ9 . eyJzdWIiOiI0MiIsIm5hbWUiOiJBZGEifQ . sflKxwRJSMeKKF2...
  ──── header ────     ─────── payload ───────             ── signature ──

// decoded — header and payload are just base64url-encoded JSON
header:  { "alg": "HS256", "typ": "JWT" }
payload: { "sub": "42", "name": "Ada", "iat": 1713500000, "exp": 1713500900 }
signature: HMACSHA256(base64url(header) + "." + base64url(payload), SECRET)

Key properties:

  • Self-contained. All the claims (user id, role, expiry) are in the token — server can validate without a database lookup.
  • Signed, not encrypted. Anyone can read the payload; only the server can create or verify signatures (unless encrypted with JWE).
  • Stateless. No server-side session store needed.
  • Tamper-proof. Changing any character invalidates the signature.

Standard claims (RFC 7519): sub (subject / user id), iat (issued at), exp (expires at), iss (issuer), aud (audience), jti (token id for revocation).

Gotchas:

  • JWTs are not encrypted. Don't put passwords or secrets inside — attackers can base64-decode.
  • No easy revocation. Short expiry (15 min) + refresh tokens + Redis blocklist for immediate revocation.
  • Never use alg: none. Old vulnerability — always enforce a specific algorithm server-side.
  • Never store in localStorage — XSS-exposed. Use HttpOnly cookies.

JWTs are the default for stateless auth in modern APIs. For simpler apps, consider server-side sessions with Redis — easier revocation, smaller tokens.

Question 66

How do you secure an API?

Securing an API is layered — no single measure is enough. Real-world attacks target the weakest link.

The stack, from outside in:

  • HTTPS everywhere. All traffic encrypted; redirect HTTP → HTTPS; HSTS header. No plain HTTP, ever.
  • Edge layer (Cloudflare, AWS WAF) — blocks obvious attacks (SQLi probes, DDoS, known bot networks) before hitting your servers.
  • Authentication — every non-public endpoint requires valid credentials (JWT, OAuth, API key).
  • Authorization — check ownership/permissions on every request; never trust the client's stated identity.
  • Input validation. Schema-validate every input with Zod, Joi, or AJV. Reject unknown fields (mass assignment defense).
  • Parameterized queries. SQL injection only happens with string concatenation — always use $1, ? parameters.
  • Rate limiting. Throttle per-IP and per-user; especially tight on auth endpoints.
  • CORS with an allowlist (never * + credentials).
  • Security headers via Helmet — CSP, HSTS, X-Content-Type-Options, etc.
  • Secrets management. Env vars, AWS Secrets Manager, Vault — never commit keys to git.
  • Dependency hygiene. npm audit, Dependabot, Renovate — most breaches are outdated libraries.
  • Logging & monitoring. Log auth failures, 4xx spikes, new user agents; alert on anomalies.
  • Principle of least privilege. Tokens with narrow scopes; DB users with minimal grants; no "admin everywhere" accounts.

Reference: The OWASP API Security Top 10 lists the most common API-specific vulnerabilities — broken authentication, broken object-level authorization, excessive data exposure, mass assignment, etc. Every API developer should read it.

Security is not a feature you add — it's a mindset you maintain in every commit.

Question 67

What is an API call?

An API call is any individual request made to an API. It's the unit of interaction between a client and an API — one request → one response.

// each of these is an API call
GET  https://api.github.com/users/torvalds           → 1 call
POST https://api.stripe.com/v1/charges               → 1 call
GET  https://api.openai.com/v1/chat/completions      → 1 call

What happens during an API call:

  1. Client builds an HTTP request (method, URL, headers, optional body).
  2. DNS resolves the hostname (api.example.com → IP address). Usually cached.
  3. TLS handshake establishes an encrypted connection (reused for subsequent calls).
  4. Request is sent over the network.
  5. Server receives, routes, authenticates, authorizes, processes, and responds.
  6. Response flows back; client parses status, headers, body.

Why "API call" matters in practice:

  • Billing. Many APIs charge per call (OpenAI: per tokens per call; Stripe: per operation).
  • Rate limits. Measured in "calls per minute/hour" — each request counts as one call.
  • Performance. Fewer calls = faster apps. Batch operations (one call creating 100 records) beat 100 individual calls.
  • Monitoring. Metrics track "calls per second," "p99 latency per call," "error rate per call."

Minimizing calls: use bulk endpoints where possible; use GraphQL to fetch everything in one request; cache responses; batch asynchronously (debounce, throttle). Each API call has fixed overhead (DNS, TLS, network latency) — even a trivial call takes 50-200ms on the public internet. Reducing the number of calls often beats optimizing individual ones.

Question 68

What is the purpose of the Authorization header?

The Authorization header carries the credentials that prove who's making the request. It's how most API authentication happens in practice.

// Bearer tokens — the standard for modern APIs (JWT, OAuth)
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.eyJzdWIi...

// API keys — server-to-server
Authorization: Bearer sk_live_abc123xyz

// Basic auth — username:password base64-encoded (still used for simple tools)
Authorization: Basic YWxhZGRpbjpvcGVuc2VzYW1l          # → aladdin:opensesame

// Digest auth — older challenge-response (rare in modern APIs)
Authorization: Digest username="Mufasa", realm="...", ...

// custom schemes — company-specific, e.g., AWS SigV4
Authorization: AWS4-HMAC-SHA256 Credential=..., Signature=...

Why a dedicated header:

  • Standard location. Every HTTP library, proxy, and logging tool knows where to find credentials — and where to redact them.
  • Not in the URL. URLs end up in logs, browser history, referrer headers — leaking credentials. Headers are safer.
  • Survives redirects carefully. Good HTTP clients strip Authorization on cross-origin redirects to prevent credential leaks.

The scheme token (Bearer, Basic, etc.) tells the server how to interpret the value. Bearer means "this token alone grants access" — no challenge, no username/password exchange. The name comes from "whoever bears this token has access."

Critical rules: always over HTTPS (header is plaintext otherwise); never log the raw value (redact to ***); reject requests missing or malformed. 401 responses typically include a WWW-Authenticate header hinting at the expected scheme: WWW-Authenticate: Bearer realm="api".

Question 69

What is an API response?

An API response is the server's answer to a request. Every API call gets exactly one response — even if the request is malformed or unauthorized, you still get a response (with an error status).

HTTP/1.1 200 OK                                     ← status line
Content-Type: application/json; charset=utf-8       ← headers
Content-Length: 54
Date: Sun, 19 Apr 2026 12:00:00 GMT
ETag: "abc123"
Cache-Control: public, max-age=60

{ "id": 42, "name": "Ada", "role": "admin" }        ← body

Three parts, parallel to a request:

  • Status line — HTTP version + status code + reason phrase (200 OK, 404 Not Found).
  • Headers — metadata: content type, caching, CORS, server info, rate-limit counters.
  • Body — optional payload (JSON, HTML, binary). Empty for 204 or 304.

What clients look at first:

  1. Status code — tells whether the request succeeded, what went wrong, whether to retry.
  2. Content-Type — how to parse the body.
  3. Rate-limit headers (RateLimit-Remaining) — pace future calls.
  4. ETag / Last-Modified — for caching and conditional requests.
  5. Body — the actual data.

Good API responses are consistent:

  • Same error shape across all endpoints ({ error: { code, message } }).
  • Always include a request id for support.
  • Match the HTTP status to the actual result.
  • Never leak stack traces or internal details.
  • Include all relevant rate-limit headers.

Inconsistent responses (different error shapes per endpoint, wrong status codes, mixing HTML and JSON) are the single biggest complaint developers have about APIs.

Question 70

Explain the difference between 4xx and 5xx status codes.

Both 4xx and 5xx are errors, but they assign blame to different parties.

The client sent a bad requestThe server failed to handle a valid request
Fixing requires client changeFixing requires server-side work
Retrying the same request won't helpRetrying might help (with backoff)
Not a server bug; not alerted onA server bug; triggers alerts
400 Bad Request, 401, 403, 404, 409, 422, 429500 Internal Error, 502 Bad Gateway, 503 Unavailable, 504 Gateway Timeout
// 4xx — client did something wrong
POST /users
{ "email": "not-an-email" }
→ 400 Bad Request
  { "error": "email must be a valid email address" }

// 5xx — server failed unexpectedly
GET /users/42
→ 500 Internal Server Error
  { "error": "Something went wrong", "reqId": "req_abc" }

Operational implications:

  • Monitoring: 5xx errors are usually alerts (your app is broken). 4xx errors are information (you're being probed, or a client has bugs). A spike in 401s often indicates a credential-stuffing attack.
  • Retries: client libraries retry 5xx with exponential backoff; they don't retry most 4xx (except 429 "too many requests").
  • Logging: log 5xx with full stack trace; log 4xx with just the request summary.
  • SLA impact: 5xx counts against your uptime SLA; 4xx doesn't (the client caused the error).

Common confusion: is a validation failure a 400 or 422? Both are valid — 400 for malformed (can't even parse), 422 for well-formed but semantically wrong ("email already taken"). Pick one style and stay consistent.

Question 71

What is a RESTful web service?

A RESTful web service is a web API built following REST architectural principles. It exposes resources over HTTP, uses standard HTTP methods for actions, and returns representations (usually JSON).

// a small RESTful web service for a library
GET    /books              # list
GET    /books/42           # one book
POST   /books              # add new book
PUT    /books/42           # replace book 42
PATCH  /books/42           # update fields
DELETE /books/42           # remove

GET    /books/42/reviews   # sub-resource

What makes a web service "RESTful":

  • Resources have URLs. Every entity you can access has a stable URI.
  • HTTP methods carry meaning. GET reads, POST creates, DELETE removes — you don't invent your own verbs.
  • Stateless. Each request is self-contained; no server-side session needed.
  • Standard status codes. 200, 201, 400, 404, 500 — not custom or reused incorrectly.
  • Multiple representations. JSON by default; XML, HTML possible.
  • Cacheable. Responses declare what can be cached.

Historical context: "Web service" was the 2000s term when SOAP-based services dominated. As REST won out in the 2010s, people started saying "REST API" or "RESTful API" more often — but "RESTful web service" still appears in textbooks and enterprise codebases.

In practice, these terms are interchangeable:

  • RESTful web service
  • REST API
  • RESTful API
  • HTTP API (when being pedantic about true REST compliance)

They all mean: a server that speaks HTTP, organizes data as resources, and uses standard verbs. That's the web-service API of 2026.

Question 72

What are the principles of REST?

REST is defined by six architectural constraints from Roy Fielding's dissertation. A service that follows all six is "fully RESTful"; most real APIs follow 4-5 and still call themselves REST.

  1. Client-Server. Clear separation: client handles UI and state; server handles data and business logic. Each can evolve independently.
  2. Stateless. Each request carries all context (auth, parameters). The server does not remember previous requests. Scales trivially — any server can handle any request.
  3. Cacheable. Responses declare whether they can be cached (Cache-Control, ETag). Clients and intermediaries cache to reduce load and latency.
  4. Uniform Interface. Consistency across the API:
    • Resources identified by URIs (/users/42).
    • Manipulated through representations (JSON).
    • Self-descriptive messages (status codes, content types).
    • HATEOAS (responses include links to related resources).
  5. Layered System. Clients can't tell if they're talking to the origin server or a cache/proxy/load balancer. Enables horizontal scaling, security boundaries, and gradual migration.
  6. Code on Demand (optional). Servers can deliver executable code (JavaScript) to clients. Rarely used; the only optional constraint.

Practical 2026 reality: most APIs follow 1-5 but skip HATEOAS (part of "uniform interface"). Strict Fielding purists call this REST-ish; the rest of the industry just says "REST."

Why these constraints won: they describe how the web itself already worked. Adopting them gets you scalability, caching, and interoperability for free — using infrastructure already deployed everywhere (browsers, CDNs, proxies). That fit with existing systems is REST's killer feature.

Question 73

What is an API proxy?

An API proxy is a lightweight layer that sits in front of an API, forwarding requests but adding or adjusting behavior. Simpler than a full API gateway, which does more (routing, aggregation, transformations).

client  ──▶  API Proxy  ──▶  real API
              ├─ adds authentication
              ├─ logs the request
              ├─ caches the response
              └─ transforms headers

Common proxy uses:

  • Hide API keys from the browser. Your frontend calls your proxy; the proxy adds the real API key server-side before forwarding. Users never see the key.
  • Add authentication. Proxy validates tokens for unauthenticated backends.
  • Rate limiting. One place to throttle external API usage across multiple clients.
  • Request/response transformation. Fix a third-party API's ugly response shape; add headers; rewrite paths.
  • Caching. Cache expensive upstream calls.
  • CORS bypass. Your server calls the third-party API from a same-origin proxy, avoiding browser CORS restrictions.
  • Logging & monitoring. Observe all traffic in one place.
// Node.js example — proxy weather API with caching
app.get("/weather/:city", async (req, res) => {
  const cached = cache.get(req.params.city);
  if (cached) return res.json(cached);

  const r = await fetch(`https://api.openweather.com/${req.params.city}`, {
    headers: { "API-Key": process.env.OPENWEATHER_KEY },
  });
  const data = await r.json();
  cache.set(req.params.city, data, 300);  // 5 min TTL
  res.json(data);
});

Proxy vs gateway: a proxy typically fronts one API with simple logic; a gateway fronts many APIs with routing, auth, and orchestration. For small needs, a Node proxy is enough; for multi-service architectures, reach for Kong, KrakenD, or AWS API Gateway.

Question 74

How do you implement caching in APIs?

Caching in APIs comes in layers — from the browser to the edge to the server. Each layer saves bandwidth, reduces latency, and shields your server from load.

Key HTTP caching headers:

  • Cache-Control — public, max-age=60 (cache 60s anywhere), private (browser only), no-store (don't cache at all).
  • ETag — fingerprint of the response. Client sends it back as If-None-Match; server returns 304 if unchanged.

Redis caching pattern:

if (cached) return res.json(JSON.parse(cached));

const user = await db.user.findUnique({ where: { id } });
await redis.setex(`user:${id}`, 60, JSON.stringify(user));   // 60s TTL
res.json(user);

Cache invalidation — the hard part. When data changes, delete the cache entry (redis.del(...)); or use short TTLs and tolerate brief staleness. Mixing strategies (CDN + Redis + HTTP headers) is common but tricky to debug — add logs that show cache hits/misses.

Question 75

What is a RESTful API endpoint?

A RESTful API endpoint is a specific HTTP method + URL combination exposed by a REST API. Each endpoint does one thing — fetch a resource, create one, update one, delete one.

// each of these is a distinct endpoint
GET    /users              # list users            (method: GET    + path: /users)
POST   /users              # create a user         (method: POST   + path: /users)
GET    /users/42           # fetch user 42         (method: GET    + path: /users/:id)
PATCH  /users/42           # update user 42        (method: PATCH  + path: /users/:id)
DELETE /users/42           # delete user 42        (method: DELETE + path: /users/:id)

Characteristics of a good RESTful endpoint:

  • Noun-based URL. /users, not /getUsers. Let the method carry the verb.
  • Resource-oriented. Corresponds to one entity or collection.
  • Predictable. Same shape across the API (/users, /orders, /products all work the same).
  • Uses the right HTTP method. GET = read, POST = create, PUT = replace, PATCH = update, DELETE = remove.
  • Returns proper status codes. 200/201/204 for success, 400/401/403/404 for client errors, 500 for server errors.
  • Consistent error format. Clients can parse one error shape across all endpoints.
  • Documented. OpenAPI spec describes inputs, outputs, and errors precisely.

Anti-pattern examples (not RESTful):

  • POST /getUser?id=42 — should be GET /users/42.
  • GET /users/42/delete — should be DELETE /users/42.
  • POST /users/delete-many — acceptable for bulk ops, but prefer DELETE with a filter if possible.
  • Returning 200 with { error: "..." } — should be 400/404/500 with a proper body.

The endpoint is the touchable surface of your API. Every endpoint lives forever once shipped (versioning to change), so design thoughtfully.

Question 76

What are the benefits of using RESTful APIs?

RESTful APIs dominate the web because they hit a sweet spot of simplicity, scalability, and compatibility with existing infrastructure.

Key benefits:

  • Uses existing infrastructure. HTTP, URLs, status codes, caching, load balancers, CDNs, proxies — all already deployed worldwide. No new protocols, no new tools.
  • Language-agnostic. Any language with an HTTP library (all of them) can call a REST API. Your Python script talks to a Node backend written by strangers. That interoperability is unmatched.
  • Easy to learn. If you know HTTP, you know 80% of REST. No special toolchain to install.
  • Stateless = scalable. Any server can handle any request — add more servers behind a load balancer, done. No session replication, no sticky sessions.
  • Cacheable. GET responses work with browser caches, CDNs, reverse proxies out of the box — huge performance wins for free.
  • Self-documenting (with OpenAPI). The same spec generates docs, typed clients, and contract tests.
  • Mature ecosystem. Every framework (Express, Rails, Django, Spring, FastAPI) has first-class REST support.
  • Good developer tooling. Postman, curl, Bruno, browsers' DevTools — you can inspect and debug easily.

Trade-offs to be aware of:

  • Over/under-fetching. REST endpoints return fixed shapes; you might get fields you don't need or need multiple round-trips for a view. GraphQL fixes this for complex frontends.
  • Not ideal for real-time. REST is request-response; WebSockets or SSE suit live updates.
  • No formal schema by default. You need OpenAPI or similar for strong typing — it's optional, not built-in.

In 2026, REST is still the default for new APIs. Reach for alternatives (GraphQL, gRPC, tRPC) when you hit REST's specific limits — not before.

Question 77

What is a microservice?

A microservice is a small, independent service that owns one piece of a larger system. Instead of one big monolithic application handling everything (users + orders + payments + notifications), you split it into separate services that communicate over the network, usually via HTTP APIs.

// monolith — one big app
┌──────────────────────────────────┐
│  MyApp                           │
│  ├─ User logic                   │
│  ├─ Order logic                  │
│  ├─ Payment logic                │
│  └─ Notification logic           │
└──────────────────────────────────┘

// microservices — each concern is its own service
┌─────────────┐  ┌──────────────┐  ┌─────────────┐  ┌──────────────────┐
│ user-service│  │orders-service│  │payment-svc  │  │notification-svc  │
└─────────────┘  └──────────────┘  └─────────────┘  └──────────────────┘
        ↑                 ↑                ↑                 ↑
        └─────────── API Gateway / service mesh ─────────────┘

Typical properties:

  • Single responsibility. Each service owns one domain (users, orders, billing).
  • Own database. Services don't share DB tables — they talk via APIs.
  • Independently deployed. Update user-service without redeploying the whole system.
  • Technology choice per service. One might be Node, another Python, another Go — pick the best tool for each job.
  • Horizontally scalable. Scale payment-service separately if it's the bottleneck.

The trade-off: microservices solve scaling and team-organization problems but introduce new ones — distributed tracing, network failures, data consistency across services, operational complexity. For small teams and simple apps, a well-structured monolith is usually better. Move to microservices when you hit real scaling or team-coordination pain, not because it's trendy.

Question 78

How does API design differ in microservices architecture?

API design in microservices differs from monolithic APIs in a few key ways because there are now two audiences: external clients (browsers, mobile apps) and other services (internal microservice-to-microservice calls).

Monolith APIMicroservices API
One API surfaceMany internal APIs + one external facade
Callers share memory/DBEvery call crosses the network
Simple auth (session)Service-to-service auth (mTLS, JWT, API keys)
ACID transactionsEventual consistency, sagas, compensation
One error handlerPer-service errors + aggregation

Key microservice API patterns:

  • API Gateway. One external-facing entry point routes to internal services; handles auth, rate limits, aggregation. Clients don't know the internal topology.
  • Backend for Frontend (BFF). Per-client gateways (web BFF, mobile BFF) that stitch multiple internal services into one response tailored for that client.
  • Service discovery. Services find each other by logical name (http://payment-service), not IP. Kubernetes DNS, Consul, or service meshes handle this.
  • Resilience patterns. Circuit breakers (don't hammer a failing service), retries with exponential backoff, timeouts at every layer, bulkheads.
  • Async messaging. For non-critical work, use message queues (RabbitMQ, Kafka, SQS) instead of sync HTTP — decouples services.
  • Distributed tracing. Thread a trace-id across every hop (OpenTelemetry); debug issues that span 5+ services.

Critical rule: every network call can fail. Code defensively — timeouts, retries, fallbacks, circuit breakers. What works in a monolith (just call the function) becomes a reliability problem across services. Libraries like Axios, got, or service meshes (Istio, Linkerd) bake in these patterns.

Question 79

What is an API schema?

An API schema is the formal description of an API's structure: what endpoints exist, what parameters they take, what responses they return, and what types the data uses. It's the "shape" of the API, written in a machine-readable format.

# OpenAPI schema fragment describing a user endpoint
paths:
  /users/{id}:
    get:
      parameters:
        - name: id
          in: path
          required: true
          schema: { type: string, format: uuid }
      responses:
        "200":
          content:
            application/json:
              schema: { $ref: "#/components/schemas/User" }

components:
  schemas:
    User:
      type: object
      required: [id, email]
      properties:
        id:    { type: string, format: uuid }
        email: { type: string, format: email, maxLength: 254 }
        name:  { type: string }
        age:   { type: integer, minimum: 0, maximum: 150 }

Common schema formats by API type:

  • REST — OpenAPI (formerly Swagger), using JSON Schema internally.
  • GraphQL — GraphQL Schema Definition Language (SDL): built-in, required.
  • gRPC — Protocol Buffers (protobuf).
  • SOAP — WSDL (XML-based).
  • JSON APIs generally — JSON Schema for payload validation.

Why schemas matter:

  • Documentation. Swagger UI renders your schema as browsable docs — no separate docs to maintain.
  • Client SDK generation. One schema → typed clients for every language.
  • Server stubs. Generate route definitions and typed handlers from the schema.
  • Runtime validation. Reject requests that violate the schema (AJV, Zod, Ajv middleware).
  • Contract testing. Verify that real requests/responses match the schema (Dredd, Schemathesis).
  • Mocking. Generate fake servers from the schema (Prism).

A schema is the contract between API and clients. Writing it first (spec-first development) or extracting it from code (code-first) are both viable — but skipping it means every change is a surprise.

Question 80

What is the role of middleware in APIs?

Middleware in APIs is code that runs between the incoming HTTP request and the final handler. Each piece of middleware inspects, modifies, or short-circuits the request — then passes control to the next. Think of it as a pipeline.

// typical Express middleware pipeline
incoming request
      ↓
[CORS middleware]        → sets CORS headers
      ↓
[JSON parser]            → parses the body into req.body
      ↓
[Rate limit]             → rejects if too many requests
      ↓
[Auth middleware]        → validates token, sets req.user
      ↓
[Validation middleware]  → schema-checks req.body
      ↓
[Your route handler]     → actual business logic
      ↓
[Error handler]          → catches any throws
      ↓
response

What middleware typically handles:

  • Logging — record method, URL, status, response time for every request.
  • Authentication — validate tokens; populate req.user.
  • Authorization — check roles/permissions before the handler.
  • Body parsing — turn raw bytes into req.body (JSON, form data, files).
  • Validation — reject malformed input with 400 before logic runs.
  • CORS — set the right headers for browser cross-origin access.
  • Rate limiting — count requests; reject over-limit ones.
  • Compression — gzip/brotli responses.
  • Error handling — catch and format exceptions into consistent error responses.
// simple Express middleware
function logger(req, res, next) {
  console.log(`${req.method} ${req.url}`);
  next();                // → pass to the next middleware
}
app.use(logger);

Why middleware is powerful: cross-cutting concerns (auth, logging, validation) stay out of your handlers. Each handler focuses on business logic; the pipeline handles everything else. Most popular frameworks (Express, Koa, Fastify, FastAPI, Django) are built around middleware.

Question 81

What is the difference between SOAP and REST in terms of flexibility?

REST is dramatically more flexible than SOAP — which is exactly why it replaced SOAP for most modern APIs.

GET /users/42
{ "id": 42, "name": "Ada", "email": "ada@...", "twoFactorEnabled": true }
//                                              ↑ new field — old clients simply ignore it

Question 82

How do you handle large payloads in APIs?

Large payloads — either in requests (bulk imports) or responses (big datasets) — can overwhelm servers if handled naively. The right strategy depends on size, structure, and how often it happens.

SizeApproach
< 100 KBHandle normally; cap body size at 1 MB default
100 KB - 10 MBPaginate responses; raise request body limit on specific routes
10 MB - 1 GBStream NDJSON / CSV; compression (gzip, brotli)
> 1 GBObject storage (S3) with pre-signed URLs; async job + webhook
// pagination — chunked retrieval of large collections
GET /events?limit=100&cursor=abc
→ { "data": [...100 items], "nextCursor": "def" }

// streaming NDJSON response — constant memory regardless of size
res.setHeader("Content-Type", "application/x-ndjson");
for await (const row of db.query.stream()) {
  res.write(JSON.stringify(row) + "\n");
}
res.end();

// pre-signed S3 URL for large uploads — bytes never touch your server
POST /uploads/request
→ { "url": "https://s3...", "key": "uploads/abc123" }
// browser PUTs the file directly to S3

Key techniques:

  • Paginate collections. Never return 10,000 items in one response — use cursor-based pagination.
  • Compress. Accept-Encoding: gzip, br — JSON compresses 70-90%. Cloudflare/Nginx can do this at the edge.
  • Stream. For exports, write rows to the response as you read them from the DB — no memory blow-up.
  • Offload to object storage. File uploads/downloads go to S3/R2 directly — your API just signs URLs.
  • Async for expensive work. Return 202 Accepted + a job URL; client polls or gets a webhook when ready.
  • Cap request body size. Even with all of the above, set a hard limit (express.json({ limit: "1mb" })) to prevent DoS.

The goal: server memory and request time don't grow with payload size. Design for that from the start.

Question 83

What is an API rate limit, and why is it important?

An API rate limit caps how many requests a client can make in a time window. It's one of the most important defenses an API has.

// response showing rate-limit state
HTTP/1.1 200 OK
RateLimit-Limit:     1000
RateLimit-Remaining: 847
RateLimit-Reset:     3600

// over the limit
HTTP/1.1 429 Too Many Requests
Retry-After: 60
{ "error": "Rate limit exceeded" }

Why rate limits matter:

  • Prevent abuse. Scrapers, bots, and attackers can send thousands of requests per second. Without limits, they take your API down for everyone.
  • Protect from buggy clients. A client stuck in a retry loop can DoS you accidentally. Rate limits stop the bleeding.
  • Fair sharing. One customer shouldn't be able to hog resources that affect everyone else on your platform.
  • Cost control. Every request costs CPU, memory, bandwidth, and (if you call paid APIs downstream) money. Limits bound that cost.
  • Monetization. Tiered plans: free users get 100 req/hour, paid plans get 10,000, enterprise gets unlimited. Rate limits differentiate plans.
  • Brute-force defense. Strict limits on /login slow password-guessing attacks.

Implementation quick guide:

  • Use express-rate-limit + Redis for multi-server apps.
  • Key by user id when authenticated; IP for public endpoints.
  • Return standard RFC 9331 headers (RateLimit-*) so well-behaved clients self-pace.
  • Include Retry-After on 429s.
  • Layer edge rate-limiting (Cloudflare, AWS WAF) in front for DDoS protection.

Skipping rate limits is one of the top mistakes in new APIs — they're easy to add upfront and painful to retrofit after an incident.

Question 84

What is the difference between SOAP and REST in terms of transport?

SOAP and REST use different approaches for how messages travel between client and server.

AspectRESTSOAP
ProtocolHTTP(S) onlyTransport-agnostic — HTTP, SMTP, TCP, JMS, MQ
Default port80 / 443Same, but can be any
Message formatPlain HTTP body (JSON, XML, whatever)SOAP envelope wrapping XML content

// REST — uses HTTP as HTTP was meant to be used
GET /users/42 HTTP/1.1

// SOAP — everything is POST; HTTP is just a tunnel
POST /UserService HTTP/1.1
Host: api.example.com
Content-Type: text/xml; charset=utf-8
SOAPAction: "http://example.com/GetUser"

  </soap:Header>

  </soap:Body>

Why this matters:

  • REST leverages HTTP. Methods, status codes, URLs, caching, CORS, headers — all part of the design. Every HTTP tool understands REST traffic.
  • SOAP tunnels through HTTP. HTTP is just transport; the real "protocol" is inside the SOAP envelope. HTTP caching doesn't help (every call is POST); browsers can't natively debug it.
  • SOAP can work over non-HTTP transports. Email (SMTP), message queues (MQ), TCP — rare in practice but real. This was the enterprise selling point in the 2000s.

REST's tight HTTP integration is a feature, not a limitation. The entire internet is optimized for HTTP traffic — REST rides that wave. SOAP's transport flexibility was a promise that rarely paid off; for new projects, pick REST (or gRPC if you need high-performance binary).

Question 85

How do you handle versioning in RESTful APIs?

Versioning in RESTful APIs is how you evolve the API without breaking existing clients. Once an API is public, changing it is like changing a public contract — you need a strategy.

// URL path (most common)
GET /v1/users/42
GET /v2/users/42        # new version alongside

// Header (cleaner URLs)
GET /users/42
Accept: application/vnd.myapi.v2+json

// Custom header
GET /users/42
X-API-Version: 2

// Query parameter (discouraged)
GET /users/42?version=2

// Subdomain (rare)
GET https://v2.api.example.com/users/42

What counts as a breaking change (needs a new version):

  • Removing a field from a response.
  • Renaming a field.
  • Changing a field's type (string → integer).
  • Adding a required request field.
  • Removing an endpoint.
  • Changing a status code for a case.

What's safe (no version bump needed):

  • Adding new optional response fields.
  • Adding new endpoints.
  • Adding optional request parameters with sensible defaults.
  • Loosening validation (accepting more input formats).

Deprecation workflow:

  1. Ship v2 alongside v1.
  2. Announce v1 deprecation: set Deprecation: true and Sunset: <date> headers on v1 responses.
  3. Email developers, update docs, publish migration guide.
  4. Monitor v1 usage; nudge remaining users.
  5. Remove v1 after the sunset date (usually 6-12 months later).

Best practices: pick one versioning strategy and stick with it; keep at most 2-3 versions live; use semantic versioning discipline (breaking = new major). GitHub's v3 REST API lived for over a decade before v4 (GraphQL) — stable APIs win trust.

Question 86

What is an API endpoint in the context of microservices?

In microservices, an API endpoint is the specific URL a service exposes for external or internal consumers. Each service owns a set of endpoints scoped to its domain.

// each service owns endpoints for its domain
user-service:
  GET    /users/{id}
  POST   /users
  PATCH  /users/{id}

orders-service:
  GET    /orders/{id}
  POST   /orders
  GET    /orders?customerId={id}

payments-service:
  POST   /payments/charge
  POST   /payments/refund
  GET    /payments/{id}

// externally — one gateway URL routes to the right service
https://api.example.com/users/42        → user-service
https://api.example.com/orders/42       → orders-service
https://api.example.com/payments/xyz    → payments-service

Two kinds of endpoints in microservices:

  • External endpoints — exposed through the API gateway; public contract for clients (browsers, mobile apps, partners).
  • Internal endpoints — service-to-service calls; private, behind the network perimeter; use mTLS or internal JWTs.

Design considerations specific to microservices:

  • Service boundaries. An endpoint on the user-service shouldn't look up order data — that crosses service boundaries. The gateway (or BFF) aggregates.
  • Versioning. Each service versions independently. Coordinating cross-service changes needs care.
  • Service discovery. Services find each other by logical name (http://user-service), not IP. Kubernetes DNS handles this.
  • Health checks. Every service exposes /health and /ready for orchestrators.
  • Observability. Every endpoint emits traces, metrics, structured logs with a trace-id.
  • Resilience. Every call to another service has timeouts, retries, circuit breakers.

A common pattern: the gateway rewrites external URLs (/api/users/42) to internal ones (http://user-service/users/42). Clients see a clean, versioned public API; internally, services evolve freely.

Question 87

What is the role of an API gateway in microservices architecture?

An API gateway in microservices is the single entry point for all external traffic. It sits in front of your services, handling the cross-cutting concerns so individual services don't each reinvent them.

                       ┌────────────────────┐
                       │   API Gateway      │
external clients  ────▶│ • auth             │──▶ user-service
(browsers, apps)       │ • rate limiting    │──▶ orders-service
                       │ • TLS termination  │──▶ payments-service
                       │ • routing          │──▶ notifications-service
                       │ • logging          │
                       │ • caching          │
                       │ • aggregation      │
                       └────────────────────┘

Core roles of the gateway:

  • Routing. URL path decides which service handles the request (/users/* → user-service).
  • Authentication. Validate tokens once at the edge; pass a verified identity header downstream. Services don't duplicate auth logic.
  • Authorization. Coarse-grained policy checks (role, plan tier) before touching services.
  • Rate limiting. One consistent policy across all services.
  • TLS termination. Internal services can run plain HTTP; gateway handles certs.
  • Request/response transformation. Aggregate multiple service calls (BFF pattern); translate protocols (gRPC ↔ REST).
  • Caching. Cache GET responses at the edge.
  • Logging & metrics. One place to instrument every request with trace-ids.

Benefits:

  • Decouples clients from service topology. Split or merge services without client rewrites.
  • Consistent security. Every request passes through auth — no service can accidentally leak unauthenticated data.
  • Smaller services. Each focuses on its domain; cross-cutting concerns live centrally.
  • Easier for clients. Public API is one URL; clients don't discover services.

Popular gateways: Kong, KrakenD, Traefik, Envoy, AWS API Gateway, Google Cloud Endpoints, Azure API Management. For simpler needs, Nginx or a thin Node/Go service handles most of what a gateway does.

Question 88

What is the purpose of the Host header in HTTP requests?

The Host header tells the server which domain the client is trying to reach. It's mandatory in HTTP/1.1 — every request sent must include it. Without it, a server hosting multiple websites on the same IP wouldn't know which one to respond with.

GET /users/42 HTTP/1.1
Host: api.example.com           ← which virtual host?
User-Agent: curl/8.4.0
Accept: application/json

Why the Host header exists:

  • Virtual hosting. One web server at one IP can serve hundreds of domains. Host tells it which one. Without Host, api.example.com and app.example.com at the same IP would be indistinguishable.
  • Routing in load balancers and proxies. Nginx and cloud LBs inspect Host to decide which backend service to forward to.
  • TLS SNI pair. Host (in HTTP) + SNI (in TLS handshake) together let HTTPS servers pick the right certificate for the requested domain.
# Nginx — route based on Host header
server {
  listen 80;
  server_name api.example.com;
  location / { proxy_pass http://api-backend; }
}
server {
  listen 80;
  server_name app.example.com;
  location / { proxy_pass http://app-backend; }
}

Security note: apps must not blindly trust the Host header for building absolute URLs in responses (like password reset emails) — an attacker can send Host: evil.com and redirect victims. Use an allowlist or a configured canonical hostname. HTTP/2 and HTTP/3 replace Host with :authority pseudo-header — same purpose, different spelling.

Client libraries (fetch, axios, requests) set Host automatically from the URL; you only need to set it manually in rare cases like testing against a server with a different DNS name.

Question 90

What is a SOAP envelope?

A SOAP envelope is the XML wrapper around every SOAP message. It's the outermost element that contains the header (metadata) and body (actual request/response data).

<?xml version="1.0" encoding="UTF-8"?>
<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">
  <soap:Header>                              <!-- optional: auth, routing, transaction info -->
    <wsse:Security>
      <wsse:UsernameToken>
        <wsse:Username>ada</wsse:Username>
        <wsse:Password>***</wsse:Password>
      </wsse:UsernameToken>
    </wsse:Security>
  </soap:Header>

  <soap:Body>                                 <!-- required: the actual call payload -->
    <GetUser xmlns="http://example.com/">
      <Id>42</Id>
    </GetUser>
  </soap:Body>
</soap:Envelope>

Parts of the envelope:

  • Envelope — the outermost wrapper. Always present; names the SOAP version (1.1 or 1.2) via the namespace.
  • Header — optional; carries metadata like WS-Security credentials, transaction ids, routing hints.
  • Body — required; contains the actual request or response payload: the method being called and its parameters.
  • Fault — SOAP's error element (appears inside Body on errors). Has a fault code, reason, and detail.
<!-- SOAP error response -->
<soap:Envelope>
  <soap:Body>
    <soap:Fault>
      <faultcode>soap:Client</faultcode>
      <faultstring>User not found</faultstring>
      <detail><UserId>42</UserId></detail>
    </soap:Fault>
  </soap:Body>
</soap:Envelope>

Why the envelope exists: SOAP was designed as transport-agnostic, so the envelope packages everything needed to process the message — independent of whether it's sent over HTTP, SMTP, or MQ. It's also what makes SOAP verbose: every call carries this XML wrapping overhead, even for tiny payloads. REST avoided this entirely by using HTTP itself as the envelope.

Question 91

What is the role of the WSDL in SOAP APIs?

WSDL (Web Services Description Language) is the XML contract for a SOAP service. It describes every operation, its parameters, return types, and how to call it. Think of it as OpenAPI for SOAP — the machine-readable spec.

<?xml version="1.0"?>
<wsdl:definitions xmlns:wsdl="http://schemas.xmlsoap.org/wsdl/">

  <!-- types: define the data shapes -->
  <wsdl:types>
    <xs:schema targetNamespace="http://example.com/">
      <xs:element name="GetUser">
        <xs:complexType>
          <xs:sequence>
            <xs:element name="Id" type="xs:int"/>
          </xs:sequence>
        </xs:complexType>
      </xs:element>
    </xs:schema>
  </wsdl:types>

  <!-- operations: what the service can do -->
  <wsdl:portType name="UserService">
    <wsdl:operation name="GetUser">
      <wsdl:input  message="GetUserRequest"/>
      <wsdl:output message="GetUserResponse"/>
    </wsdl:operation>
  </wsdl:portType>

  <!-- binding: how operations travel over the wire -->
  <wsdl:binding ...> ... </wsdl:binding>

  <!-- service: where the endpoint lives -->
  <wsdl:service name="UserService">
    <wsdl:port binding="...">
      <soap:address location="http://example.com/UserService"/>
    </wsdl:port>
  </wsdl:service>
</wsdl:definitions>

What WSDL enables:

  • Code generation. Tools like wsimport (Java) or svcutil (.NET) read a WSDL and generate client stubs — typed classes and methods that call the service.
  • Discovery. Clients fetch the WSDL from the service URL (?wsdl) and learn all available operations.
  • Validation. Servers and clients validate requests/responses against the schema.
  • Documentation. WSDL is the authoritative reference for the SOAP service — what's possible and what formats are expected.

WSDL's strengths and weaknesses: it's extremely strict — breaking changes require regenerating every client's stubs. Perfect for formal enterprise contracts (banking, insurance); painful for APIs that evolve quickly. REST/OpenAPI's looser approach (add optional fields freely) is why REST won for modern APIs.

Question 92

What is the purpose of the Accept-Language header?

The Accept-Language header tells the server what languages the client prefers. Servers use it to return localized responses — error messages, UI strings, dates, and numbers formatted for the user's locale.

// client tells server "I prefer English, then German"
GET /api/errors
Accept-Language: en-US, en;q=0.9, de;q=0.8

// server returns localized error messages
{ "error": { "code": "INVALID_EMAIL", "message": "Invalid email address" } }

// or if de is picked
{ "error": { "code": "INVALID_EMAIL", "message": "Ungültige E-Mail-Adresse" } }

How the value works:

  • Language codes — en (English), en-US (American English), de (German), ja-JP (Japanese as used in Japan).
  • Quality values (q=0.8) rank preferences from 1.0 (best) down to 0.0 (not acceptable). If omitted, q=1.0.
  • Fallback. If the server can't match any preference, it serves its default (usually English).

What servers typically do:

  1. Parse the header, sort by q-value.
  2. Match against supported languages in priority order.
  3. Use the match to pick translation files (i18n).
  4. Return a Content-Language: de response header confirming what was used.
  5. Add Vary: Accept-Language so CDNs cache per-language variants correctly.
// Express example with i18next
import middleware from "i18next-http-middleware";
app.use(middleware.handle(i18next));       // populates req.t() based on Accept-Language

app.get("/welcome", (req, res) => {
  res.set("Content-Language", req.language);
  res.json({ greeting: req.t("welcome") }); // "Welcome" or "Willkommen"
});

Important notes: browsers send Accept-Language automatically based on the user's system language. APIs should respect it by default but let clients override (query param ?lang=fr or X-Language header). Without Vary: Accept-Language, caches serve the wrong language to different users — a classic i18n bug.

Question 93

What is an API proxy, and how is it used?

An API proxy is a server that forwards requests to an upstream API, typically adding cross-cutting behavior along the way. It's a thin intermediary — simpler than a full API gateway.

client  ──▶  your proxy  ──▶  real API
              (adds auth, caching, logging)

Common uses:

  • Hide API keys. Browser calls your proxy; proxy adds the real API key before forwarding. End users never see the secret.
  • CORS bypass. Third-party API doesn't support CORS? Proxy it from your server to avoid browser restrictions.
  • Rate limiting. One place to cap your upstream API usage across all your clients.
  • Caching. Cache expensive upstream calls (weather, pricing data).
  • Transform responses. Fix an ugly third-party response shape; add/remove fields.
  • Authentication. Require your own auth on proxied endpoints even if the upstream doesn't.
  • Analytics & logging. See all traffic to the upstream in one place.
// Node example — proxy weather API with caching + hide the API key
const cache = new Map();

app.get("/weather/:city", async (req, res) => {
  const key = req.params.city;
  const cached = cache.get(key);
  if (cached && cached.expiresAt > Date.now()) return res.json(cached.data);

  const r = await fetch(`https://api.openweather.com/data?q=${key}`, {
    headers: { "x-api-key": process.env.OPENWEATHER_KEY },      // server-side only
  });
  const data = await r.json();

  cache.set(key, { data, expiresAt: Date.now() + 5 * 60_000 });
  res.json(data);
});

Proxy vs gateway: a proxy fronts one upstream API with simple logic; a gateway fronts many services with routing, orchestration, and aggregation. For a single third-party API integration, a proxy is enough. For a multi-service architecture, use a full gateway (Kong, KrakenD, AWS API Gateway).

Question 94

How do you handle rate limiting in APIs?

Handling rate limiting has two sides: enforcing it on your own API, and respecting it when calling someone else's.

Enforcing (your API):

import rateLimit from "express-rate-limit";
import RedisStore from "rate-limit-redis";

const limiter = rateLimit({
  windowMs: 15 * 60 * 1000,                      // 15 minutes
  max: 100,                                       // 100 requests per IP/user
  standardHeaders: true,                          // RateLimit-* headers per RFC 9331
  store: new RedisStore({ sendCommand: (...a) => redis.call(...a) }),
  keyGenerator: (req) => req.user?.id ?? req.ip,
  handler: (req, res) => {
    res.set("Retry-After", "60");
    res.status(429).json({ error: "Rate limit exceeded" });
  },
});

app.use("/api", limiter);

Respecting (when you're a client):

async function callWithRateLimit(url, options, maxRetries = 5) {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    const res = await fetch(url, options);

    if (res.status !== 429) return res;

    // server tells us when to retry
    const retryAfter = Number(res.headers.get("Retry-After") ?? 60);
    const jitter = Math.random() * 500;                // avoid thundering herd
    await new Promise((r) => setTimeout(r, retryAfter * 1000 + jitter));
  }
  throw new Error("Rate limit retries exhausted");
}

Best practices for enforcing:

  • Use Redis for multi-server apps — in-memory counters diverge across instances.
  • Key by user id when authenticated; IP only for public endpoints.
  • Tighter limits on auth endpoints (/login: 5/min) to slow brute force.
  • Layer edge rate limits (Cloudflare, AWS WAF) for DDoS protection.
  • Return standard headers — well-behaved clients self-pace.

Best practices for respecting:

  • Honor Retry-After — don't guess.
  • Exponential backoff + jitter on retries.
  • Cache responses to reduce calls.
  • Batch requests when the API supports it.
  • Monitor your own remaining quota (RateLimit-Remaining).

Rate-limit-aware client libraries (AWS SDK, Stripe's SDK) do all this automatically.

Question 95

What is a RESTful API resource?

In REST, a resource is the fundamental unit of data the API exposes. Every URL points to a resource; every method acts on one. Designing your API starts by identifying resources.

// resources (nouns) with URLs
/users                # collection of users
/users/42             # one specific user
/users/42/posts       # that user's posts (sub-resource)
/posts/7/comments     # comments on a post
/orders/abc/items     # items in an order

Properties of a RESTful resource:

  • Noun-based. Never verbs: /users yes; /getUsers no.
  • Identified by URL. Each resource has a unique, stable URI.
  • Singular or collection. /users is the collection; /users/42 is an individual item.
  • Plural names for collections. /users, not /user.
  • Hierarchy for relationships. /users/42/posts — posts belonging to user 42.
  • Self-contained. The URL alone (with auth) should be enough to identify and operate on the resource.

Handling actions that don't fit CRUD nicely: sometimes you need to express things that aren't simple create/read/update/delete. Common patterns:

  • Model the action as a resource. POST /orders/42/refund creates a refund resource. POST /emails/verify creates an email-verification.
  • State transitions via PATCH. PATCH /orders/42 { "status": "cancelled" }.
  • Sub-resource for relationships. PUT /users/42/follows/99 to follow user 99.

Resource design rules of thumb: plural + lowercase + hyphens for URLs (/user-profiles); 2-3 levels of nesting max; avoid verbs in paths; keep identifiers stable forever (URIs are a public contract). Design resources first, endpoints second — the endpoints follow naturally from the resource model.

An API is the contract (endpoints + request/response shapes); an SDK is a language-specific library that wraps the API to make it easier to call from your code.

What SDKs add beyond raw HTTP:

  • Automatic authentication — inject keys/tokens.
  • Retries with backoff on transient failures.
  • Rate-limit awareness — honor Retry-After.
  • Pagination helpers — auto-iterate pages as an async iterator.
  • Type safety — compile-time checks for request/response shapes.
  • Typed errors — RateLimitError, AuthError instead of HTTP codes.
  • Idempotency keys generated automatically for safe retries.
  • Parsing & serialization — convert between wire format and language types.

Relationship: the API comes first — it's the source of truth. The SDK is a convenience layer. Most modern companies generate their SDKs from an OpenAPI or GraphQL schema, so the API and SDK stay in sync. If an API has no official SDK, you can often generate one with openapi-generator or Orval.

Question 98

What is a RESTful API server?

A RESTful API server is the server-side software that receives HTTP requests from clients, processes them, and returns responses — following REST principles.

// minimal REST API server in Node + Express
import express from "express";
const app = express();
app.use(express.json());

const users = new Map();              // imagine this is a database

app.get("/users/:id", (req, res) => {
  const user = users.get(req.params.id);
  if (!user) return res.sendStatus(404);
  res.json(user);
});

app.post("/users", (req, res) => {
  const id = crypto.randomUUID();
  users.set(id, { id, ...req.body });
  res.status(201).location(`/users/${id}`).json(users.get(id));
});

app.listen(3000);

What a REST server typically does:

  • Listen for HTTP requests on a port (80/443 via nginx in front).
  • Route based on method + URL to the right handler.
  • Run middleware (parse body, check auth, validate input, rate-limit).
  • Execute business logic — query DB, call other services, perform the action.
  • Return an HTTP response with the right status code, headers, and body.
  • Handle errors consistently — 4xx for client mistakes, 5xx for server bugs.
  • Log every request with a correlation id for debugging.
  • Emit metrics & traces for observability.

Common building blocks in 2026:

  • Framework — Express, Fastify, Hono, NestJS (Node); FastAPI, Django (Python); Spring Boot (Java); Gin, Chi (Go); Rails (Ruby).
  • Deployment — Docker + Kubernetes, serverless (AWS Lambda, Vercel, Cloudflare Workers), or classic VMs.
  • Behind a reverse proxy — Nginx, Caddy, or cloud LB for TLS, compression, static files.
  • Observability — structured logs (JSON), metrics (Prometheus), traces (OpenTelemetry).

A good REST server is invisible when it works: clean URLs, consistent errors, proper status codes, predictable shapes. The craft is in making the whole thing boringly reliable.

Question 99

What are the advantages of using RESTful APIs?

RESTful APIs have become the default because of a combination of simplicity, scalability, and ecosystem maturity. The advantages compound.

Technical advantages:

  • Simple. Just HTTP + JSON — no custom protocols, no special toolchain. New developers are productive in hours.
  • Language-agnostic. Any language with an HTTP client can call any REST API. Your Python script, Go service, and Swift mobile app all work.
  • Stateless and scalable. Any server can handle any request. Add more instances behind a load balancer — instant horizontal scaling, no session replication.
  • Cacheable. GET + Cache-Control + ETag leverages the entire HTTP caching stack: browser, CDN, reverse proxy. Dramatic performance wins for free.
  • Human-readable. JSON is debuggable with curl and DevTools. Contrast with binary protocols (Protocol Buffers, MessagePack) that need tools.
  • Works through corporate networks. Firewalls, proxies, and corporate IT allow HTTP(S). WebSocket and gRPC often get blocked.

Ecosystem advantages:

  • Mature tooling. Postman, Bruno, curl, Swagger UI, OpenAPI codegen, monitoring, testing frameworks — all built for REST.
  • OpenAPI standard. One spec generates docs, typed clients, server stubs, mocks, contract tests.
  • Standard infrastructure. Every LB, CDN, WAF, gateway (Cloudflare, AWS, Nginx) understands HTTP traffic out of the box.
  • Universal knowledge. Every backend developer knows REST. Lower hiring/onboarding cost than niche protocols.

Developer experience advantages:

  • Predictable conventions. Same verbs, same status codes, same URL patterns across every API — quick mental model transfer.
  • Easy to version and evolve. Add optional fields freely; bump major version for breaking changes.
  • Good enough for 95% of use cases. CRUD apps, mobile backends, SaaS integrations — REST handles them all cleanly.

Reach for alternatives (GraphQL, gRPC, WebSockets, tRPC) when you hit REST's specific limits — over-fetching, real-time, or typed cross-service calls. Otherwise, REST is the safe default.

Question 100

How do you monitor the performance of an API?

API performance monitoring tracks how fast, reliable, and healthy your API is under real traffic. It's the feedback loop that catches regressions before users complain.

MetricWhat it tells you
LatencyHow long requests take — track p50, p95, p99 (not averages!)
ThroughputRequests per second — capacity utilization
Error rate% of 4xx / 5xx responses — leading indicator of problems
Availability% of successful requests over time — your SLA number
SaturationCPU, memory, DB connections — running out of capacity?

  • Metrics (Prometheus, Datadog, New Relic) — numerical time-series. Dashboards, alerts.
  • Logs (structured JSON to stdout, aggregated by Loki, Elasticsearch, CloudWatch) — per-request detail for debugging.
  • Traces (OpenTelemetry, Jaeger, Tempo) — follow a request across multiple services.
